# STORY-0032-0002: Add TaskCreate/TaskUpdate for Execution Visibility (Level 2)

## Status: Pendente

## Parent Epic: EPIC-0032

## Depends On: STORY-0032-0001

## Summary

Add `TaskCreate` and `TaskUpdate` tool invocations to `x-dev-epic-implement`, `x-dev-story-implement`, and `x-tdd` skills to provide Level 2 visibility in the Claude Code task list. Level 2 means: 1 task per story (epic level), 1 task per phase + 1 per task-execution (story level).

## User Story

As a **developer running skill orchestration**, I want to see real-time progress in the Claude Code task list showing which phase and task is currently executing, so that I can track progress and detect stalled executions without reading log output.

## Background

Currently, no skill uses `TaskCreate`/`TaskUpdate`. All tracking is via custom `execution-state.json` files that are invisible in the Claude Code UI. During 30+ minute executions, the user sees an empty task list with no indication of progress.

The Level 2 granularity creates ~10-15 tasks per story:
- 1 task for Phase 0 (Preparation)
- 1 task for Phase 1 (Planning)
- N tasks for Phase 2 (1 per task in execution loop)
- 1 task for Phase 3 (Verification)

At epic level: 1 task per story dispatch.

## Acceptance Criteria

### AC-1: x-dev-epic-implement creates tasks for story dispatch

```gherkin
Given x-dev-epic-implement is dispatching stories
When a story subagent is launched via Agent tool
Then a TaskCreate MUST be emitted with description "Story {storyId}: {title}"
And when the subagent returns SubagentResult
Then TaskUpdate MUST set status to "completed" (SUCCESS) or leave in_progress (FAILED)
```

### AC-2: x-dev-story-implement creates tasks per phase

```gherkin
Given x-dev-story-implement is executing phases
When Phase 0 starts
Then TaskCreate MUST emit "Phase 0: Preparation — Story {storyId}"
And when Phase 0 completes, TaskUpdate sets status to "completed"

When Phase 1 starts
Then TaskCreate MUST emit "Phase 1: Planning — {N} subagents"
And when Phase 1 completes, TaskUpdate sets status to "completed"

When Phase 2 starts
Then TaskCreate MUST emit "Phase 2: Task Execution Loop — {M} tasks"
And when Phase 2 completes, TaskUpdate sets status to "completed"

When Phase 3 starts
Then TaskCreate MUST emit "Phase 3: Story-Level Verification"
And when Phase 3 completes, TaskUpdate sets status to "completed"
```

### AC-3: x-dev-story-implement creates tasks per task-execution

```gherkin
Given Phase 2 is iterating over tasks
When a task begins execution (status -> IN_PROGRESS)
Then TaskCreate MUST emit "Task {taskId}: {description} — /x-tdd"
And when the task completes (status -> PR_CREATED or DONE)
Then TaskUpdate sets status to "completed"
And when the task fails (status -> FAILED or BLOCKED)
Then TaskUpdate sets status to "completed" with updated description "(FAILED)"
```

### AC-4: allowed-tools updated

```gherkin
Given x-dev-epic-implement, x-dev-story-implement, and x-tdd SKILL.md files
When I read the allowed-tools frontmatter
Then x-dev-story-implement MUST include "TaskCreate, TaskUpdate"
And x-dev-epic-implement MUST include "TaskCreate, TaskUpdate"
```

### AC-5: TaskCreate complements execution-state.json

```gherkin
Given the existing execution-state.json tracking
When TaskCreate/TaskUpdate are added
Then execution-state.json MUST continue to be updated as before
And TaskCreate provides UI visibility ONLY (complementary, not replacement)
```

## Risks

| Risk | Mitigation |
|------|------------|
| TaskCreate overhead in subagents | Subagents spawned via Agent tool have access to TaskCreate by default (general-purpose agent type) |
| Task list becomes cluttered | Level 2 produces ~10-15 tasks per story, manageable; Level 3 deferred |
| Duplicate tracking (execution-state.json + TaskCreate) | TaskCreate is for UI visibility only; execution-state.json remains source of truth for resume/checkpoint |
