# STORY-0032-0001: Add Skill to allowed-tools and Standardize Delegation Chain

## Status: Pendente

## Parent Epic: EPIC-0032

## Depends On: STORY-0032-0004

## Summary

Add `Skill` to the `allowed-tools` of `x-tdd` and `x-commit` skills, and replace all ambiguous `/x-command` shorthand with explicit `Skill(skill: "name", args: "...")` invocations. This fixes the broken delegation chain where inner skills cannot invoke the next skill in the chain.

## User Story

As a **developer using skill orchestration**, I want the delegation chain `x-dev-story-implement -> x-tdd -> x-commit -> x-format/x-lint` to use explicit Skill tool invocations, so that each skill reliably delegates to the next without ambiguity or silent failures.

## Background

Currently:
- `x-tdd` has `allowed-tools: Bash, Read, Write, Edit, Grep, Glob` (no `Skill`)
- `x-commit` has `allowed-tools: Bash, Read, Grep, Glob, Write, Edit` (no `Skill`)
- Both use `/x-command` syntax that looks like Skill invocations but the tool is unavailable
- x-tdd also says "read only the Slim Mode section" creating ambiguity about inline vs Skill invocation

## Acceptance Criteria

### AC-1: x-tdd can invoke x-commit via Skill tool

```gherkin
Given the x-tdd SKILL.md template
When I read the allowed-tools frontmatter field
Then it MUST include "Skill" in the comma-separated list
And the RED phase commit instruction MUST use Skill(skill: "x-commit", args: "...") syntax
And the GREEN phase commit instruction MUST use Skill(skill: "x-commit", args: "...") syntax
And the REFACTOR phase commit instruction MUST use Skill(skill: "x-commit", args: "...") syntax
And the Slim Mode section MUST also use Skill(skill: "x-commit", args: "...") syntax
```

### AC-2: x-commit can invoke x-format and x-lint via Skill tool

```gherkin
Given the x-commit SKILL.md template
When I read the allowed-tools frontmatter field
Then it MUST include "Skill" in the comma-separated list
And the format step MUST use Skill(skill: "x-format", args: "...") syntax
And the lint step MUST use Skill(skill: "x-lint", args: "...") syntax
And NO ambiguous "/x-format" or "/x-lint" shorthand remains in non-user-facing instructions
```

### AC-3: No ambiguous /x-command shorthand in delegation instructions

```gherkin
Given all skill SKILL.md files in core/
When I search for "/x-" patterns in delegation instructions (not user-facing triggers/examples)
Then all inter-skill delegation instructions MUST use explicit Skill(skill: ...) syntax
And user-facing trigger examples (## Triggers section) MAY still use /x-command shorthand
```

### AC-4: Golden files are regenerated

```gherkin
Given the changes to x-tdd and x-commit templates
When the generator produces output for all 8 profiles
Then the golden files for x-tdd and x-commit MUST reflect the new allowed-tools
And the golden files MUST contain the new Skill(skill: ...) syntax
```


### Output: x-tdd SKILL.md frontmatter (after)
```yaml

```

### Input: x-commit SKILL.md frontmatter (before)
```yaml
allowed-tools: Bash, Read, Grep, Glob, Write, Edit
```

### Output: x-commit SKILL.md frontmatter (after)
```yaml
allowed-tools: Bash, Read, Grep, Glob, Write, Edit, Skill
```

### Delegation syntax (before)
```markdown
/x-commit --task TASK-XXXX-YYYY-NNN --type test --subject "add test for [description]" --tdd RED
```

### Delegation syntax (after)
```markdown
Skill(skill: "x-commit", args: "--task TASK-XXXX-YYYY-NNN --type test --subject \"add test for [description]\" --tdd RED")
```

## Risks

| Risk | Mitigation |
|------|------------|
| Adding Skill tool increases context consumption per invocation | Slim Mode sections remain as compact reference; Skill tool adds ~500 tokens overhead per invocation |
| Existing manual invocations break | User-facing trigger section (`## Triggers`) preserves `/x-command` shorthand for human use |
| LLM may still attempt inline execution | Explicit instruction: "MUST use Skill tool, do NOT read and execute inline" |
