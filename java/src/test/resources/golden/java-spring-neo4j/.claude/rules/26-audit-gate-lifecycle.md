# Rule 26 — Audit Gate Lifecycle

> **Related:** Rule 13 (Skill Invocation Protocol), Rule 14 (Project Scope Guard),
> Rule 19 (Backward Compatibility), Rule 21 (Epic Branch Model),
> Rule 22 (Skill Visibility), Rule 23 (Model Selection), Rule 24 (Execution Integrity),
> Rule 25 (Task Hierarchy & Phase Gate Contract), Rule 45 (CI-Watch Integrity).
> **Introduced by:** EPIC-0058 (Audit Scripts Lifecycle & Generation).
> **ADR:** ADR-0015 (Audit Gate Lifecycle Convention).

## Purpose

Every gate of governance in this repository falls into one of four canonical layers.
Before EPIC-0058, decisions about where to place a new gate (Hook? CI script? Java test?
Workflow?) were made epic-by-epic without formal guidance:

- EPIC-0046 chose Java test (`LifecycleIntegrityAuditTest`).
- EPIC-0050 chose CI script (`audit-model-selection.sh`).
- EPIC-0052 chose a hybrid (Stop hook + CI script + baseline file).

Three audit scripts referenced in Rules 19, 21, and 22 (`audit-flow-version.sh`,
`audit-epic-branches.sh`, `audit-skill-visibility.sh`) did not exist on disk.
This fragmentation raised onboarding cost, hindered review, and blocked reuse.

Rule 26 defines the **taxonomy**, **naming convention**, **exit code contract**,
and **matrix of decision** that every new gate MUST follow.

## Taxonomy

Every governance gate belongs to exactly one of the four layers below.
Choose the layer whose *Decision Criteria* column best matches the gate's
operational needs.

| Layer | Location | Generated By | Decision Criteria |
| :--- | :--- | :--- | :--- |
| **Hook runtime** | `.claude/hooks/verify-*.sh` | `HooksAssembler` | Must execute during the LLM's turn (Stop, PreToolUse, PostToolUse events); results feed directly back to the model |
| **CI script** | `scripts/audit-*.sh` | `ScriptsAssembler` (EPIC-0058) | Stateless check over repository artefacts; runs on PR open/sync via GitHub Actions; produces named exit code |
| **Java test** | `src/test/java/**/*AuditTest.java` or `**/*Lint.java` | Hand-authored (in-scope per Rule 14) | Requires structured code parsing (AST, regex over compiled classes, resource scanning); runs under `mvn verify` |
| **CI workflow** | `.github/workflows/*.yml` | `CicdAssembler` (EPIC-0058 sub-scope) | Orchestrates one or more of the three layers above; provides fail-fast job matrix and SARIF artifact upload |

### Decision Matrix (flow)

```
New gate proposed
│
├─ Needs to run during LLM turn? ─── Yes ──► Hook runtime (.claude/hooks/)
│
└─ No
   │
   ├─ Requires structured code parsing (AST, class loading, resource scan)? ─── Yes ──► Java test (src/test/java/)
   │
   └─ No
      │
      ├─ Checks artefacts on PR / push? ─── Yes ──► CI script (scripts/audit-*.sh)
      │
      └─ Orchestrates other gates? ─────────── Yes ──► CI workflow (.github/workflows/)
```

## Naming & Exit Codes

### Naming Conventions

| Layer | Pattern | Examples |
| :--- | :--- | :--- |
| Hook runtime | `verify-{subject}.sh` (kebab-case) | `verify-phase-gates.sh`, `verify-story-completion.sh` |
| CI script | `audit-{subject}.sh` (kebab-case; prefix `audit-` mandatory) | `audit-flow-version.sh`, `audit-skill-visibility.sh` |
| Java test | `{Subject}AuditTest.java` or `{Subject}Lint.java` | `LifecycleIntegrityAuditTest.java`, `TelemetryMarkerLint.java` |
| CI workflow | `{action}.yml` (kebab-case) | `audit.yml`, `ci.yml`, `release.yml` |

### Standardized Exit Codes (CI scripts only)

| Code | Meaning | Named constant (stderr) | Example message |
| :--- | :--- | :--- | :--- |
| 0 | Success — no violations detected | — | — |
| 1 | Violation detected | `<RULE>_VIOLATION` (e.g., `FLOW_VERSION_VIOLATION`) | `FLOW_VERSION_VIOLATION: execution-state.json has flowVersion=3` |
| 2 | Operational error | `OPERATIONAL_ERROR` | `OPERATIONAL_ERROR: jq not found on PATH` |
| 3 | Baseline / exemption file corrupt | `BASELINE_CORRUPT` or `INVALID_EXEMPTION` | `BASELINE_CORRUPT: audits/baseline.txt malformed at line 12` |

Java tests and CI workflows do not follow this matrix — they use JUnit assertion failures and GitHub Actions job exit codes respectively.

## `--self-check` Flag (CI Scripts Only)

Every CI script (`audit-*.sh`) MUST implement the `--self-check` flag:

- **Exit 0** when the script is structurally valid (required dependencies on `PATH`,
  baseline file readable if declared, `--self-check` logic itself reachable).
- **Exit 2** (`OPERATIONAL_ERROR`) when any structural prerequisite is missing.

The CI workflow runs `--self-check` for all scripts **before** the main audit matrix.
A structurally broken script aborts the job without polluting other audit results
(RULE-006 in EPIC-0058: fail-fast).

```bash
# Minimal --self-check implementation template
case "${1:-}" in
  --self-check)
    command -v jq >/dev/null 2>&1 || { echo "OPERATIONAL_ERROR: jq required" >&2; exit 2; }
    [[ -f "audits/my-baseline.txt" ]] || { echo "OPERATIONAL_ERROR: baseline missing" >&2; exit 2; }
    exit 0
    ;;
esac
```

## Catalog-before-Add (RULE-004)

No gate of any layer may be introduced — in any Rule, ADR, SKILL.md, or code comment —
without a simultaneous entry in `docs/audit-gates-catalog.md`.

A PR that adds a Rule reference to an `audit-*.sh` script without a matching catalog
entry fails `scripts/audit-skill-visibility.sh` (extended in story-0058-0005 to
check orphan script references).

## Forbidden

- Introducing a CI script without the `audit-` prefix (e.g., `check-flow.sh`) — violates naming convention; script will not be catalogued by `ScriptsAssembler`.
- Emitting exit codes outside the 0–3 range without a rule amendment.
- Adding a gate to any Rule without a catalog entry (violates RULE-004).
- Implementing a Hook runtime gate for a check that only needs to run on PR (wrong layer — adds latency to every LLM turn unnecessarily).
- Implementing a CI script gate for a check that requires compiled Java class inspection (wrong layer — use Java test instead).
- Omitting `--self-check` from a CI script — the CI workflow's pre-flight suite will fail.
- Using the prefix `verify-` for a CI script (reserved for Hook runtime layer) or `audit-` for a Hook (reversed prefixes confuse `ScriptsAssembler`).

## Audit

This rule is self-verified by the catalog check in `scripts/audit-skill-visibility.sh`
(EPIC-0058, story-0058-0005): every `audit-*.sh` reference found in any `.md` file
under `java/src/main/resources/targets/claude/` MUST have a corresponding entry in
`docs/audit-gates-catalog.md`. Missing entry → exit 1 `ORPHAN_SCRIPT_REFERENCE`.

The smoke test `Epic0058Rule26SmokeTest` validates:
1. This rule file exists in source-of-truth.
2. All 8 mandatory sections are present.
3. The taxonomy table contains ≥ 4 rows.
4. The exit-code table contains rows for 0, 1, 2, and 3.
