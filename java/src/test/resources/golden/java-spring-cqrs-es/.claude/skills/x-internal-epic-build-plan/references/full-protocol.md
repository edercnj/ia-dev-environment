## 1. Argument-Parser Rejection Matrix

The parser is a tight single-file loop (`while (($#)); case "$1" in …`)
to keep the SKILL.md within the SkillSizeLinter 500-line threshold
without delegating to `x-internal-args-normalize` (peer, not a
dependency — same policy as `x-internal-story-build-plan` §1).

| Input | Result | Exit |
| :--- | :--- | :--- |
| `--epic-id 0049 --mode sequential --output out.md` | accepted (`strict-overlap=false`) | 0 |
| `--epic-id=0049 --mode=parallel --output=out.md` | accepted (`=` form) | 0 |
| `--epic-id 49 --mode sequential --output out.md` | accepted (`49` → `0049`) | 0 |
| `--epic-id 0049 --mode PARALLEL --output out.md` | accepted (lowercased) | 0 |
| `--epic-id 0049 --mode fancy --output out.md` | `usage: --mode must be sequential / parallel` | 64 |
| `--epic-id 0049 --mode sequential` | `usage: --output is required` | 64 |
| `--epic-id abc --mode sequential --output out.md` | `usage: --epic-id must be a 4-digit integer` | 64 |
| `--epic-id 0049 --mode parallel --output /root/out.md` (unwritable) | `usage: --output path not writable` | 64 |
| `--epic-id 0049 --mode parallel --output out.md --strict-overlap` | accepted (strict=true) | 0 |
| `--epic-id 0049 --mode sequential --output out.md --strict-overlap` | accepted but `strictOverlap` field echoed as `true` with no effect on matrix (null) | 0 |
| `--unknown-flag` | `usage: unknown flag --unknown-flag` | 64 |
| `--help` | print banner + usage; exit 0 | 0 |

The banner is intentionally terse (< 20 lines) so sourcing the
skill from a parent script that accidentally passes `--help` does
not flood the caller's stdout.

## 2. IMPLEMENTATION-MAP Parser Grammar

The IMPLEMENTATION-MAP.md dependency table is the canonical source
of `blockedBy` edges. Every epic in the project follows the same
layout (auto-generated by `x-epic-map`):

```markdown
| Story | Name | Jira | Blocked By | Blocks | Status |
| :--- | :--- | :--- | :--- | :--- | :--- |
| story-0049-0001 | … | — | — | story-0049-0005, story-0049-0007 | Concluída |
| story-0049-0006 | … | — | story-0049-0001 | story-0049-0009, story-0049-0010 | Concluída |
```

Parser rules:

1. Locate the header row by exact match on the six-column pipe
   pattern above. Stop reading at the first blank line after a data
   row.
2. Column index is 0-based across pipes; `Story` column = 0,
   `Blocked By` column = 3.
3. For each data row: trim each cell, split `Blocked By` on `,`,
   strip per-item whitespace; `-`, `—`, or empty cell → empty list.
4. Stories listed in `Blocks` are IGNORED by this skill (it is
   redundant information — `A blocks B` is the inverse of
   `B blockedBy A`, and only `blockedBy` is used for the DAG).
5. Trailing `🔒` / `⚠` / unicode markers inside `Name` MUST NOT
   affect parsing of `Story` or `Blocked By` cells — the former
   are cosmetic and contained within their own cell.

When no header row matches within the first 200 lines, emit:
`IMPLEMENTATION-MAP.md missing header row under plans/epic-<id>`
and exit 2 (treated equivalently to a missing file — the contract
is on the table, not on file presence alone).

## 3. Kahn's Algorithm Pseudocode

```python
stories = set(on_disk_story_ids)
edges   = { storyId: set(blockedBy) }  # parsed from map

in_degree = { s: 0 for s in stories }
for s, deps in edges.items():
    for d in deps:
        if d not in stories:
            exit(4, f"Story file missing: {d}.md")
    in_degree[s] = len(deps)

queue = sorted([s for s, d in in_degree.items() if d == 0])
phases = []
while queue:
    phase = sorted(queue)
    phases.append(phase)
    queue = []
    for s in phase:
        for dependent in reverse_edges[s]:  # who is blocked by s
            in_degree[dependent] -= 1
            if in_degree[dependent] == 0:
                queue.append(dependent)

if sum(len(p) for p in phases) < len(stories):
    # cycle — walk from any unvisited node with in_degree > 0
    # following outgoing edges until revisiting a node
    cycle_chain = find_cycle(in_degree, edges)
    exit(3, f"Cycle detected: {' -> '.join(cycle_chain)} -> {cycle_chain[0]}")
```

Determinism is guaranteed by the per-phase `sorted()` calls; two
runs on the same input produce the same `phases` array ordering.

## 4. Critical-Path Computation

```python
longest = { s: 1 for s in stories }      # base case: standalone
back    = { s: None for s in stories }

# iterate stories in topological (Kahn) order — dependencies first
for phase in phases:
    for s in phase:
        for dep in edges[s]:
            if longest[dep] + 1 > longest[s]:
                longest[s] = longest[dep] + 1
                back[s]    = dep

target = max(stories, key=lambda s: (longest[s], -ord_of(s)))
chain  = []
while target is not None:
    chain.append(target)
    target = back[target]
critical_path = list(reversed(chain))
```

The `-ord_of(s)` tie-break selects the lexicographically smaller
storyId when two endpoints have equal `longest` — this matches the
story-0049-0009 §7 requirement of "most-long chain, deterministic
ordering".

## 5. File-Footprint Parser

Each plan file under `plans/epic-XXXX/plans/plan-story-*.md`,
`plan-task-*.md`, and `tasks-story-*.md` may contain a
`## File Footprint` or `## Story File Footprint` block introduced
by EPIC-0041:

```markdown
## Story File Footprint

write:
- java/src/main/.../FooAssembler.java

regen:
- src/test/resources/golden/java-spring/**

read:
- java/src/main/.../BarUseCase.java
```

Parser extracts `write:` + `regen:` entries under `## (Story )?File Footprint`
headers; `read:` entries are ignored (they do not cause
write-collisions). Glob patterns (`**`) are preserved as literal
strings and compared with glob-aware equality (two `**` patterns
rooted at the same directory count as overlapping).

Stories with NO plan file, OR with plan files that lack the
`## File Footprint` block, are classified `footprint-unknown` and
participate in the matrix with an empty peer list plus a record in
the envelope's `warnings` array. This is the "warn, do not block"
posture dictated by RULE-006 (EPIC-0041).

## 6. Overlap Severity Decision Table

Per RULE-004 (ADR-0006), every write-collision is classified:

| Overlap Contents | Severity | Rationale |
| :--- | :--- | :--- |
| Empty (disjoint footprints) | `none` | No collision possible |
| Only non-hotspot hand-edited files (e.g., distinct `*.java` sources) | `soft` | Safe to parallelise; typical file-level divergence |
| Only regen-output files (e.g., `.claude/settings.json`, golden files) | `regen` | Last-writer-wins — acceptable if waves serialise the regen step |
| Any hotspot file listed below | `hard` | MUST degrade to serial; mid-wave conflict is near-certain |

RULE-004 hotspot list (authoritative source of truth lives in
`.claude/rules/04-architecture-summary.md` §Restrições de
Paralelismo; this skill copies the list verbatim):

- `java/src/main/java/dev/iadev/application/assembler/SettingsAssembler.java`
- `java/src/main/java/dev/iadev/application/assembler/HooksAssembler.java`
- `CLAUDE.md`
- `CHANGELOG.md`
- `pom.xml`
- `.gitignore`
- `src/test/resources/golden/**`

Envelope-level `overlapSeverity` is `max(severity for every pair)`.
When `warnings` contains footprint-unknown advisories, the severity
computation still runs on the pairs WITH known footprints; the
unknowns do NOT inflate the severity.

## 7. `x-internal-report-write` stdin Schema

The downstream `x-internal-report-write` invocation consumes a
JSON document on stdin. The schema passed by this skill:

```json
{
  "template": "_TEMPLATE-EPIC-EXECUTION-PLAN.md",
  "placeholders": {
    "{{EPIC_ID}}":          "0049",
    "{{MODE}}":             "sequential",
    "{{PHASE_COUNT}}":      "5",
    "{{STORY_COUNT}}":      "22",
    "{{PHASES_TABLE}}":     "… rendered markdown table …",
    "{{CRITICAL_PATH}}":    "story-0049-0001 → story-0049-0005 → story-0049-0018 → story-0049-0020",
    "{{OVERLAP_MATRIX}}":   "(null or rendered markdown table)",
    "{{OVERLAP_SEVERITY}}": "(null or one of none/soft/regen/hard)",
    "{{RENDERED_AT}}":      "2026-04-22T13:00:00Z"
  }
}
```

`x-internal-report-write` applies the placeholders to the template
literal and writes the result to its own `--output`. This skill
passes `--output` verbatim from its own argument; the child does
NOT re-resolve.

On child non-zero exit, this skill does NOT retry — retry logic
belongs upstream in `x-epic-implement`'s resume path. The
non-retried error yields exit 5 with the child's first stderr line
appended.

## 8. Interaction with `x-parallel-eval` (Downstream)

The Phase 1.5 collision gate in `x-epic-implement` consumes the
`overlapMatrix` and `overlapSeverity` fields of this skill's
envelope and compares them against the per-wave planning scope.
The boundary of responsibility:

- **This skill:** detects and classifies write-collisions across
  every Kahn-co-scheduled pair; advisory only — never aborts on
  severity.
- **`x-parallel-eval`:** applies wave-specific thresholds,
  downgrades waves to serial, and records
  `ExecutionState.parallelismDowngrades` (EPIC-0041
  story-0041-0006).

The split allows `x-internal-epic-build-plan` to remain pure
computation + render — it can be invoked with `--mode parallel`
from non-orchestrator contexts (e.g., ad-hoc audits) without side
effects.

## 9. Performance Notes

- **Epic load:** O(N) file stats for `story-*.md` + one read each
  for `epic-*.md` and `IMPLEMENTATION-MAP.md`. 22-story epic: ~30
  ms cold cache.
- **Footprint parse:** O(N · M) where M = average `## File Footprint`
  block length (typically < 20 lines). 22 stories × 20 = 440 regex
  applications; negligible.
- **Kahn + critical path:** O(V + E) = O(22 + ~40 edges) for
  epic-0049. Sub-millisecond.
- **Overlap matrix:** O(K²) per phase where K is the phase size.
  For the widest phase in epic-0049 (phase index 2 with 9 stories),
  81 pair comparisons × footprint-set-intersection. < 5 ms.
- **Render delegation:** ~500–2000 ms (network-free, template-only
  substitution) — bounded by `x-internal-report-write`'s own
  perf contract, not this skill's.

Total wall-clock for the happy path on a 22-story epic: consistently
under 3 seconds. The 5-second DoD is set with 40% headroom for
larger epics (up to 30 stories).

## 10. Edge Cases

| Case | Behaviour |
| :--- | :--- |
| Epic file exists but IMPLEMENTATION-MAP has zero data rows | Treat as zero-edge DAG: 1 phase with all stories, no cycle, critical path length = 1 |
| Story file named `story-XXXX-NNNN.md` where NNNN contains a leading zero in the map but not in the file | Parser normalises both forms to 4-digit zero-padded |
| IMPLEMENTATION-MAP references a story in a DIFFERENT epic (`story-YYYY-NNNN` with YYYY ≠ epic_id) | Treat as `STORY_FILE_MISSING` (exit 4) — cross-epic edges are not supported by the current schema |
| `--mode parallel` but epic has only 1 story | `overlapMatrix: {}` (empty object, not null); `overlapSeverity: "none"` |
| Two stories in the same phase both have `footprint-unknown` | Both listed in `warnings`; no entry in `overlapMatrix` for either; `overlapSeverity` computed across known-footprint pairs only (may be `none`) |
| `--output` equals a path inside `plans/epic-${epic_id}/` | Supported — the skill resolves relative paths from `$CLAUDE_PROJECT_DIR`; no recursion guard needed since the skill does not re-read its own output |

These cases are covered by the acceptance scenarios in the SKILL.md
Testing section and by the future regression goldens under
`src/test/resources/golden/internal/plan/x-internal-epic-build-plan/`.
