# Implementation Plan — STORY-015: ReadmeAssembler

## Story Summary

Migrate `ReadmeAssembler` (431 lines Python) to TypeScript. This is the second-largest assembler and the **last one** in the pipeline execution order. It scans files generated by all 13 preceding assemblers to produce a comprehensive `README.md` with dynamic counts, mapping tables, knowledge pack detection, hooks/settings documentation, and a generation summary.

**Blocked by:** STORY-005 (template engine), STORY-008 (assembler helpers) — both complete.
**Blocks:** STORY-016 (pipeline orchestration).

## 2. New Classes/Interfaces to Create

### 2.1 `src/assembler/readme-assembler.ts` — ReadmeAssembler

**Purpose:** Generate `README.md` in the output directory by scanning all previously-assembled artifacts and building either a full template-based README or a minimal fallback.

```
export class ReadmeAssembler {
  assemble(
    config: ProjectConfig,
    outputDir: string,
    resourcesDir: string,
    engine: TemplateEngine,
  ): string[]
}
```

**Note on signature:** The Python version's constructor takes `resources_dir` and the `assemble` method takes `(config, output_dir, engine)`. Following the established TypeScript pattern (as in `SkillsAssembler`, `GithubAgentsAssembler`, etc.), the TypeScript version passes `resourcesDir` per-call instead of via constructor. This avoids constructor state and aligns with all other migrated assemblers.

#### Key Logic Mapping (Python to TypeScript)

**Full README mode (`generateReadme`):**
1. Read `resources/readme-template.md`.
2. Replace 12 `{{PLACEHOLDER}}` tokens with dynamically-built content:
   - `{{PROJECT_NAME}}` — from `config.project.name`
   - `{{RULES_COUNT}}` — count `.md` files in `outputDir/rules/`
   - `{{SKILLS_COUNT}}` — count `SKILL.md` files in `outputDir/skills/*/`
   - `{{AGENTS_COUNT}}` — count `.md` files in `outputDir/agents/`
   - `{{RULES_TABLE}}` — formatted markdown table with #, file, scope
   - `{{SKILLS_TABLE}}` — formatted table (excludes knowledge packs)
   - `{{AGENTS_TABLE}}` — formatted table with agent name and file
   - `{{HOOKS_SECTION}}` — post-compile check info from stack-mapping
   - `{{KNOWLEDGE_PACKS_TABLE}}` — table of skills with `user-invocable: false`
   - `{{SETTINGS_SECTION}}` — static content about settings.json
   - `{{MAPPING_TABLE}}` — `.claude/ <-> .github/` equivalence table
   - `{{GENERATION_SUMMARY}}` — breakdown by component with version stamp

**Important:** These are **NOT** the `{placeholder}` replacements handled by `TemplateEngine.replacePlaceholders()`. The template uses `{{DOUBLE_BRACES}}` which are distinct from the engine's `{single_brace}` pattern. The Python code uses `str.replace()` directly for these. The TypeScript must do the same — **do not** route through `TemplateEngine` for these replacements.

**Minimal README mode (`generateMinimalReadme`):**
1. Build header with project name.
2. Build directory structure block (hardcoded).
3. Build tips block with architecture style and interfaces.
4. Concatenate and return.

**Knowledge pack detection (`isKnowledgePack`):**
1. Read SKILL.md content.
2. Return `true` if content contains `"user-invocable: false"` OR any line starts with `"# Knowledge Pack"`.

**Hooks section (`buildHooksSection`):**
1. Call `getHookTemplateKey(config.language.name, config.framework.buildTool)` from `src/domain/stack-mapping.ts`.
2. If key is empty, return "No hooks configured."
3. Else, look up `LANGUAGE_COMMANDS[language-buildTool]` for `fileExtension` and `compileCmd`.
4. Return formatted markdown with event, script, behavior, purpose.

**Generation summary (`buildGenerationSummary`):**
1. Count all component types (rules, skills minus KPs, KPs, agents, hooks, settings).
2. Count GitHub components (instructions, skills, agents, prompts, hooks, MCP).
3. GitHub instructions count includes global `copilot-instructions.md` (+1).
4. Format as markdown table.
5. Append `Generated by \`ia-dev-env v${DEFAULT_FOUNDATION.version}\`.` — uses `DEFAULT_FOUNDATION.version` from `src/models.ts` (Python uses `__version__`).

### 5.1 Stack Mapping (read-only)

- `buildHooksSection` imports `getHookTemplateKey` and `LANGUAGE_COMMANDS` from `src/domain/stack-mapping.ts` to determine file extension and compile command for the hooks documentation section.
- Python equivalent: `from ia_dev_env.domain.stack_mapping import get_hook_template_key` and inline import of `LANGUAGE_COMMANDS`.

### 5.2 ProjectConfig + DEFAULT_FOUNDATION (read-only)

- `config.project.name` — for `{{PROJECT_NAME}}` replacement.
- `config.language.name`, `config.framework.buildTool` — for hooks section.
- `config.architecture.style`, `config.interfaces` — for minimal README tips.
- `DEFAULT_FOUNDATION.version` — replaces Python's `__version__` for the generation summary footer.

### 5.3 Output directory scanning (read-only, filesystem)

The ReadmeAssembler **reads** from the output directory to count and catalog artifacts:
- `outputDir/rules/*.md` — count and table
- `outputDir/skills/*/SKILL.md` — count, table, KP detection
- `outputDir/agents/*.md` — count and table
- `outputDir/hooks/*` — count
- `outputDir/settings.json`, `outputDir/settings.local.json` — count
- `outputDir/github/` — recursive file count, component counts

This is the reason it must run **last** in the pipeline.

### 5.4 Resources (read-only)

- `resources/readme-template.md` — the template with 12 `{{PLACEHOLDER}}` tokens.

### 5.5 Pipeline integration (future STORY-016)

The `ReadmeAssembler` follows the established `assemble(config, outputDir, resourcesDir, engine)` signature. It must be the **last** assembler called in the pipeline, matching the Python `_build_assemblers` order where `ReadmeAssembler` is the final entry.

## 10. Risk Assessment

| Risk | Likelihood | Impact | Mitigation |
|------|-----------|--------|------------|
| `{{PLACEHOLDER}}` collision with TemplateEngine's `{placeholder}` | Medium | High | The template uses `{{DOUBLE_BRACES}}` while TemplateEngine matches `{SINGLE_BRACES}`. However, `{{PROJECT_NAME}}` would match the regex `\{(\w+)\}` and could be partially consumed. The Python code does NOT pass the template through the engine — it uses `str.replace()` directly. TypeScript must do the same. Test by verifying `{{RULES_TABLE}}` survives intact. |
| Knowledge pack detection false positives | Low | Medium | Test with files that contain `user-invocable: true` (should NOT be detected) and partial matches like `user-invocable: false` in comments. |
| `DEFAULT_FOUNDATION.version` vs Python `__version__` divergence | Low | Low | Both currently `"0.1.0"`. Use the constant directly; no hardcoding. |
| File counting race conditions (files not yet written) | Low | High | This assembler MUST run last in the pipeline. Validate in STORY-016 pipeline integration. For unit tests, pre-create fixture files. |
| `extractRuleNumber` regex differs between Python and TypeScript | Low | Medium | Python regex: `r"^(\d+)"`. TypeScript: `/^(\d+)/`. Functionally identical. Test with `"01-project-identity.md"`, `"no-number.md"`, and `"123-many-digits.md"`. |
| GitHub directory named `github/` not `.github/` | Low | Medium | The Python code scans `output_dir / "github"` (not `.github/`). The pipeline generates into a temp dir using `github/` as the subdirectory. The TypeScript must use the same path. |
| Mapping table static content drifts from Python | Low | Low | The 8-row mapping table is hardcoded in both Python and TypeScript. Snapshot-test the exact output. |
| `countGithubFiles` using `rglob("*")` equivalent | Low | Low | Python uses `Path.rglob("*")`. TypeScript: use recursive `readdirSync` or `globSync`. Test with nested directory structures. |
| Skills table excludes knowledge packs | Medium | Medium | The `_build_skills_table` function calls `_is_knowledge_pack` to skip KPs. Must test with a mix of regular skills and KPs. |
| `buildHooksSection` accessing `LANGUAGE_COMMANDS` with composite key | Low | Low | Python uses tuple key `(lang, build_tool)`. TypeScript `LANGUAGE_COMMANDS` already uses `"lang-buildTool"` string keys. Use `getHookTemplateKey` plus `LANGUAGE_COMMANDS[key]` lookup. |

### Key differences from Python

1. **Constructor vs per-call:** Python `ReadmeAssembler(resources_dir)` stores resources_dir. TypeScript passes it per-call.
2. **`__version__` vs `DEFAULT_FOUNDATION.version`:** Python imports `__version__` from package. TypeScript uses `DEFAULT_FOUNDATION.version` from `src/models.ts`.
3. **`Path.glob()` vs `fs.readdirSync()` + filter:** TypeScript uses synchronous `node:fs` directory reading with manual glob-like filtering.
4. **`Path.rglob("*")` vs recursive readdir:** For `countGithubFiles`, implement a recursive file counter using `fs.readdirSync({ withFileTypes: true, recursive: true })` (Node 20+) or a manual recursive walk.
5. **`Path.removesuffix()` vs `.replace()`:** Python `removesuffix(".md")` becomes TypeScript `filename.replace(/\.md$/, "")` or `filename.slice(0, -3)`.
6. **`Path.stem` vs `path.basename() + replace`:** Python `agent_file.stem` becomes `path.basename(file, ".md")` in TypeScript.
7. **`Path.iterdir()` vs `fs.readdirSync()`:** Direct mapping.
8. **`str.replace()` for `{{PLACEHOLDER}}`:** Both Python and TypeScript use direct string replacement, NOT the template engine. The template engine's `{single_brace}` pattern is distinct.
