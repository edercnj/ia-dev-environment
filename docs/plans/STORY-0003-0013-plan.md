## 3. Existing Files to Modify

### 3.1 `resources/skills-templates/core/x-git-push/SKILL.md` (Source of Truth -- Claude)

**Change:** Append three new sections before the existing `## Integration Notes` section. The new content must be inserted between `## Tagging Releases` and `## Integration Notes` to maintain logical document flow.

**New sections to add:**

1. **`## TDD Commit Format`** -- A subsection with a table/list showing the 4 TDD commit formats:
   - `test(scope): add test for [behavior] [TDD:RED]` -- failing test only
   - `feat(scope): implement [behavior] [TDD:GREEN]` -- make test pass
   - `refactor(scope): [improvement] [TDD:REFACTOR]` -- refactoring step
   - `feat(scope): implement [behavior] [TDD]` -- combined (recommended default)

2. **`## Atomic TDD Commit Rules`** -- A numbered list with the 5 atomicity rules:
   - One commit per Red-Green-Refactor cycle
   - Test and implementation in the SAME commit (avoid separate test commits)
   - Refactoring may be a separate commit (immediately after green)
   - Each commit adds ONE testable behavior
   - Maximum ~50 lines changed per commit; larger commits warrant splitting

3. **`## Git History Storytelling`** -- A guideline section describing TDD progression in commit history:
   - First commit: acceptance test + test infrastructure
   - Subsequent commits: incremental unit tests (TPP order)
   - Final commits: refactoring and polish
   - History should read as "progression from simple to complex"

**Backward compatibility (RULE-003):** All existing Conventional Commits types and rules MUST remain unchanged. The `[TDD]`, `[TDD:RED]`, `[TDD:GREEN]`, `[TDD:REFACTOR]` tags are ADDITIONAL suffixes to the existing `<type>(<scope>): <subject>` format -- they do not replace the type field. The combined `[TDD]` format must be documented as the recommended default, with the individual phase tags (`RED`/`GREEN`/`REFACTOR`) as optional alternatives.

## Content Specification

### Section: TDD Commit Format

Insert after `## Tagging Releases` and before `## Integration Notes`.

The section must document 4 commit formats:

| Format | Phase | Usage |
|--------|-------|-------|
| `test(scope): add test for [behavior] [TDD:RED]` | Red | Failing test only (no implementation) |
| `feat(scope): implement [behavior] [TDD:GREEN]` | Green | Minimal implementation to pass test |
| `refactor(scope): [improvement] [TDD:REFACTOR]` | Refactor | Restructuring, no new behavior |
| `feat(scope): implement [behavior] [TDD]` | Combined | Test + implementation in one commit (**recommended**) |

The combined format `[TDD]` must be documented as the recommended default. The individual phase tags are optional alternatives for finer-grained history.

### Section: Atomic TDD Commit Rules

5 rules:
1. One commit per complete Red-Green-Refactor cycle
2. Test and implementation in the SAME commit (avoid separate test-only commits)
3. Refactoring MAY be a separate commit (immediately after green)
4. Each commit adds ONE testable behavior
5. Maximum ~50 lines changed per commit; larger changes should be split

### Section: Git History Storytelling

Guidelines for commit sequence:
- First commit: acceptance test + test infrastructure setup
- Subsequent commits: incremental unit tests following TPP (Transformation Priority Premise) order
- Final commits: refactoring and polish
- The git log should read as a progression from simple to complex
