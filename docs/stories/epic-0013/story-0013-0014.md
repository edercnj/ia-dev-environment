## 3. Descricao

Como **tech lead**, eu quero uma skill de orquestracao de release end-to-end, para que o processo de version bump, changelog, tag, verificacao de artefatos e publicacao seja automatizado e consistente.

### Contexto

O ia-dev-env gera `x-changelog` para gerar changelogs e `x-git-push` para operacoes git, mas nao existe uma skill que orquestre o fluxo completo de release. Um release envolve: determinar versao (automatica ou explicita), validar pre-condicoes (testes, cobertura, working directory limpo), atualizar versao nos arquivos do projeto, gerar/atualizar changelog, criar commit de release, criar tag git e opcionalmente publicar. Sem essa orquestracao, cada equipe inventa seu proprio processo, levando a releases inconsistentes e propensos a erro.

### 3.1 Skill Path e Frontmatter

- Path: `skills-templates/x-release/SKILL.md`
- Frontmatter:
  - `user-invocable: true`
  - `argument-hint: "[major|minor|patch|version] [--dry-run] [--skip-tests] [--no-publish]"`
  - `allowed-tools: [Read, Write, Edit, Bash, Glob, Grep, Agent]`

### 3.2 Workflow

1. **Determine Version:** Parse argument — if `major`, `minor`, or `patch`, calculate next version from current; if explicit version (e.g., `2.1.0`), use directly; if no argument, auto-detect from Conventional Commits since last tag (breaking change -> major, feat -> minor, fix -> patch)
2. **Validate Pre-conditions:** All tests pass (unless `--skip-tests`), coverage meets threshold, no uncommitted changes in working directory, current branch is main/master (or release branch)
3. **Update Version:** Update version in project-specific files (pom.xml for Maven, package.json for npm, Cargo.toml for Rust, pyproject.toml for Python, build.gradle for Gradle)
4. **Generate Changelog:** Delegate to `x-changelog` skill to generate/update CHANGELOG.md with changes since last tag
5. **Create Release Commit:** Conventional Commits format: `release: v{version}`
6. **Create Git Tag:** Annotated tag `v{version}` with release notes summary
7. **Dry-Run Mode:** If `--dry-run`, show complete plan (version, files to update, changelog preview, commit message, tag) without executing any changes
8. **Publish:** If not `--no-publish`, push commit and tag to trigger release workflow (CI/CD)

**CLI Arguments:**

| Argumento | Tipo | Obrigatorio | Descricao |
| :--- | :--- | :--- | :--- |
| `major\|minor\|patch` | Enum | N | Increment type for automatic version calculation |
| `{version}` | String | N | Explicit version (e.g., "2.1.0") |
| `--dry-run` | Flag | N | Show plan without executing |
| `--skip-tests` | Flag | N | Skip test validation step |
| `--no-publish` | Flag | N | Skip push/publish step |

**Version File Patterns:**

| Linguagem | Arquivo | Pattern |
| :--- | :--- | :--- |
| Java (Maven) | `pom.xml` | `<version>X.Y.Z</version>` |
| Java (Gradle) | `build.gradle` | `version = 'X.Y.Z'` |
| TypeScript | `package.json` | `"version": "X.Y.Z"` |
| Python | `pyproject.toml` | `version = "X.Y.Z"` |
| Rust | `Cargo.toml` | `version = "X.Y.Z"` |
| Go | `go module` | Git tags only (no version file) |

### 6.1 Release Workflow

```mermaid
flowchart TD
    A["/x-release [args]"] --> B{Parse arguments}
    B -->|major/minor/patch| C[Calculate next version]
    B -->|explicit version| D[Use provided version]
    B -->|no argument| E[Auto-detect from Conventional Commits]
    C --> F{--dry-run?}

    E --> F
    F -->|yes| G[Show plan and exit]

    H -->|tests fail| J{--skip-tests?}
    J -->|yes| K[Continue with warning]
    J -->|no| L[ABORT: tests failing]
    H -->|valid| M[Update version files]
    K --> M
    M --> N[Generate/update CHANGELOG.md]
    N --> O[Create release commit]
    O --> P[Create git tag v{version}]
    P --> Q{--no-publish?}
    Q -->|yes| R[Done - local only]
    Q -->|no| S[Push commit and tag]
    S --> T[Done - release triggered]
```

```gherkin
  DADO que existem alteracoes nao commitadas no working directory
  QUANDO o usuario invoca "/x-release minor"

  E a mensagem contem "uncommitted changes"
  E nenhum arquivo de versao e modificado

Cenario: Skill gerada com frontmatter correto para todos os targets
  DADO que o pipeline e executado para qualquer perfil
  QUANDO a skill x-release e gerada
  ENTAO o SKILL.md existe com `user-invocable: true`
  E contem `argument-hint` com opcoes de versao
  E contem `allowed-tools` com lista de tools
```
