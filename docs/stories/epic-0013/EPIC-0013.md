# Épico: Cobertura Completa do Ciclo de Desenvolvimento de Software (SDLC)

**Autor:** Product Owner / Claude
**Data:** 2026-03-26
**Versão:** 1.0
**Status:** Em Refinamento

## 1. Visão Geral

**Chave Jira:** SCRUM-3

Uma análise abrangente do ia-dev-environment revelou que a cobertura atual do ciclo de desenvolvimento de software (SDLC) está em **~75-80%**. As áreas fortes — planejamento, arquitetura, implementação (TDD), code review e segurança — estão em 90-95%. Porém, existem lacunas significativas em **CI/CD** (~50%), **operações** (~60%), **gestão de dados** (~40%), **engenharia de performance** (~50%) e **release management** (~70%) que impedem a autonomia completa do desenvolvimento assistido por IA.

Este épico fecha essas lacunas criando novos knowledge packs, skills, templates, agents e rules que elevam a cobertura total para **~92-95%**. Todos os artefatos são templates de geração (recursos em `java/src/main/resources/`) e assemblers (classes em `java/src/main/java/dev/iadev/assembler/`) integrados ao pipeline existente do ia-dev-env.

**Escopo incluído:**
- 7 novos Knowledge Packs (CI/CD, SRE, Release Management, Data Management, Performance Engineering, Feature Flags, Disaster Recovery)
- 6 novas Skills (x-ci-cd-generate, x-ops-incident, x-release, x-perf-profile, x-threat-model, x-setup-dev-environment)
- 5 extensões de KPs existentes (observability, resilience, api-design, security, database-patterns)
- 2 extensões de Skills existentes (x-dependency-audit, run-perf-test)
- 2 novos Agents (sre-engineer, database-engineer)
- 3 novas Rules (07-operations, 08-release, 09-data-management)
- 7 novos Templates (incident response, postmortem, operational runbook, SLO/SLI, release checklist, data migration plan, contributing guide)
- GitHub PR/Issue templates assembler
- CD workflow templates (release, deploy, rollback)

**Escopo excluído:**
- Implementação de funcionalidades de runtime (este épico cria templates de geração, não código de aplicação)
- Alteração de skills existentes que já estão bem cobertos (x-review, x-review-pr, x-dev-implement)
- Suporte a novos frameworks ou linguagens
- Alterações no schema YAML de configuração (apenas novos campos condicionais)

## 3. Definições de Qualidade Globais

### Global Definition of Ready (DoR)

- Pipeline de geração funcional para todos os 10 perfis bundled
- Golden file tests existentes passando
- Estrutura de resources templates compreendida
- Assembler pipeline e factory pattern compreendidos
- Impacto em assemblers downstream mapeado

### Global Definition of Done (DoD)

- **Cobertura:** >= 95% Line, >= 90% Branch
- **Testes Automatizados:** Unit tests para assemblers, integration tests para pipeline completo. Cada história DEVE ter pelo menos 1 teste automatizado validando o critério de aceite principal.
- **Smoke Tests:** Obrigatório. Cada novo artefato gerado deve aparecer no manifesto de artefatos esperados.
- **Relatório de Cobertura:** JaCoCo HTML + console summary
- **Documentação:** README.md e CLAUDE.md atualizados com novos artefatos
- **Regressão:** Todos os golden file tests existentes continuam passando
- **Consistência:** Novos templates seguem convenções de naming e estrutura dos existentes
- **TDD Compliance:** Commits mostram padrão test-first. Refactoring explícito após green. Testes incrementais via TPP.
- **Double-Loop TDD:** Acceptance tests derivados dos cenários Gherkin (outer loop). Unit tests guiados por TPP (inner loop).

## 4. Regras de Negócio Transversais (Source of Truth)

| ID | Título | Descrição |
| :--- | :--- | :--- |
| **[RULE-001]** | Template Consistency | Todos os novos templates DEVEM seguir a estrutura e convenções dos templates existentes no diretório `resources/`. Naming: `kebab-case` para diretórios, `UPPER-CASE` para templates de documentação (`_TEMPLATE-*.md`), `SKILL.md` para skills, frontmatter YAML para skills invocáveis. |
| **[RULE-002]** | Assembler Integration | Todo novo artefato gerado DEVE ter um assembler correspondente registrado no `AssemblerPipeline`. O assembler DEVE ser condicional quando o artefato depende de configuração específica (ex: compliance, database, message_broker). |
| **[RULE-003]** | Pebble Template Variables | Templates DEVEM usar variáveis Pebble existentes (`{{LANGUAGE}}`, `{{FRAMEWORK}}`, `{{ARCHITECTURE}}`, `{{PROJECT_NAME}}`, etc.) e NÃO inventar variáveis novas sem registrá-las no `TemplateVariable` enum. |
| **[RULE-004]** | Conditional Generation | Artefatos que dependem de configuração específica DEVEM ser gerados condicionalmente. Exemplo: compliance KP só é gerado se `security.compliance` estiver configurado; database patterns só se `data.database.type != none`. |
| **[RULE-005]** | Golden File Compatibility | Nenhuma alteração pode quebrar golden file tests existentes. Novos artefatos DEVEM ser adicionados aos manifestos de artefatos esperados em `tests/golden/`. |
| **[RULE-006]** | Multi-Target Output | Cada novo skill/agent/rule DEVE gerar artefatos para os 3 targets: Claude Code (`.claude/`), GitHub Copilot (`.github/`), e OpenAI Codex (`.codex/`). Knowledge packs geram apenas para Claude e GitHub. |
| **[RULE-007]** | Knowledge Pack Structure | Knowledge packs DEVEM seguir a estrutura: `skills-templates/{name}/SKILL.md` com frontmatter `user-invocable: false`, conteúdo organizado em seções com headers `##`, referências em `references/` subdirectory quando aplicável. |
| **[RULE-008]** | Skill Invocability | Skills invocáveis DEVEM ter frontmatter com `user-invocable: true`, `argument-hint` descritivo, lista de `allowed-tools`, e seções padronizadas (Description, Workflow, Integration Notes). |
| **[RULE-009]** | Agent Persona Contract | Agents DEVEM definir: role, expertise areas, numbered checklist (mínimo 15 pontos), severity classification (CRITICAL/MEDIUM/LOW), e integration notes com skills que os utilizam. |
| **[RULE-010]** | Backward Compatibility | Todas as alterações em templates e assemblers existentes DEVEM manter backward compatibility. Novos campos em templates são aditivos. Remoção de seções existentes é proibida. |
