## 1. Visao Geral

**Chave Jira:** EPIC-0009

O EPIC-002 estabeleceu o suporte basico ao OpenAI Codex com 3 assemblers: `CodexAgentsMdAssembler` (AGENTS.md na raiz), `CodexConfigAssembler` (.codex/config.toml) e `CodexSkillsAssembler` (.agents/skills/). No entanto, a estrutura gerada ainda nao alcanca paridade completa com o padrao do Codex CLI. Este epico fecha as lacunas restantes para que o ia-dev-environment gere **todos** os artefatos nativos do Codex.

As lacunas identificadas sao:

1. **`.codex/skills/`** — O Codex descobre skills em `.codex/skills/` alem de `.agents/skills/`. Atualmente so geramos `.agents/skills/`.
2. **Subagent definitions `[agents.*]` em config.toml** — O Codex configura subagents via secoes `[agents.role_name]` no config.toml (equivalente a `.claude/agents/`). Atualmente o config.toml nao inclui definicoes de agentes.
3. **`.codex/requirements.toml`** — Arquivo de constraints administrativas (approval_policy, sandbox_mode). Sem equivalente no Claude Code.
4. **`AGENTS.override.md`** — Template documentado para override de instrucoes por subdiretorio. Mecanismo nativo do Codex.
5. **Secao Security Baseline no AGENTS.md** — A secao de seguranca no AGENTS.md atual e minimalista. Precisa incluir OWASP Top 10, gestao de segredos e validacao de input quando `security.frameworks` esta configurado.

**Escopo incluido:** Novos assemblers, templates Pebble/Nunjucks, extensoes em assemblers existentes, testes unitarios/integracao, atualizacao de golden files.
**Escopo excluido:** Alteracoes no formato YAML de configuracao, multi-agent orchestration real do Codex, implementacao de MCP servers, alteracao de output `.claude/` ou `.github/`.

## 5. Regras de Negocio Transversais (Source of Truth)

> Regras que se aplicam a multiplas historias. Cada historia referencia as regras pelo ID. Alteracoes de regra propagam automaticamente para todas as historias dependentes.

| ID | Titulo | Descricao |
| :--- | :--- | :--- |
| **[RULE-201]** | Dual output de skills | O `CodexSkillsAssembler` deve gerar skills em **dois** caminhos: `.agents/skills/` (existente) e `.codex/skills/` (novo). Ambos devem ter conteudo identico. O Codex CLI descobre skills em ambos os caminhos; `.codex/skills/` tem precedencia. |
| **[RULE-202]** | Subagents derivados deterministicamente | As secoes `[agents.*]` no config.toml devem ser derivadas dos agents ja gerados em `.claude/agents/`. Cada agente mapeado como `[agents.{nome}]` com `description` extraida do arquivo. Sem input adicional do usuario. |
| **[RULE-203]** | requirements.toml seguro por padrao | O `requirements.toml` deve enforcar constraints de seguranca: `approval_policy` minimo `"on-request"`, `sandbox_mode` minimo `"workspace-write"`. Valores derivados do config do projeto. |
| **[RULE-204]** | AGENTS.override.md documentado | O `AGENTS.override.md` gerado deve conter comentarios explicativos sobre o mecanismo de override do Codex (substitui completamente o AGENTS.md do mesmo diretorio). O conteudo default e vazio (apenas documentacao). |
| **[RULE-205]** | Security Baseline condicional | A secao Security Baseline no AGENTS.md so e incluida quando `security.frameworks` nao esta vazio. Deve cobrir: OWASP Top 10, gestao de segredos, validacao de input, e headers de seguranca. |
| **[RULE-206]** | Impacto zero no output existente | Extensoes nos assemblers Codex NAO devem alterar nenhum byte do output `.claude/`, `.github/`, ou dos artefatos Codex ja gerados no EPIC-002 (AGENTS.md, config.toml sem agents, .agents/skills/). Testes de regressao obrigatorios. |
| **[RULE-207]** | Padrao de extensao do pipeline | Novos assemblers devem implementar a interface `Assembler`, ser inseridos na posicao correta do pipeline (apos CodexSkillsAssembler, antes de DocsAdrAssembler), e retornar `AssemblerResult` com files + warnings. |
| **[RULE-208]** | TOML e Markdown via template | Todos os artefatos (`.toml`, `.md`) devem ser gerados via template Nunjucks, nao via serializacao programatica, para manter controle total do formato e comentarios inline. |
| **[RULE-209]** | Paridade de placeholders | Os mesmos 25 campos de placeholder context devem ser reutilizados nos novos templates. Campos adicionais necessarios (agents_list, security_rules) ja existem no context estendido do EPIC-002. |
| **[RULE-210]** | Golden files obrigatorios | Toda mudanca no output gerado deve atualizar golden files para os 8 perfis. Testes byte-for-byte confirmam consistencia. |
