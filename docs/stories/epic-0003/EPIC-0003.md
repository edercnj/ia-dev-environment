# Épico: Migração do Ciclo de Desenvolvimento para TDD (Test-Driven Development)

**Autor:** Product Owner / Architect
**Data:** 2026-03-15
**Versão:** 1.0
**Status:** Em Refinamento

## 1. Visão Geral

**Chave Jira:** EPIC-0003

Migrar todo o ciclo de desenvolvimento orquestrado pelo `ia-dev-environment` para TDD
(Test-Driven Development), aplicando o padrão Red-Green-Refactor em todas as fases.
Isso inclui reestruturar o lifecycle de 8 fases, promover o test plan a driver de
implementação, inverter a relação test plan → task decomposition, adotar Double-Loop
TDD (acceptance + unit), garantir commits atômicos por ciclo TDD, enriquecer o Gherkin
das stories com cenários de degenerate cases, boundary values e error paths, e adicionar
critérios TDD às reviews. O escopo abrange skills, rules, knowledge packs, agents e
templates — todos na camada `resources/` (source of truth), com propagação para as
cópias `.claude/` e `.github/`.

## 3. Definições de Qualidade Globais

### Global Definition of Ready (DoR)

- Spec de migração TDD lida e compreendida por todos os envolvidos
- Knowledge packs de testing e coding-standards acessíveis e versionados
- Templates _TEMPLATE-STORY.md e _TEMPLATE-EPIC.md existentes e versionados
- Convenção SD-09 de diretórios documentada e seguida
- Regras de dual copy (resources/skills-templates/core/ + resources/github-skills-templates/) compreendidas

### Global Definition of Done (DoD)

- **Cobertura:** ≥ 95% Line, ≥ 90% Branch (thresholds existentes mantidos)
- **Testes Automatizados:** Unitários, integração (byte-for-byte golden file tests). Todos os cenários Gherkin cobertos.
- **TDD Compliance:** Commits mostram padrão test-first (teste precede implementação no git log). Existe refactoring explícito após green. Testes são incrementais (do simples ao complexo via TPP).
- **Relatório de Cobertura:** Vitest coverage report com thresholds validados no CI
- **Documentação:** Skills, KPs, rules e agents atualizados em ambas as cópias (resources/skills-templates/core/ + resources/github-skills-templates/)
- **Backward Compatibility:** Projetos já gerados continuam funcionando. Novas seções TDD são aditivas.
- **Paralelismo Preservado:** Subagents em layers independentes continuam sendo lançados em single message

## 4. Regras de Negócio Transversais (Source of Truth)

> Regras que se aplicam a múltiplas histórias. Cada história referencia as regras pelo ID. Alterações de regra propagam automaticamente para todas as histórias dependentes.

| ID | Título | Descrição |
| :--- | :--- | :--- |
| **[RULE-001]** | Dual Copy Consistency | Todo recurso (skill, KP, agent) existe em `resources/skills-templates/core/` E `resources/github-skills-templates/`. Ambas as cópias DEVEM ser atualizadas juntas. Mudança em uma sem a outra é violação. |
| **[RULE-002]** | Source of Truth é resources/ | Todas as mudanças devem ser feitas em `resources/`. Os diretórios `.claude/` e `.github/` são outputs gerados pelo CLI `ia-dev-env`. Nunca editar os outputs diretamente. |
| **[RULE-003]** | Backward Compatibility | Projetos já gerados devem continuar funcionando. Novas seções TDD são aditivas — não remover nem renomear seções existentes. Novos campos em templates devem ter defaults seguros. |
| **[RULE-004]** | Coverage Thresholds Mantidos | Thresholds de cobertura permanecem: ≥ 95% line, ≥ 90% branch. TDD naturalmente atinge esses valores, mas eles continuam sendo validados explicitamente. |
| **[RULE-005]** | Red-Green-Refactor Cycle | Toda unidade de implementação segue: (1) RED — escrever teste que falha, (2) GREEN — implementar o mínimo para passar, (3) REFACTOR — melhorar design sem quebrar testes. Refactoring NUNCA adiciona comportamento. |
| **[RULE-006]** | Transformation Priority Premise (TPP) | Cenários de teste devem ser ordenados por complexidade crescente: (1) degenerate cases (null, empty, zero, constant), (2) unconditional paths, (3) simple conditions (if/else), (4) complex conditions, (5) iterations/recursions, (6) edge cases e boundary values. |
| **[RULE-007]** | Double-Loop TDD | Loop externo (Acceptance): derivado do Gherkin da story, escrito PRIMEIRO, fica vermelho durante todo o desenvolvimento, só fica verde no final.<br>Loop interno (Unit): Red-Green-Refactor rápido para cada unidade. Cada ciclo interno aproxima o acceptance test de ficar verde. |
| **[RULE-008]** | Atomic TDD Commits | Cada commit representa UM ciclo Red-Green-Refactor completo. Teste e implementação no mesmo commit. Formato: `test(scope): add test for X` ou `feat(scope): implement X [TDD]`. Commits devem ser pequenos e atômicos. |
| **[RULE-009]** | Parallel Subagent Preservation | Subagents que operam em layers independentes DEVEM continuar sendo lançados em single message (paralelo). A migração TDD NÃO pode serializar o que hoje roda em paralelo sem justificativa técnica documentada. |
| **[RULE-010]** | Gherkin Completeness | Toda story deve incluir cenários de: (1) degenerate cases (null, empty, zero), (2) boundary values como triplet (at-min, at-max, past-max), (3) error paths completos. Mínimo de 4 cenários por story. |
| **[RULE-011]** | SD-09 Directory Convention | Todos os artefatos em `docs/stories/epic-XXXX/` com subdiretórios `plans/` e `reviews/`. Nomenclatura: `story-XXXX-YYYY.md`, `EPIC-XXXX.md`, `IMPLEMENTATION-MAP.md`. |
| **[RULE-012]** | Generated Content Language | Corpo em Português Brasileiro (pt-BR). Termos técnicos, identificadores de código, nomes de campos e IDs em Inglês. Gherkin em PT-BR: `Cenario`, `DADO`, `QUANDO`, `ENTÃO`, `E`, `MAS`. |
