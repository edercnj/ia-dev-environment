# Épico: Integração Jira para Geração de Épicos e Histórias

**Autor:** Product Owner
**Data:** 2026-03-25
**Versão:** 1.0
**Status:** Em Refinamento

## 1. Visão Geral

**Chave Jira:** —

Este épico adiciona integração opcional com o Jira ao pipeline de geração de épicos e histórias do ia-dev-environment. Quando o usuário invocar os skills de decomposição (`/x-story-epic`, `/x-story-create`, `/x-story-epic-full`), o sistema perguntará se deseja criar os artefatos também no Jira via Atlassian MCP. A integração é não-bloqueante: falhas no Jira nunca impedem a geração do markdown. Quando o usuário opta por criar o épico no Jira, todas as histórias filhas são criadas automaticamente (cascade). Os templates de história e implementation map são atualizados para incluir o campo `Chave Jira`.

**Escopo incluído:**
- Modificação dos templates `_TEMPLATE-STORY.md` e `_TEMPLATE-IMPLEMENTATION-MAP.md`
- Modificação dos 4 skills de geração (epic, create, epic-full, map)
- Espelhamento das mudanças nos counterparts GitHub Copilot
- Atualização de golden files e testes de integração

**Escopo excluído:**
- Criação automática de sub-tarefas no Jira (seriam 75-150+ issues; markdown checklist é suficiente)
- Sincronização bidirecional Jira → markdown (apenas markdown → Jira)
- Configuração do MCP server do Jira (responsabilidade do ambiente do usuário)

## 3. Definições de Qualidade Globais

### Global Definition of Ready (DoR)

- Templates de epic, story e implementation map lidos e compreendidos
- Skills atuais de geração revisados (workflow completo)
- Atlassian MCP tools disponíveis no ambiente de desenvolvimento
- Counterparts GitHub Copilot identificados e mapeados

### Global Definition of Done (DoD)

- **Cobertura:** ≥ 95% Line, ≥ 90% Branch
- **Testes Automatizados:** Unit tests para lógica de templates, integration tests para golden files
- **Relatório de Cobertura:** JaCoCo HTML + console summary
- **Documentação:** Skills atualizados com instruções de integração Jira
- **Backward Compatibility:** Skills funcionam normalmente quando Jira MCP não está disponível (skip silencioso)
- **TDD Compliance:** Commits mostram padrão test-first (teste precede implementação no git log), refactoring explícito após green, testes incrementais via TPP
- **Double-Loop TDD:** Acceptance tests derivados dos cenários Gherkin (outer loop), unit tests guiados por TPP (inner loop)

## 4. Regras de Negócio Transversais

| ID | Título | Descrição |
| :--- | :--- | :--- |
| RULE-001 | Integração Não-Bloqueante | Falhas na criação de issues no Jira NUNCA devem bloquear a geração do arquivo markdown. O skill deve avisar o usuário, registrar `<CHAVE-JIRA>` como `—` ou mensagem de falha, e continuar normalmente. |
| RULE-002 | Verificação Silenciosa de MCP | Antes de perguntar ao usuário sobre Jira, verificar se o tool MCP do Jira está disponível. Se não estiver, pular toda a integração silenciosamente — sem warning, sem prompt ao usuário. |
| RULE-003 | Cascade de Decisão | Quando o usuário responde "Sim" para criar o épico no Jira no orchestrator (`x-story-epic-full`), todas as histórias devem ser criadas no Jira automaticamente sem prompts adicionais. A decisão propaga via `jiraContext`. |
| RULE-004 | Dual-Mode (Standalone vs Cascaded) | O skill `x-story-create` opera em dois modos: (A) cascaded — recebe `jiraContext` do orchestrator e age silenciosamente; (B) standalone — pergunta ao usuário diretamente. O comportamento muda com base na presença do `jiraContext`. |
| RULE-005 | Placeholder Consistency | O placeholder `<CHAVE-JIRA>` deve ser substituído em todos os casos: por uma chave real do Jira (ex: `PROJ-123`), por `—` (quando Jira não habilitado), ou por mensagem de erro (quando falha na criação). Nunca deixar o placeholder literal no output final. |
| RULE-006 | Labels de Rastreabilidade | Toda issue criada no Jira via integração deve incluir a label `generated-by-ia-dev-env` para permitir filtragem e rastreabilidade. |
| RULE-007 | Dependency Linking Best-Effort | A criação de links de dependência entre issues no Jira (Blocked By / Blocks) é uma operação best-effort executada em segundo passo, após todas as stories terem chaves Jira. Falhas no linking são logadas mas não bloqueiam o pipeline. |
