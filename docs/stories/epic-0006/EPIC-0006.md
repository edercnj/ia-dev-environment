# Epico: Migracao ia-dev-env — Node.js/TypeScript para Java 21

**Autor:** Tech Lead
**Data:** 2026-03-19
**Versao:** 1.0.0
**Status:** Em Refinamento

## 1. Visao Geral

**Chave Jira:** EPIC-0006

Migrar a CLI `ia-dev-environment` de Node.js/TypeScript 5 para Java 21, mantendo **100% de paridade funcional** com a implementacao TypeScript existente. A ferramenta gera boilerplate `.claude/`, `.github/`, `.codex/` e `.agents/` para ambientes de desenvolvimento assistidos por IA. A versao Java DEVE produzir output **byte-a-byte identico** ao da versao TypeScript para todos os 8 perfis bundled.

**Escopo incluido:**
- Porta de todos os 86 arquivos-fonte TypeScript (~10.734 linhas) para Java 21
- 17 data classes, 23 assemblers, domain logic, checkpoint system, implementation map parser, progress reporter
- CLI com Picocli (generate, validate, interactive mode)
- Template engine Pebble com compatibilidade Jinja2/Nunjucks
- Golden file tests para os 8 perfis (go-gin, java-quarkus, java-spring, kotlin-ktor, python-click-cli, python-fastapi, rust-axum, typescript-nestjs)
- Empacotamento como fat JAR e, opcionalmente, native image via GraalVM

**Escopo excluido:**
- Features novas alem do que a versao TypeScript possui
- Modo API server (REST/gRPC) — apenas CLI
- Sistema de plugins
- Paridade com a versao Python historica

## 4. Regras de Negocio Transversais (Source of Truth)

> Regras que se aplicam a multiplas historias. Cada historia referencia as regras pelo ID. Alteracoes de regra propagam automaticamente para todas as historias dependentes.

| ID | Titulo | Descricao |
| :--- | :--- | :--- |
| **[RULE-001]** | Paridade Byte-a-Byte | O output gerado pela versao Java DEVE ser identico ao output da versao TypeScript para todos os 8 perfis bundled. Validado via comparacao de golden files. Diferencas de whitespace, line endings ou ordenacao de campos sao consideradas falhas. |
| **[RULE-002]** | Formatacao Python-Bool | Valores booleanos no contexto de template DEVEM ser renderizados como `"True"`/`"False"` (strings Python-style), NAO `"true"`/`"false"` (Java-style). Isso e necessario para manter compatibilidade com os templates Nunjucks/Jinja2 existentes que usam convencao Python. |
| **[RULE-003]** | Factory Method fromMap() | Toda data class DEVE ter um metodo estatico `fromMap(Map<String, Object>)` para desserializacao de YAML, replicando o padrao `fromDict()` do TypeScript. O metodo DEVE tratar tipos ausentes com defaults sensiveis e lancar `ConfigValidationException` para valores invalidos. |
| **[RULE-004]** | Interface Assembler Uniforme | Todo assembler DEVE implementar a interface `Assembler` com contrato padronizado. O metodo `assemble()` recebe `ProjectConfig`, `TemplateEngine` e `Path` (output dir), retornando `List<String>` (caminhos dos arquivos gerados). Assemblers NAO podem ter efeitos colaterais alem da escrita de arquivos. |
| **[RULE-005]** | Ordem de Execucao Pipeline | Os 23 assemblers DEVEM ser executados na ordem fixa RULE-008: Rules → Skills → Agents → Patterns → Protocols → Hooks → Settings → GithubInstructions → GithubMcp → GithubSkills → GithubAgents → GithubHooks → GithubPrompts → Docs → GrpcDocs → Runbook → CodexAgentsMd → CodexConfig → CodexSkills → DocsAdr → Cicd → EpicReport → Readme. Alterar a ordem pode causar dependencias entre artefatos gerados. |
| **[RULE-006]** | Cobertura JaCoCo | Maven build DEVE falhar se cobertura ficar abaixo de 95% line ou 90% branch. Configurar `jacoco-maven-plugin` com `check` goal no phase `verify`. Excluir apenas classes geradas (se houver) e a classe `main()`. |
| **[RULE-007]** | Zero Dependencia de Framework no Dominio | Pacotes `model` e `domain` NAO podem importar classes de Picocli, Pebble, Jackson, SnakeYAML ou JLine diretamente. Toda interacao com frameworks externos DEVE ser via interfaces (ports) definidas no dominio. Isso garante testabilidade e isolamento. |
| **[RULE-008]** | Output Atomico | Toda escrita de arquivos DEVE usar diretorio temporario + `Files.move()` com `ATOMIC_MOVE`. Em caso de falha, o diretorio temporario DEVE ser preservado para debug e o caminho reportado ao usuario. NAO usar `REPLACE_EXISTING` sem `--force`. |
| **[RULE-009]** | Compatibilidade Cross-Platform | Usar `java.nio.file.Path` em todo lugar (nunca concatenar strings com `/` ou `\\`). Line endings DEVEM ser LF (`\n`) em todos os arquivos gerados, independente do SO. Encoding DEVE ser UTF-8. Testar no macOS e Linux no minimo. |
| **[RULE-010]** | Contexto de Template 25 Campos | O `ContextBuilder` DEVE produzir exatamente os mesmos 25 campos do TypeScript `buildDefaultContext()`, com os mesmos nomes (`project_name`, `language_name`, etc.) e tipos. Campos booleanos seguem RULE-002. Campos ausentes no config DEVEM ter defaults identicos ao TypeScript. |
| **[RULE-011]** | Rejeicao de Caminhos Perigosos | Diretorios home (`~`, `$HOME`), root (`/`), e diretorios de sistema (`/usr`, `/etc`, `/var`, `/bin`, `/sbin`) DEVEM ser rejeitados como destino de output. Lancar `CliException` com mensagem explicativa. |
| **[RULE-012]** | Deteccao de Overwrite | Antes de gerar, verificar existencia de `.claude/`, `.github/`, `.codex/`, `.agents/`, `docs/` no destino. Se existirem e `--force` NAO for fornecido, abortar com mensagem listando conflitos e sugerindo `--force`. |
