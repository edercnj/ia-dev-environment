## 1. Visão Geral

**Chave Jira:** EPIC-0008

Este épico abrange a correção de todos os problemas identificados no Codebase Audit Report de 2026-03-20, que atribuiu nota **32/100** ao projeto. O escopo inclui: extração de utilitários duplicados (DRY), eliminação de `return null`, unificação de `buildContext()`, divisão de 24 classes acima de 250 linhas, decomposição de 50+ métodos acima de 25 linhas, correções de segurança, padronização de testes e documentação de desvios arquiteturais. O objetivo é elevar a nota do audit para ≥ 85/100.

**Escopo incluído:** Todas as 49 findings (7 CRITICAL, 24 MEDIUM, 18 LOW) do audit report.
**Escopo excluído:** Funcionalidades novas, migração de arquitetura (hexagonal completa), alterações no `pom.xml` de dependências.

## 3. Definições de Qualidade Globais

### Global Definition of Ready (DoR)

- Audit report lido e finding correspondente identificado
- Arquivos afetados mapeados com números de linha
- Dependências da story (Blocked By) estão concluídas
- Padrão de refactoring definido (extract method, extract class, replace null with Optional, etc.)
- Golden files atualizados se a saída do gerador muda

### Global Definition of Done (DoD)

- **Cobertura:** ≥ 95% Line, ≥ 90% Branch (mantida ou melhorada)
- **Testes Automatizados:** Todos os testes existentes passando. Novos testes para lógica extraída.
- **Relatório de Cobertura:** JaCoCo report gerado via `mvn verify`
- **Documentação:** Javadoc atualizado quando assinaturas de métodos mudam
- **Persistência:** N/A
- **Performance:** Sem degradação mensurável no tempo de geração
- **TDD Compliance:** Commits mostram padrão test-first. Refactoring explícito após green. Testes incrementais (do simples ao complexo via TPP — Transformation Priority Premise).
- **Double-Loop TDD:** Testes de aceitação derivados dos cenários Gherkin (outer loop). Testes unitários guiados por TPP (inner loop).
- **Compilação limpa:** Zero warnings do compilador Java 21

## 4. Regras de Negócio Transversais (Source of Truth)

> Regras que se aplicam a múltiplas histórias. Cada história referencia as regras pelo ID.

| ID | Título | Descrição |
| :--- | :--- | :--- |
| **[RULE-001]** | Cobertura obrigatória | Todas as mudanças devem manter cobertura ≥ 95% line e ≥ 90% branch. Se a extração de método/classe reduz cobertura, adicionar testes antes do refactoring. |
| **[RULE-002]** | Comportamento externo inalterado | Refactoring não deve alterar o comportamento observável. Todos os 1.814 testes existentes devem continuar passando. Golden files devem ser regenerados se a saída muda. |
| **[RULE-003]** | Commits atômicos | Cada story produz commits atômicos com Conventional Commits (`refactor:`, `fix:`, `test:`, `docs:`). Cada commit compila e todos os testes passam. |
| **[RULE-004]** | Limites de tamanho | Classe ≤ 250 linhas. Método ≤ 25 linhas. Parâmetros ≤ 4 (usar parameter object se mais). Linha ≤ 120 caracteres. |
| **[RULE-005]** | Null proibido | Nunca retornar `null` — usar `Optional<T>`, coleção vazia, ou Result type. Nunca passar `null` como argumento. |
| **[RULE-006]** | Domínio puro | Pacotes `domain.*` e `model.*` não importam classes de framework (Jackson, Pebble, SnakeYAML, picocli). Apenas standard library. |
| **[RULE-007]** | DRY absoluto | Zero duplicação de código entre classes. Extrair para utilitários compartilhados (`CopyHelpers`, `JsonHelpers`). |
| **[RULE-008]** | Nomenclatura de testes | Formato: `[methodUnderTest]_[scenario]_[expectedBehavior]`. Sem camelCase genérico. |
| **[RULE-009]** | String formatting | Usar `String.formatted()` ou `String.format()` para mensagens de erro. Nunca concatenação com `+`. |
| **[RULE-010]** | Golden files | Quando a saída do gerador muda, atualizar golden files via `GoldenFileRegenerator` e verificar byte-for-byte no commit. |
