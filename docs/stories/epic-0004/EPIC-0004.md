# Épico: Evolução do Feature Lifecycle — Documentação, Arquitetura e Entrega

**Autor:** Product Owner / Architect
**Data:** 2026-03-15
**Versão:** 1.0
**Status:** Em Refinamento

## 1. Visão Geral

**Chave Jira:** EPIC-0004

Evoluir o ciclo de desenvolvimento (Feature Lifecycle) orquestrado pelo `ia-dev-environment` para
cobrir as lacunas entre implementação e entrega em produção. O escopo inclui: criação de uma fase
dedicada de documentação automática no lifecycle (`x-dev-lifecycle`), uma nova skill de planejamento
de arquitetura (`x-dev-architecture-plan`), templates padronizados para ADRs, documentação de
arquitetura de serviço e runbooks de deploy, geradores de documentação por tipo de interface
(REST/OpenAPI, gRPC, CLI, Event-Driven), melhorias nos diagramas Mermaid da skill `x-story-create`,
integração do architecture plan no lifecycle completo, geração de artefatos CI/CD, automação de ADRs,
verificação pós-deploy, documentação de threat model e tracking de performance baseline. Todas as
mudanças são aditivas — skills e fases existentes permanecem intactas. Os artefatos são gerados
como templates em `resources/` (source of truth) e propagados para `.claude/` e `.github/`.

## 3. Definições de Qualidade Globais

### Global Definition of Ready (DoR)

- Spec de evolução do lifecycle lida e compreendida por todos os envolvidos
- Knowledge packs de architecture, protocols, security, observability acessíveis e versionados
- Templates existentes (_TEMPLATE-STORY.md, _TEMPLATE-EPIC.md, _TEMPLATE.md) versionados
- Convenção SD-09 de diretórios documentada e seguida
- Regras de dual copy (resources/skills-templates/core/ + resources/github-skills-templates/) compreendidas
- x-dev-lifecycle atual (8 fases, 0-7) compreendido em detalhe
- x-story-create e sua geração de diagramas compreendidos

### Global Definition of Done (DoD)

- **Cobertura:** ≥ 95% Line, ≥ 90% Branch
- **Testes Automatizados:** Unitários, integração (byte-for-byte golden file tests). Cenários Gherkin cobertos.
- **TDD Compliance:** Commits mostram padrão test-first. Refactoring explícito após green. Testes incrementais via TPP.
- **Double-Loop TDD:** Acceptance tests derivados do Gherkin (outer loop). Unit tests guiados por TPP (inner loop).
- **Relatório de Cobertura:** Vitest coverage report com thresholds validados
- **Documentação:** Skills, KPs, templates atualizados em ambas as cópias (resources/ + .claude/ + .github/)
- **Backward Compatibility:** Projetos existentes continuam funcionando. Novas fases/skills são aditivas.
- **Paralelismo Preservado:** Subagents em layers independentes continuam sendo lançados em single message

## 4. Regras de Negócio Transversais (Source of Truth)

> Regras que se aplicam a múltiplas histórias. Cada história referencia as regras pelo ID. Alterações de regra propagam automaticamente para todas as histórias dependentes.

| ID | Título | Descrição |
| :--- | :--- | :--- |
| **[RULE-001]** | Dual Copy Consistency | Todo recurso (skill, KP, agent, template) existe em `resources/skills-templates/core/` E `resources/github-skills-templates/`. Ambas as cópias DEVEM ser atualizadas juntas. Mudança em uma sem a outra é violação. |
| **[RULE-002]** | Source of Truth é resources/ | Todas as mudanças são feitas em `resources/`. Os diretórios `.claude/` e `.github/` são outputs gerados pelo CLI `ia-dev-env`. Nunca editar outputs diretamente. |
| **[RULE-003]** | Backward Compatibility | Projetos já gerados devem continuar funcionando. Novas fases, skills e templates são aditivos — não remover nem renomear artefatos existentes. Novos campos em templates devem ter defaults seguros. |
| **[RULE-004]** | Interface-Aware Generation | Geradores de documentação checam o campo `interfaces` do project identity (cli, rest, grpc, graphql, websocket, kafka) para determinar quais tipos de doc gerar. Geradores para interfaces não configuradas são no-ops silenciosos. Nunca gerar documentação para interface inexistente no projeto. |
| **[RULE-005]** | Template-Based Artifacts | Todo documento gerado (ADR, architecture doc, runbook, API doc) segue template padronizado em `resources/templates/`. Nunca gerar documentação freeform. Templates usam placeholders `{{PLACEHOLDER}}` resolvidos em runtime. |
| **[RULE-006]** | ADR Sequential Numbering | ADRs usam numeração sequencial `ADR-NNNN-titulo.md` a partir de 0001. O índice de ADRs (`docs/adr/README.md`) é auto-atualizado a cada novo ADR criado. Nunca reutilizar ou pular números. Referência cruzada entre ADRs e stories é obrigatória. |
| **[RULE-007]** | Mermaid Diagram Mandatory for Flows | Stories com fluxos de aplicação (request→response, event→processing, CLI→execution) DEVEM incluir diagramas Mermaid de sequência mostrando interação inter-camadas (inbound adapter → application → domain → outbound adapter). Stories de infraestrutura usam deployment diagrams. |
| **[RULE-008]** | Incremental Architecture Updates | Documentos de arquitetura de serviço são documentos vivos. Cada feature que altera arquitetura atualiza (não substitui) o documento existente. Mudanças são adicionadas como novas seções ou emendas a seções existentes. Histórico de mudanças é preservado. |
| **[RULE-009]** | Documentation Output Convention | Artefatos de documentação gerados vão para `docs/` com subdiretórios organizados: `docs/adr/`, `docs/architecture/`, `docs/api/`, `docs/runbook/`. Nunca misturar tipos de doc no mesmo diretório. Changelog entries vão para `CHANGELOG.md` na raiz. |
| **[RULE-010]** | Lifecycle Phase Integrity | Novas fases no `x-dev-lifecycle` mantêm compatibilidade com a estrutura existente. A contagem total de fases e a regra "NEVER stop before final phase" são atualizadas. Skills invocadas por novas fases seguem o mesmo padrão de delegação (subagent ou skill invocation). |
| **[RULE-011]** | SD-09 Directory Convention | Artefatos em `docs/stories/epic-XXXX/` com subdiretórios `plans/` e `reviews/`. Nomenclatura: `story-XXXX-YYYY.md`, `EPIC-XXXX.md`, `IMPLEMENTATION-MAP.md`. |
| **[RULE-012]** | Generated Content Language | Corpo em Português Brasileiro (pt-BR). Termos técnicos, identificadores de código, nomes de campos e IDs em Inglês. Gherkin em PT-BR: `Cenario`, `DADO`, `QUANDO`, `ENTÃO`, `E`, `MAS`. |
