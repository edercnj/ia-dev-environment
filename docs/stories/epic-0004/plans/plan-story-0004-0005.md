## 2. New Sections to Create

A new `## Phase 3 — Documentation` section must be inserted into both template files. This section contains:

### 2.1 Phase 3 — Documentation (New Section)

Position: Between current Phase 2 (Implementation) and current Phase 3 (Review).

Content structure:

```markdown
## Phase 3 — Documentation (Orchestrator — Inline)

1. Read `interfaces` field from project identity (Rule 01)
2. Build list of documentable interfaces: `rest`, `grpc`, `graphql`, `cli`, `websocket`, `event-consumer`, `event-producer`
3. For each configured interface, dispatch the corresponding documentation generator:
   - `rest` → OpenAPI/Swagger generator (story-0004-0007)
   - `grpc` → gRPC/Proto doc generator (story-0004-0008)
   - `cli` → CLI doc generator (story-0004-0009)
   - `websocket`, `event-consumer`, `event-producer` → Event-Driven doc generator (story-0004-0010)
4. If no documentable interfaces configured: emit log `"No documentable interfaces configured. Skipping interface documentation."`
5. Generate changelog entry (ALWAYS, regardless of interfaces):
   - Read commits since branch point (`git log main..HEAD --oneline`)
   - Group by Conventional Commits type (feat, fix, refactor, test, docs, chore)
   - Append formatted entry to `CHANGELOG.md`
6. Documentation output saved to `docs/` with subdirectories per type (RULE-009):
   - API docs → `docs/api/`
   - Architecture docs → `docs/architecture/`
```

## 5. Integration Points

### 5.1 How Phase 3 Integrates with the Lifecycle Flow

- **Predecessor:** Phase 2 (Implementation) must complete before Phase 3 starts. The implementation produces the code changes that documentation generators will introspect.
- **Successor:** Phase 4 (Review, formerly Phase 3) now reviews both code AND generated documentation.
- **Parallel potential:** Interface documentation generators (REST, gRPC, CLI, Event) can run in parallel since they produce independent outputs. The changelog generator runs after all interface generators complete.

### 5.2 Skills Invoked by Phase 3

Phase 3 will eventually invoke documentation generator skills (stories 0007-0010). For this story, the phase defines the **dispatch skeleton** — the actual generator implementations are stubs that will be filled by downstream stories.

## 10. Risk Assessment

### 10.1 Backward Compatibility (RULE-003) — MEDIUM RISK

**Risk:** Phase renumbering could break external references (scripts, documentation, developer habits).

**Mitigation:**
- Phase renumbering is internal to the SKILL.md template. The lifecycle is executed by AI agents that read the template fresh each time.
- No external scripts reference phase numbers — they are human-readable milestones.
- The SKILL.md is regenerated for every project — old projects keep their old SKILL.md until regenerated.
- When regenerated, the new numbering takes effect seamlessly.

**Residual risk:** LOW after mitigation.

### 10.2 Golden File Divergence — LOW RISK

**Risk:** All 8 golden files must be updated simultaneously. Missing one causes byte-for-byte test failure.

**Mitigation:**
- Run the pipeline for all 8 profiles and regenerate golden files in a single operation.
- The `byte-for-byte.test.ts` integration test validates all profiles automatically.

### 10.3 Dual Copy Inconsistency (RULE-001) — LOW RISK

**Risk:** The Claude template and GitHub template have slightly different structures (frontmatter format, path references). Changes must be applied to both with awareness of their differences.

**Mitigation:**
- Both templates are in `resources/` (RULE-002).
- The Claude template uses `skills/` paths; the GitHub template uses `.github/skills/` paths.
- The phase content is identical between both; only the YAML frontmatter and reference paths differ.
- Review both files side-by-side during implementation.

### 10.4 Downstream Story Compatibility — LOW RISK

**Risk:** Stories 0007-0010 (documentation generators) depend on the dispatch skeleton defined here. If the skeleton changes later, those stories need updating.

**Mitigation:**
- Define the dispatch interface clearly and document it in the Phase 3 content.
- Use simple conditional structure (if interface X configured, invoke generator X).
- Downstream stories only add generator implementations — they do not modify the dispatch skeleton.

### Generator Source Understanding

- **Claude copy:** `SkillsAssembler.assembleCore()` in `src/assembler/skills-assembler.ts` scans `resources/skills-templates/core/` directories and copies them to `skills/` output via `copyTemplateTree()`. The `x-dev-lifecycle/` directory is picked up automatically.
- **GitHub copy:** `GithubSkillsAssembler.assemble()` in `src/assembler/github-skills-assembler.ts` iterates `SKILL_GROUPS["dev"]` which includes `"x-dev-lifecycle"`. It reads `resources/github-skills-templates/dev/x-dev-lifecycle.md` and writes to `skills/x-dev-lifecycle/SKILL.md`.
- **Template engine:** Both use `TemplateEngine.replacePlaceholders()` which substitutes `{{VARIABLE}}` tokens. The lifecycle template uses `{{PROJECT_NAME}}`, `{{LANGUAGE}}`, `{{LANGUAGE_VERSION}}`, `{{COMPILE_COMMAND}}`, `{{TEST_COMMAND}}`, `{{COVERAGE_COMMAND}}` — all runtime markers that are NOT resolved during generation (per the Integration Notes in the template itself).
