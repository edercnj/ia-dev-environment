# Épico: Smoke Tests End-to-End e Integração no Ciclo de Desenvolvimento

**Autor:** Product Owner
**Data:** 2026-03-25
**Versão:** 1.0
**Status:** Em Refinamento

## 1. Visão Geral

**Chave Jira:** —

Este épico implementa uma suíte completa de smoke tests para o ia-dev-env, garantindo que a aplicação produz os artefatos corretos em quantidade, estrutura e conteúdo para todos os 8 perfis bundled. A motivação vem de incidentes anteriores onde: (1) a geração para GitHub Copilot produzia menos arquivos que o esperado, e (2) a migração de TypeScript para Java resultou em contagem de arquivos divergente entre as implementações. Os golden file tests existentes validam paridade byte-a-byte, mas não cobrem cenários de integridade sistêmica como placeholders não resolvidos, arquivos vazios, frontmatter inválido, modos CLI, e consistência cross-profile.

Além da suíte de testes, este épico modifica os skills `x-dev-lifecycle` e `x-dev-epic-implement` para integrar a execução de smoke tests no fluxo de desenvolvimento. Após a implementação de cada história (ou fase de épico), os smoke tests são executados automaticamente antes da etapa de review. Falhas nos smoke tests bloqueiam a progressão e exigem correções imediatas.

**Escopo incluído:**

- Infraestrutura de smoke tests (classe base, utilities, manifesto de artefatos)
- Smoke tests parametrizados para todos os 8 perfis
- Validação de integridade de conteúdo (vazios, placeholders, JSON/YAML)
- Validação de frontmatter e estrutura de skills
- Validação de modos CLI (dry-run, force, verbose, help)
- Validação de consistência cross-profile
- Validação de regressão de assemblers
- Integração no skill `x-dev-lifecycle` (smoke gate antes de review)
- Integração no skill `x-dev-epic-implement` (smoke gate nos integrity gates)
- Skill `/run-smoke` para execução on-demand

**Escopo excluído:**

- Modificação dos golden file tests existentes (eles continuam validando paridade byte-a-byte)
- Testes de performance ou carga
- Testes contra ambientes externos (Jira, GitHub)
- Testes de instalação nativa (GraalVM native-image)

## 3. Definições de Qualidade Globais

### Global Definition of Ready (DoR)

- Pipeline de geração funcional para todos os 8 perfis
- Golden file tests existentes passando
- Estrutura de test resources compreendida (`src/test/resources/golden/`)
- Skills `x-dev-lifecycle` e `x-dev-epic-implement` revisados e compreendidos
- Perfis de build Maven compreendidos (unit-tests, integration-tests, all-tests)

### Global Definition of Done (DoD)

- **Cobertura:** >= 95% Line, >= 90% Branch
- **Testes Automatizados:** Smoke tests executando como integration tests (profile `integration-tests`)
- **Relatório de Cobertura:** JaCoCo HTML + console summary
- **Regressão:** Todos os testes existentes continuam passando
- **Skills Atualizados:** `x-dev-lifecycle` e `x-dev-epic-implement` integram smoke gate
- **TDD Compliance:** Commits mostram padrão test-first (teste precede implementação no git log), refactoring explícito após green, testes incrementais via TPP
- **Double-Loop TDD:** Acceptance tests derivados dos cenários Gherkin (outer loop), unit tests guiados por TPP (inner loop)

## 4. Regras de Negócio Transversais

| ID | Título | Descrição |
| :--- | :--- | :--- |
| RULE-001 | Parametrização por Perfil | Todos os smoke tests que validam output devem ser parametrizados para os 8 perfis bundled. Um teste que passa para 7 perfis e falha para 1 é uma violação. |
| RULE-002 | Independência de Golden Files | Smoke tests NÃO dependem dos golden files existentes. Eles validam propriedades estruturais e de integridade, não paridade byte-a-byte. Os dois tipos de teste são complementares. |
| RULE-003 | Non-Blocking no Pipeline de Geração | Os smoke tests executam APÓS a geração, nunca durante. Eles não alteram o comportamento do pipeline — apenas validam o output. |
| RULE-004 | Smoke Gate Bloqueante | Quando integrado no dev lifecycle ou epic implementation, a falha de qualquer smoke test BLOQUEIA a progressão para a fase seguinte. O desenvolvedor deve corrigir antes de continuar. |
| RULE-005 | Manifesto Versionado | O manifesto de artefatos esperados é versionado junto com o código. Qualquer mudança no pipeline que altere a contagem ou estrutura de arquivos DEVE atualizar o manifesto. |
| RULE-006 | Execução em Temp Directory | Todos os smoke tests DEVEM gerar output em diretórios temporários (`@TempDir`). Nunca escrever no working directory do projeto. |
| RULE-007 | Relatório Estruturado | O skill `/run-smoke` deve produzir relatório com: total de testes, passados, falhados, skipped, tempo de execução, e detalhes das falhas. |
