# Épico: Fortalecimento do TDD como Prática Mandatória no ia-dev-environment

**Autor:** Product Owner
**Data:** 2026-03-27
**Versão:** 1.0
**Status:** Em Refinamento

## 1. Visão Geral

**Chave Jira:** —

O ia-dev-environment declara TDD como mandatório em múltiplos artefatos: Rule 03 ("Red-Green-Refactor is mandatory for all production code"), Rule 05 ("Coverage thresholds are NOT a substitute for TDD — high coverage with test-after is insufficient"), e testing-philosophy.md ("NEVER write production code without a failing test first"). Porém, uma auditoria profunda dos skills de desenvolvimento (`x-dev-lifecycle`, `x-dev-implement`, `x-dev-epic-implement`) e dos templates de histórias revelou **10 gaps críticos** que enfraquecem ou contradizem essa mandatoriedade.

Os gaps se organizam em quatro categorias:

1. **Templates ausentes (3 de 4):** Os templates `_TEMPLATE-STORY.md`, `_TEMPLATE-EPIC.md` e `_TEMPLATE-IMPLEMENTATION-MAP.md` são referenciados em 7+ skills mas não existem em `.claude/templates/`. A regra SD-08 proíbe explicitamente hardcoding de template structure e manda ler de `.claude/templates/` em runtime — uma regra que é impossível seguir com templates inexistentes.

2. **Escape hatch do TDD (fallback G1-G7):** Quando não há test plan, `x-dev-lifecycle` e `x-dev-implement` caem silenciosamente num modo G1-G7 onde testes são escritos "junto com" o código (test-with) em vez de test-first. Isto contradiz diretamente Rules 03, 05 e testing-philosophy.

3. **Rastreabilidade fraca (Gherkin ↔ AT-N):** Cenários Gherkin não têm IDs estáveis, o mapeamento para acceptance tests é por descrição textual, e sub-tarefas usam tags `[Dev]`/`[Test]` ao invés de mapear ciclos TDD.

4. **Verificação incompleta:** O integrity gate do epic verifica compile/tests/coverage mas não padrão test-first nos commits. Checklists de QA e Tech Lead não verificam existência de test plan ou rastreabilidade @GK-N→AT-N.

**Escopo incluído:**

- Criação dos 3 templates ausentes com seções TDD integradas
- Remoção do fallback G1-G7 (tornar test plan mandatório)
- Introdução de IDs @GK-N para cenários Gherkin com rastreabilidade bidirecional
- Reestruturação de sub-tarefas para formato TDD-cycle
- Correção de commit ordering no x-dev-implement
- TDD compliance check no integrity gate do epic
- Expansão de checklists QA/Tech Lead
- TDD metrics no Epic Execution Report

**Escopo excluído:**

- Mudanças no gerador `ia-dev-env` upstream (este épico modifica os artefatos gerados; a sincronização com o gerador é uma ação separada)
- Novos hooks de pre-commit para validação automatizada de TDD (pode ser um épico futuro)
- Mudanças na testing-philosophy.md ou architecture-principles.md (já estão corretos)

## 3. Definições de Qualidade Globais

### Global Definition of Ready (DoR)

- Skills `x-dev-lifecycle`, `x-dev-implement`, `x-dev-epic-implement` revisados e compreendidos
- Skills `x-story-create`, `x-test-plan`, `x-story-epic` revisados e compreendidos
- Templates existentes em `.claude/templates/` identificados
- Regras em `.claude/rules/` (03, 05, 06) revisadas
- Agentes `qa-engineer.md` e `tech-lead.md` revisados (checklists atuais)
- Análise dos 10 gaps documentada no plano de auditoria

### Global Definition of Done (DoD)

- Todos os 3 templates ausentes criados em `.claude/templates/`
- Fallback G1-G7 removido de `x-dev-lifecycle` e `x-dev-implement`
- Cenários Gherkin com IDs @GK-N nos templates e skills
- Sub-tarefas no formato TDD-cycle nos templates e skills
- DoD Local com referência mandatória a test plan
- Integrity gate com TDD compliance check
- Checklists QA e Tech Lead expandidos
- Quality Gates com linguagem prescritiva (BLOCKING violations)
- Epic Execution Report com seção TDD Compliance
- Zero regressão: skills existentes continuam funcionais
- **TDD Compliance:** Commits mostram padrão test-first (teste precede implementação no git log), refactoring explícito após green, testes incrementais via TPP
- **Double-Loop TDD:** Acceptance tests derivados dos cenários Gherkin (outer loop), unit tests guiados por TPP (inner loop)
- **Rastreabilidade:** Todo @GK-N mapeia para ≥1 AT-N, todo AT-N referencia um @GK-N válido

## 4. Regras de Negócio Transversais

| ID | Título | Descrição |
| :--- | :--- | :--- |
| RULE-001 | Template como Single Source of Truth | Toda estrutura de story, epic ou implementation map DEVE ser lida de `.claude/templates/` em runtime. Hardcoding de estrutura em skills é PROIBIDO (SD-08). Skills que geram esses artefatos devem referenciar o template, não duplicar sua estrutura. |
| RULE-002 | TDD Sem Escape Hatch | Test plan é pré-requisito mandatório para implementação. Não existe fallback "test-with" ou "test-after". Se o test plan não puder ser gerado, o lifecycle DEVE abortar com mensagem clara apontando para `/x-test-plan`. |
| RULE-003 | Rastreabilidade Bidirecional @GK-N ↔ AT-N | Todo cenário Gherkin deve ter ID estável `@GK-N`. Todo acceptance test (AT-N) deve referenciar um `@GK-N`. A verificação é bidirecional: nenhum @GK-N pode ficar sem AT-N correspondente, e nenhum AT-N pode referenciar @GK-N inexistente. |
| RULE-004 | Sub-tarefas Refletem Ciclos TDD | Sub-tarefas devem usar tag `[TDD]` com referência a AT-N/UT-N, não tags `[Dev]`/`[Test]` que sugerem fases sequenciais. O formato é: `[TDD] AT-N (@GK-N): Escrever acceptance test (RED)`. |
| RULE-005 | Commit Atômico por Ciclo TDD | Cada ciclo Red-Green-Refactor produz um commit imediato. Commits não são adiados para um step final. O commit é parte integral do ciclo TDD, não uma fase separada. |
| RULE-006 | Backward Compatibility | Mudanças nos skills devem ser backward-compatible: stories existentes sem @GK-N devem ser tratáveis (migração automática ou warning), e skills devem funcionar com artefatos pré-existentes. |
| RULE-007 | Verificação em Múltiplos Níveis | TDD compliance é verificada em 3 níveis: (1) Phase 0 do lifecycle (abort se test plan ausente), (2) Integrity gate do epic (warning de TDD compliance), (3) Tech Lead review (NO-GO se itens TDD falham). |
