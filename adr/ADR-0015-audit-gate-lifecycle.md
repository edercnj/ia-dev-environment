## Context

The `ia-dev-environment` repository has accumulated multiple governance gates across
five epics (EPIC-0041, EPIC-0046, EPIC-0050, EPIC-0052, EPIC-0055) with no consistent
framework for deciding where a new gate should live. Three recurring problems were
documented before EPIC-0058:

### Problem 1 — Ghost script references
Rules 19, 21, and 22 reference CI scripts (`audit-flow-version.sh`,
`audit-epic-branches.sh`, `audit-skill-visibility.sh`) that do not exist on disk.
Any developer following those rules is silently unprotected.

### Problem 2 — Inconsistent layer choices
| Epic | Gate | Layer chosen | Rationale documented? |
| :--- | :--- | :--- | :--- |
| EPIC-0046 | LifecycleIntegrityAuditTest | Java test | Yes (Rule 14 scope guard) |
| EPIC-0050 | audit-model-selection.sh | CI script | Implicit |
| EPIC-0052 | audit-execution-integrity.sh + Stop hook | Hybrid | Implicit |
| EPIC-0055 | audit-task-hierarchy.sh + PreToolUse hook | Hybrid | Implicit |

Hybrid gates (script + hook) that emerged without a decision framework are
harder to maintain and duplicate logic.

### Problem 3 — No catalog
Without `docs/audit-gates-catalog.md`, there is no single place to understand
which gates protect which invariants. New contributors must read 5+ rules and
2+ ADRs to reconstruct the picture.

---

## Decision

Adopt the **Audit Gate Lifecycle Convention** as documented in Rule 26:

1. **Four canonical layers** — Hook runtime / CI script / Java test / CI workflow —
   with a decision matrix. Every gate belongs to exactly one layer.

2. **Naming convention** — `verify-*.sh` (hooks), `audit-*.sh` (CI scripts),
   `*AuditTest.java` / `*Lint.java` (Java), `*.yml` (workflows).

3. **Standardized exit codes** for CI scripts: 0 (success), 1 (violation + named code),
   2 (operational error), 3 (baseline/exemption corrupt).

4. **Mandatory `--self-check` flag** on every `audit-*.sh`. CI workflow runs all
   self-checks before the main audit matrix.

5. **Catalog-before-Add (RULE-004)** — no gate may be referenced in any rule without
   a simultaneous entry in `docs/audit-gates-catalog.md`.

6. **Generation parity (RULE-003)** — CI scripts live in source-of-truth
   (`java/src/main/resources/targets/claude/scripts/`) and are generated to
   both `scripts/` (own repo) and `.claude/scripts/` (generated projects) by
   `ScriptsAssembler` (EPIC-0058).

---

## Consequences

### Positive

- Formal orientation: new engineers choose the right layer from the decision matrix
  rather than guessing.
- Ghost scripts eliminated: stories 0058-0003/0004/0005 close the three Rule references.
- Catalog as single source of truth: `docs/audit-gates-catalog.md` provides a complete
  governance picture.
- Generation parity: all projects generated by `ia-dev-env` inherit the governance gates.
- Fail-fast CI: `--self-check` pre-flight aborts early on structurally broken scripts.

### Negative

- Marginal review overhead: PRs introducing new gates must reference Rule 26 and
  update the catalog.
- Migration cost: existing gates without `--self-check` must be retrofitted
  (scoped to EPIC-0058 stories 0058-0003/0004/0005).

---

## Alternatives Considered

### Alternative A — Leave ad hoc
Continue deciding layer placement per-epic without formal guidance.

**Rejected:** The three ghost scripts and the inconsistent layer choices documented
above are the direct result of this approach. The onboarding cost increases with
each gate added.

### Alternative B — Force all gates to be Java tests
Standardize on `*AuditTest.java` for all governance checks, eliminating the
layer choice entirely.

**Rejected:** Rule 14 (Project Scope Guard) explicitly prohibits Java code that
does not serve the CLI generation pipeline. Gates that check runtime artefacts
(execution-state.json, git refs, PR metadata) cannot be Java tests without
violating Rule 14. Additionally, Java tests run only under `mvn verify` and
cannot react to LLM turn events (the Hook runtime layer's raison d'être).

### Alternative C — Single unified CI script per rule
Each Rule that defines a convention owns exactly one CI script; no hooks, no
Java tests.

**Rejected:** Hooks must react to LLM turn events (Stop, PreToolUse) where GitHub
Actions has no signal. Java tests are the only practical option when AST-level
inspection is needed (e.g., `TelemetryMarkerLint` parsing SKILL.md bodies).
A single-layer approach would either miss a class of checks entirely or push
inappropriate code into the generator (Rule 14 violation).
