# Prompt: Epic and Story Generation — x-release Interactive Flow, Auto-Versioning, Smart Resume & Observability

> **Usage**: Run `/x-story-epic-full` with this file as input spec.
> Example: `/x-story-epic-full specs/SPEC-x-release-interactive-flow-v1.md`

---

## System

**Project**: `ia-dev-environment` — CLI generator of AI-assisted development environments.

**Base version analyzed**: branch `develop`, post-v3.1.0 (EPIC-0035 closed).

**Objective of this specification**: Evolve the `x-release` skill from a command-driven flow
that requires explicit version arguments and manual coordination between pauses into an
interactive, auto-versioned, state-aware release orchestrator. The new flow detects the next
version from Conventional Commits, prompts the operator at each pause point with actionable
options (including seamless handoff to `x-pr-fix-comments`), supports smart resume when
interrupted, parallelizes validation checks, emits release metrics, creates GitHub Releases
automatically (with confirmation), and closes each cycle with a didactic Git Flow diagram
explaining why `main` and `develop` legitimately diverge.

**Central principle of all stories**: the current `x-release` skill (post-EPIC-0035) works
correctly but imposes high cognitive load on operators. Key frictions observed in production
use: (a) operators must remember to pass the version explicitly; (b) pauses between phases
leave operators unsure what to do next; (c) state files can become orphaned when sessions
are interrupted; (d) no automatic release notes on GitHub; (e) VALIDATE-DEEP runs 8 checks
sequentially, dominating wall-clock time; (f) post-release, operators ask "why does develop
differ from main?" — there is no final summary explaining the Git Flow cycle. Since all
artifacts in `.claude/` and `.github/` are **generated outputs** of `ia-dev-env`, all changes
MUST be made in the generator's source code (Java assemblers, templates under
`java/src/main/resources/targets/claude/skills/core/x-release/`), NOT by manually editing
generated files.

## Scope of the Epic

### Business Context

The `x-release` skill is the single entry point for cutting production releases in every
downstream project generated by `ia-dev-env`. Today (post-EPIC-0035), the skill:

1. **Requires explicit version argument** — operator must compute the next version themselves
   (often by reading `git log` and counting feat/fix commits by hand).

2. **Pauses without guidance** — Phase 8 APPROVAL_GATE halts and exits; operator must
   remember to run `/x-pr-fix-comments` separately and then re-invoke with
   `--continue-after-merge`.

3. **No smart resume** — if the operator runs `/x-release` while a state file exists,
   the skill aborts with `STATE_CONFLICT` instead of offering to resume.

4. **Sequential validation** — VALIDATE-DEEP runs 8 checks one after another; wall-clock
   time dominates the pre-PR phase.

5. **No GitHub Release** — the tag is pushed but no Release is created, leaving
   release notes only in `CHANGELOG.md`.

6. **No telemetry** — no record of how long each phase takes or which phases block most.

7. **No closing summary** — after PUBLISH + CLEANUP, the skill exits silently. Operators
   frequently ask why `main` and `develop` diverge.

8. **Schema v1 limitations** — the state file has no fields for interactive action
   tracking (`nextActions`, `waitingFor`) or duration metrics.

### Dimensions of improvement

1. **Auto-versioning from Conventional Commits** — positional `<version>` becomes optional;
   skill computes next version from `git log <last-tag>..HEAD` classifying `feat!`/`BREAKING`
   (MAJOR), `feat` (MINOR), `fix`/`perf` (PATCH). New `--version X.Y.Z` flag for override.

2. **State file schema v2 (breaking change)** — drop v1 support. New fields: `nextActions[]`,
   `waitingFor` enum, `phaseDurations{}`, `lastPromptAnsweredAt`, `githubReleaseUrl`.
   `schemaVersion` bumped to 2.

3. **Pre-commit integrity checks** — new sub-phase in VALIDATE-DEEP validates cross-file
   consistency (`pom.xml` version alignment with README/CLAUDE.md badges, `[Unreleased]`
   non-empty, no new TODO/FIXME in commit window). New error code `VALIDATE_INTEGRITY_DRIFT`.

4. **Parallelized VALIDATE-DEEP** — 8 independent checks run concurrently; aggregator sorts
   failures by severity. Target: ≥40% wall-clock reduction. Preserves existing `VALIDATE_*`
   error codes.

5. **Git Flow summary diagram** — new Phase 13 SUMMARY (read-only) renders ASCII Git Flow
   cycle with real version numbers and PR references from state file. New reference
   `references/git-flow-cycle-explainer.md`.

6. **GitHub Release automation with confirmation** — after tag push, prompts
   "Create GitHub Release? (Y/n)"; on yes, runs `gh release create` with body from CHANGELOG
   `[X.Y.Z]` section. Persists `githubReleaseUrl` in state file. Flag `--no-github-release`
   for opt-out.

7. **Interactive prompt engine** — at halt points (APPROVAL_GATE, BACK-MERGE-DEVELOP,
   recoverable failures), presents `AskUserQuestion` with options: "run x-pr-fix-comments",
   "PR merged — continue", "exit and resume later". Each answer persisted in
   `nextActions`/`waitingFor`. Flag `--no-prompt` preserves non-interactive behavior for CI.

8. **Smart Resume** — `/x-release` with no arguments detects existing state file and
   prompts: "Resume release X.Y.Z (phase PHASE, idle 2h)?" with options resume/abort/new.
   Replaces `STATE_CONFLICT` with actionable flow.

9. **Pre-flight dashboard** — before creating `release/*`, shows summary panel: detected
   version, bump type, commit counts, CHANGELOG preview (first 10 lines), branches/PRs
   to be created, integrity check results. Single prompt "Proceed? (yes / edit version /
   abort)".

10. **Operational commands `--status` and `--abort`** — `--status` is read-only (works
    with or without state file). `--abort` closes PRs, deletes branches (local + remote),
    removes state file; double-confirms. New codes `ABORT_PR_CLOSE_FAILED`,
    `ABORT_BRANCH_DELETE_FAILED` (warn-only).

11. **`/x-pr-fix-comments` handoff** — when operator selects "run fix-pr-comments" in
    prompt, x-release invokes the skill via Skill tool with the PR number. On return,
    re-verifies PR state and re-offers options. Contract in new
    `references/prompt-flow.md`.

12. **Per-phase telemetry** — writes `plans/release-metrics.jsonl` (one line per phase:
    `{releaseVersion, phase, startedAt, endedAt, durationSec, outcome}`). Phase SUMMARY
    shows top-3 slowest phases vs. historical average (last 5 releases). File committed
    to repo under `plans/`. Flag `--telemetry off` disables.

13. **Interactive dry-run (onboarding walkthrough)** — `/x-release --dry-run --interactive`
    runs all phases in simulation mode, pausing at each with prompt "next step would be X
    — continue simulation?". No side effects. Useful for onboarding and feature validation.

14. **Hotfix parity** — replicate auto-versioning (PATCH-restricted), prompts, smart resume,
    pre-flight, summary diagram, and telemetry in the `--hotfix` path. Summary diagram
    adapted for branches cut from `main` instead of `develop`.

15. **Documentation + golden regeneration** — rewrite `SKILL.md` to reflect new defaults
    (auto-version, prompts, smart resume). New `references/interactive-flow-walkthrough.md`
    with end-to-end example. Regenerate golden files for all 17 profiles. Update project
    CHANGELOG `[Unreleased]`.

### Out of Scope

- Multi-module Maven releases (coordinated bump of multiple `pom.xml`) — future epic.
- Jira/ADO transition automation per phase.
- External registry publication beyond GitHub Release (Docker Hub, npm) — future epic.
- Automated post-tag rollback (stays manual).

## Non-Functional Requirements

- **Performance**: VALIDATE-DEEP wall-clock reduced ≥40% via parallelization (S04).
- **Backward compatibility**: state file schema v1 explicitly dropped; migration = abort
  with clear error. No silent v1→v2 upgrade.
- **Observability**: every phase emits one line to `release-metrics.jsonl`; SUMMARY phase
  includes comparison with rolling 5-release average.
- **Testability**: every prompt must have non-interactive equivalent (flags); CI flows
  never require TTY.
- **Resilience**: smart resume handles all halt points; `--abort` provides safe cleanup;
  no operation leaves orphaned state files.
- **Cloud-agnostic**: no dependency on services beyond `gh` CLI and `git`.

## Story Breakdown

### FASE 1 — Foundations (low coupling, high parallelism)

- **S01 — Auto-detect next version from Conventional Commits**
  - Parse `git log <last-tag>..HEAD --format=%s`, classify per prefix
  - Positional `<version>` optional; new `--version X.Y.Z` for override
  - Opening banner: "Next version detected: 3.2.0 (MINOR) — 7 feat, 2 fix, 0 breaking since v3.1.0"
  - Error when no versionable commits since tag
  - Dependencies: none
  - Risk: low

- **S02 — State file schema v2 (breaking change)**
  - Drop v1 support; abort with `STATE_SCHEMA_VERSION` and migration instructions
  - New fields: `nextActions[]` (`{label, command}`), `waitingFor` (enum), `phaseDurations{}`,
    `lastPromptAnsweredAt`, `githubReleaseUrl`
  - Update `ReleaseStateFileSchemaTest` + `references/state-file-schema.md`
  - Dependencies: none
  - Risk: medium (schema evolution, explicit breaking)

- **S03 — Pre-commit integrity checks**
  - New VALIDATE-DEEP sub-check: cross-file version alignment, `[Unreleased]` non-empty,
    no new TODO/FIXME in commit window
  - New error `VALIDATE_INTEGRITY_DRIFT` with file list
  - Dependencies: none
  - Risk: low

- **S04 — Parallelize VALIDATE-DEEP**
  - Run 8 independent checks concurrently (shell background jobs + wait + exit aggregation)
  - Failures sorted by severity; preserve existing `VALIDATE_*` codes
  - Target ≥40% wall-clock reduction
  - Dependencies: none
  - Risk: medium (process orchestration + correct exit code capture)

- **S05 — Phase SUMMARY with Git Flow diagram**
  - New Phase 13 (read-only); renders ASCII diagram with real versions and PR numbers
  - Explains `main` vs `develop` divergence adaptively
  - Template in new `references/git-flow-cycle-explainer.md`
  - Dependencies: none (consumes state file at end)
  - Risk: low

- **S06 — GitHub Release automation (always with confirmation)**
  - After tag push (Phase 11 PUBLISH), ALWAYS prompts "Create GitHub Release? (Y/n)"
  - On yes: `gh release create vX.Y.Z --notes "<CHANGELOG [X.Y.Z] body>"`
  - Persists `githubReleaseUrl`; flag `--no-github-release` for CI opt-out
  - Dependencies: none
  - Risk: low

### FASE 2 — Interactive UX (depends on S01 + S02)

- **S07 — Interactive prompt engine at halt points**
  - AskUserQuestion at APPROVAL_GATE, BACK-MERGE-DEVELOP, recoverable failures
  - Options: "run /x-pr-fix-comments <PR#>", "PR merged — continue", "exit and resume later"
  - Each answer updates `nextActions`/`waitingFor`
  - Flag `--no-prompt` for CI
  - Dependencies: S02
  - Risk: medium (non-TTY fallback)

- **S08 — Smart Resume**
  - `/x-release` with no args detects state file, prompts resume/abort/new
  - Replaces `STATE_CONFLICT` abort with actionable prompt
  - Integrates with S01 (new version only detected when no active release)
  - Dependencies: S01, S02
  - Risk: low

- **S09 — Pre-flight dashboard**
  - Summary panel before creating `release/*`: version, commit counts, CHANGELOG preview,
    branches/PRs to create, integrity results (from S03)
  - Single prompt "Proceed? (yes / edit version / abort)"
  - Dependencies: S01, S03
  - Risk: low

- **S10 — Operational commands `--status` and `--abort`**
  - `--status`: read-only, works with or without state file
  - `--abort`: closes PRs via `gh pr close`, deletes branches local/remote, removes state;
    double-confirm mandatory
  - New codes `ABORT_PR_CLOSE_FAILED`, `ABORT_BRANCH_DELETE_FAILED` (warn-only)
  - Dependencies: S02
  - Risk: medium (remote side effects)

### FASE 3 — Integration & Observability (depends on Phase 2)

- **S11 — Handoff to `/x-pr-fix-comments`**
  - When operator chooses "run fix-pr-comments", invoke via Skill tool with PR#
  - On return, re-verify PR state (`gh pr view`), re-offer options
  - Contract in new `references/prompt-flow.md`
  - Dependencies: S07
  - Risk: low

- **S12 — Per-phase telemetry**
  - Write `plans/release-metrics.jsonl` (one line per phase, committed to repo)
  - SUMMARY shows top-3 slowest vs. 5-release rolling average
  - Flag `--telemetry off` disables
  - Dependencies: S02
  - Risk: low

- **S13 — Interactive dry-run walkthrough**
  - `/x-release --dry-run --interactive` pauses at each phase with simulation prompt
  - No side effects; echoes commands that would run
  - Dependencies: S07, S09
  - Risk: low

### FASE 4 — Parity & Closure

- **S14a — Hotfix parity**
  - Replicate S01 (PATCH-only), S05, S07, S08, S09, S12 in `--hotfix` path
  - Summary diagram adapted for branches from `main`
  - Dependencies: S01, S05, S07, S08, S09, S12
  - Risk: low

- **S14b — Documentation + golden regeneration**
  - Rewrite SKILL.md with new defaults
  - New `references/interactive-flow-walkthrough.md`
  - Regenerate golden files for all 17 profiles
  - Update project CHANGELOG `[Unreleased]`
  - Dependencies: all previous
  - Risk: low

## Cross-Cutting Risks

- State files in-flight at deployment time: schema v1 explicitly rejected with clear migration error.
- `AskUserQuestion` in non-TTY CI: universal `--no-prompt` + explicit flags document CI usage in S14b.
- Parallel validation failure ordering: S04 aggregator sorts by severity before printing.
- `--abort` human-error: double-confirm with dry-run listing of resources to be destroyed.
- Golden file churn: S14b regenerates once, after SKILL.md is finalized — not per story.

## Quality Gates (DoD)

- All new paths have unit + integration tests; coverage ≥95% line, ≥90% branch (Rule 05).
- VALIDATE-DEEP wall-clock benchmark documented (before/after) in S04 closure.
- All 17 golden profiles regenerated and byte-for-byte equal in S14b.
- SKILL.md passes dual-copy consistency (claude + github) checks.
- CI continues to pass with `--no-prompt` flags applied.
- No state file format upgrade path is offered; v1 → v2 is a hard break documented in CHANGELOG.

## Glossary

- **APPROVAL_GATE**: Phase 8 halt, awaiting human PR merge to main.
- **BACK-MERGE-DEVELOP**: Phase 10, re-integrates `main` into `develop` post-release.
- **VALIDATE-DEEP**: Phase 2, runs 8 checks before creating release branch.
- **Smart Resume**: auto-detection of existing state file with actionable prompt (vs. abort).
- **Pre-flight dashboard**: summary panel before first branch/PR is created.
- **Halt point**: phase where skill exits intentionally, awaiting human action.
