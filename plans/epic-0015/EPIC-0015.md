# Epico: Migracao para Arquitetura Hexagonal (Ports & Adapters)

**Autor:** Architect Agent / Claude
**Data:** 2026-04-03
**Versao:** 1.0
**Status:** Concluído

## 1. Visao Geral

**Chave Jira:** —

O `ia-dev-environment` e uma CLI Java 21 (Picocli 4.7) que gera boilerplate completo de ambientes de desenvolvimento assistido por IA. O projeto atualmente possui uma estrutura de pacotes plana com separacao funcional (`cli`, `config`, `model`, `domain`, `assembler`, `template`, `checkpoint`, `progress`, `exception`, `util`), sem fronteiras arquiteturais explicitas que separem o nucleo de negocio das preocupacoes de infraestrutura e entrega.

Esta iniciativa migra a arquitetura atual para **Arquitetura Hexagonal (Ports & Adapters)**, alinhando a base de codigo com os proprios templates que a ferramenta gera para projetos terceiros. A inconsistencia de uma ferramenta que gera arquitetura hexagonal nao seguir esse padrao internamente e o principal motivador desta migracao.

**Escopo incluido:**

- Migracao estrutural completa de 194+ classes Java para pacotes hexagonais (domain, application, infrastructure)
- Definicao de Input Ports (Use Cases) e Output Ports (interfaces para infraestrutura)
- Implementacao de Domain Services, Driven Adapters e Driving Adapters
- Criacao de Composition Root (ApplicationFactory) para wiring manual
- Testes de arquitetura com ArchUnit cobrindo todas as regras de fronteira
- Migracao dos 23 assemblers para camada de aplicacao

**Escopo excluido:**

- Alteracoes nos ~470 templates Pebble/Nunjucks em `src/main/resources/`
- Mudancas de comportamento funcional (a migracao e exclusivamente estrutural)
- Introducao de frameworks de injecao de dependencia (Spring, Guice, etc.)
- Refatoracao de logica de negocio dos assemblers (apenas realocacao)
- Alteracoes na interface CLI visivel ao usuario

## 4. Regras de Negocio Transversais

| ID | Titulo | Descricao |
| :--- | :--- | :--- |
| RULE-001 | Dependency Rule Estrita | O pacote `domain/` NUNCA pode importar de `infrastructure/`, `application/assembler/` ou `application/factory/`. Violacoes devem ser detectadas por ArchUnit em tempo de build via `mvn verify`. |
| RULE-002 | Ports como Contratos | Todo acesso a recursos externos (filesystem, templates, YAML, checkpoints, progresso) deve ocorrer exclusivamente via Output Port (interface em `domain/port/output/`). Nenhum adapter concreto pode ser instanciado dentro do dominio. |
| RULE-003 | Use Cases como Ponto de Entrada | A CLI (`infrastructure/adapter/input/cli/`) interage com o dominio exclusivamente via Input Ports (`domain/port/input/`). Jamais acessa `domain/service/` diretamente. |
| RULE-004 | Records Imutaveis no Dominio | Todos os modelos de dominio permanecem como Java Records imutaveis. Nenhuma anotacao de framework (Spring, Jackson, Picocli, etc.) pode aparecer em `domain/model/`. |
| RULE-005 | Composition Root Unico | A montagem do grafo de dependencias (wiring) ocorre exclusivamente em `infrastructure/config/ApplicationFactory.java`. Nao ha injecao de dependencia por framework — e manual e explicita. |
| RULE-006 | ArchUnit como Guardiao | Testes de arquitetura com ArchUnit devem ser criados junto com cada historia de migracao, garantindo que as fronteiras hexagonais sejam verificadas automaticamente em `mvn verify`. |
| RULE-007 | Paridade Funcional Total | Ao final da migracao, todos os 1961 testes existentes devem passar sem modificacao de comportamento. O golden file parity test e o criterio final de aceitacao. |
| RULE-008 | Migracao Incremental sem Big Bang | Cada historia deve ser deployavel de forma independente. O projeto deve compilar e todos os testes devem passar apos cada historia individualmente. |
| RULE-009 | Cobertura de Testes Mantida | Line Coverage >= 95% e Branch Coverage >= 90% devem ser mantidos em cada historia, conforme thresholds do JaCoCo ja configurados no `pom.xml`. |
| RULE-010 | Preservacao de Contratos de Template | Os ~470 arquivos de template Pebble/Nunjucks em `src/main/resources/` nao devem ser alterados. A migracao e exclusivamente estrutural no codigo Java. |
