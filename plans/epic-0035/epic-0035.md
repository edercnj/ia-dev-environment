# Épico: Extensão do `x-release` com Approval Gate, PR-Flow e Deep Validation

**Autor:** Platform Team
**Data:** 2026-04-10
**Versão:** 1.0
**Status:** Em Refinamento

## 1. Visão Geral

**Chave Jira:** —

Extensão do skill `x-release` (`java/src/main/resources/targets/claude/skills/core/x-release/SKILL.md`, 449 linhas) para **corrigir violação da Rule 09** (merge local direto para `main`/`develop` sem PR), **introduzir approval gate humano explícito** entre a abertura do PR da release e a aplicação de tag + back-merge, e **aprofundar a validação** pré-release (coverage, golden files, consistência cruzada de versão). O épico preserva integralmente os comportamentos corretos atuais (bump auto-detect, criação de `release/*`, update de version files, SNAPSHOT handling, delegação a `x-release-changelog`, hotfix workflow, dry-run) e adiciona cinco phases novas: `VALIDATE-DEEP` (substitui Step 2), `OPEN-RELEASE-PR` (substitui Step 7 MERGE-MAIN), `APPROVAL-GATE` (nova pausa humana), `RESUME-AND-TAG` (nova, ativada por `--continue-after-merge`) e `BACK-MERGE-DEVELOP` (substitui Step 9 MERGE-BACK). Estado entre invocações é persistido em `plans/release-state-<X.Y.Z>.json` com esquema versionado, habilitando resume seguro após fechar/reabrir sessão do Claude Code. Flag `--interactive` opcional adiciona pausa in-session via `AskUserQuestion` como alternativa ao state file.

**Motivação imediata:** a release v2.3.0 do próprio `ia-dev-env` (branch `release/2.3.0` → PR #262 → PR #263 → PR #264) passou por múltiplas etapas manuais com janelas temporais em estado intermediário, ilustrando concretamente a necessidade de um approval gate persistente. Esta extensão do `x-release` fecha essas brechas para releases futuras.

## 3. Definições de Qualidade Globais

### Global Definition of Ready (DoR)

- Especificação `spec-x-release-branch-flow.md` revisada e aprovada
- Arquivo `x-release/SKILL.md` atual lido integralmente por quem vai implementar a story
- Branch `feature/epic-0035-*` criada a partir de `develop` atualizado
- `mvn clean verify -P all-tests` verde como baseline antes da story começar
- Coverage baseline registrada (line/branch) para comparação
- `gh` CLI instalado e autenticado no ambiente de desenvolvimento
- Golden files baseline de `x-release/SKILL.md` conhecido para os 17+ profiles

### Global Definition of Done (DoD)

- **Cobertura:** ≥ 95% line, ≥ 90% branch (Rule 05). Zero degradação permitida nos testes de release.
- **Testes Automatizados:** Cada story DEVE ter ao menos 1 teste automatizado validando o critério de aceite principal. Golden file tests (17+ profiles) passando após regeneração.
- **Smoke Tests:** `PlatformDirectorySmokeTest` e `AssemblerRegressionSmokeTest` permanecem verdes.
- **Relatório de Cobertura:** JaCoCo report anexado em cada PR de story.
- **Documentação:** `SKILL.md` do x-release, `README.md`, `references/approval-gate-workflow.md` e `references/state-file-schema.md` atualizados/criados conforme story.
- **Persistência:** Schema JSON do state file validado por teste dedicado.
- **Performance:** Tempo de execução de `mvn verify -P all-tests` não pode aumentar mais que 5% vs. baseline.
- **TDD Compliance:** Commits seguem padrão test-first. Refactoring explícito após green. Tests incrementais via TPP.
- **Double-Loop TDD:** Cenários Gherkin das stories → acceptance tests (outer loop). Unit tests das classes Java → inner loop.
- **Build Verde:** Cada story termina com `mvn clean verify -P all-tests` verde antes do merge.
- **Golden Files Consistentes:** Após edição de `SKILL.md`, golden files nos 17+ profiles DEVEM ser regenerados no mesmo commit.
- **Rule 09 Compliance:** Zero `git merge` direto para `main` ou `develop` em qualquer phase do skill atualizado.
- **Dogfood Verification (Opcional):** Ao final do épico, uma release do próprio `ia-dev-env` deve ser executada usando o `x-release` atualizado, validando o fluxo end-to-end.
