## 3. Descrição

Como **usuário ou skill consumidora** de `x-git-push` que precisa criar uma branch isolada do main checkout, eu quero uma flag opcional `--worktree` que faz `x-git-push` delegar a criação para `/x-git-worktree create` antes de fazer stage/commit/push, evitando conflito quando múltiplas instâncias rodam concorrentemente, **sem quebrar invocações existentes** que não passam a flag.

Esta é uma **mudança opt-in pura**: o comportamento default de `x-git-push` permanece **idêntico** ao atual (Rule 04 do épico — backward compat). A única diferença visível é a presença da nova flag e do comportamento associado quando ela é passada.

### 3.3 Nova Subseção "Step 1.3 — Worktree-Aware Branch Creation"

Inserir após o Step 1.2 atual (ou equivalente onde o `git checkout -b` é executado):

```markdown
### Step 1.3 — Worktree-Aware Branch Creation (when `--worktree` is passed)

When `--worktree` flag is present, follow this branch creation flow instead of plain `git checkout -b`:

1. **Detect context** (RULE-018, see [Rule 14](../../../../rules/14-worktree-lifecycle.md)):
   ```bash
   # Inline snippet from x-git-worktree Operation 5: detect-context
   detect_worktree_context() { ... }
   CONTEXT_JSON=$(detect_worktree_context)
   IN_WT=$(echo "$CONTEXT_JSON" | jq -r '.inWorktree')
   ```

2. **Branch on context**:
   - **If `IN_WT == true`**: log `"Already in worktree, reusing current cwd for branch creation"`. Proceed with normal `git checkout -b "$BRANCH"` inside the existing worktree. **Do NOT** create a nested worktree (RULE-002).
   - **If `IN_WT == false`**: delegate to `/x-git-worktree create`:
     ```bash
     SLUG=$(echo "$BRANCH" | sed 's|^[a-z]*/||')  # strip feature/, fix/, hotfix/, etc.
     WT_PATH=$(/x-git-worktree create \
       --branch "$BRANCH" \
       --base "${BASE_BRANCH:-develop}" \
       --id "$SLUG")
     cd "$WT_PATH"
     ```

3. **Continue with stage/commit/push** as in the default flow. The `cd` ensures all subsequent git operations target the worktree.

4. **Cleanup**: `x-git-push --worktree` does NOT remove the worktree after pushing. Cleanup is the responsibility of the caller (caller-owns-removal per RULE-018 Section 5). The caller may invoke `/x-git-worktree remove --id $SLUG` after PR merge, or use `/x-git-worktree cleanup` periodically.
```

### 3.4 Atualização da Seção "When to Use"

Adicionar bullet:
- `--worktree` flag: when running multiple `x-git-push` invocations concurrently OR when the caller wants to isolate the branch from the main checkout (e.g., running parallel TDD loops).

## Backward Compatibility

- Without `--worktree` flag: behavior is **identical** to the pre-EPIC-0037 version. No worktree is created. `git checkout -b` runs in the current cwd as before.
- With `--worktree` flag: opt-in worktree-aware behavior (Step 1.3 above).
- Callers that never pass `--worktree` see zero behavior change.
```

## 3.6 Entrega de Valor

- **Valor Principal:** Skills consumidoras (e usuários finais) ganham isolamento de branch via uma única flag, sem precisar invocar `x-git-worktree` em separado. Detecção de aninhamento garante zero risco de nested worktree quando rodando dentro de orquestrador.
- **Métrica de Sucesso:** Um teste de regressão valida que `x-git-push` SEM `--worktree` se comporta byte-a-byte idêntico ao baseline. Um teste novo valida que COM `--worktree` o worktree é criado em `.claude/worktrees/{slug}/`.
- **Impacto no Negócio:** Reduz fricção para usuários que querem rodar branches paralelas. Habilita futuros workflows de paralelização sem necessidade de orquestrador completo.

## 7. Critérios de Aceite (Gherkin)

```gherkin
Cenario: Backward compat — sem flag, comportamento idêntico
  DADO que estou no main checkout (cwd = repo root)
  E não passo --worktree
  QUANDO executo /x-git-push feature/foo --base develop
  ENTÃO git checkout -b feature/foo é executado no cwd atual
  E nenhum worktree é criado
  E .claude/worktrees/ não tem novo entry
  E o comportamento bate byte-a-byte com o baseline pré-épico

Cenario: Happy path com --worktree em main repo
  DADO que estou no main checkout
  E passo --worktree
  QUANDO executo /x-git-push feature/story-0037-0003-foo --worktree
  ENTÃO detect_worktree_context retorna inWorktree:false
  E /x-git-worktree create --branch feature/story-0037-0003-foo --id story-0037-0003-foo é chamado
  E cwd muda para .claude/worktrees/story-0037-0003-foo/
  E git add/commit/push procedem dentro do worktree

Cenario: Happy path com --worktree dentro de worktree existente
  DADO que estou em .claude/worktrees/story-0037-0003/
  E passo --worktree
  QUANDO executo /x-git-push fix/sub-task --worktree
  ENTÃO detect_worktree_context retorna inWorktree:true
  E o log mostra "Already in worktree, reusing"
  E /x-git-worktree create NÃO é chamado
  E git checkout -b fix/sub-task é executado no cwd atual (dentro do worktree story)

Cenario: Error — /x-git-worktree create falha
  DADO que .claude/worktrees/foo/ já existe (residual)
  E estou no main checkout
  QUANDO executo /x-git-push feature/foo --worktree
  ENTÃO /x-git-worktree create retorna erro
  E /x-git-push aborta com WT_CREATE_FAILED
  E nenhum git checkout -b é executado

Cenario: Boundary — slug com prefixo não-padrão
  DADO que executo /x-git-push experimental/branch-x --worktree
  QUANDO o slug é derivado
  ENTÃO o slug é "branch-x" (strip de "experimental/")
  E o worktree é criado em .claude/worktrees/branch-x/
```
