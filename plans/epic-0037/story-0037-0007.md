## 3. Descrição

Como **operador** rodando `x-pr-fix-epic` para aplicar correções consolidadas em PRs de um épico, eu quero que a branch de correção (`fix/epic-{id}-pr-comments`) seja criada dentro de um worktree isolado, evitando conflito com trabalho concorrente no main checkout. A criação é **automática** (não opt-in via flag), pois `x-pr-fix-epic` é tipicamente invocada por usuário final ou por CI, e o isolamento sempre faz sentido neste contexto.

A skill é **idempotente** (RULE-010 da própria skill): se o worktree de correção já existe (run resumido após interrupção), reusa em vez de re-criar. Após o PR de correção ser merged, o cleanup remove o worktree.

### 3.1 Localizar e Modificar o Bloco de Branch Creation

Editar `java/src/main/resources/targets/claude/skills/core/pr/x-pr-fix-epic/SKILL.md` em torno da linha 795 (bloco onde `git checkout -b fix/epic-{epicId}-pr-comments` é executado). Substituir por:

```markdown
### Step N — Worktree-Aware Correction Branch Creation

**Step N.1 — Detect worktree context** (RULE-018):
```bash
detect_worktree_context() { ... }  # snippet from x-git-worktree Operation 5
CONTEXT_JSON=$(detect_worktree_context)
IN_WT=$(echo "$CONTEXT_JSON" | jq -r '.inWorktree')
```

**Step N.2 — Decide creation strategy:**

| Context | Action |
| :--- | :--- |
| In main repo (`IN_WT=false`) | Check if `.claude/worktrees/fix-epic-${epicId}/` already exists. If yes, reuse (idempotency). If no, call `/x-git-worktree create --branch fix/epic-${epicId}-pr-comments --base develop --id fix-epic-${epicId}`. `cd` to returned path. |
| Inside an existing worktree (`IN_WT=true`) | Edge case: should not happen in normal flow (pr-fix-epic is typically invoked from main repo). Log warning and proceed using current cwd. |

**Step N.3 — Idempotency Check (RULE-010)**:
```bash
if [ -d ".claude/worktrees/fix-epic-${epicId}" ]; then
  echo "[IDEMPOTENT] Reusing existing correction worktree fix-epic-${epicId}"
  cd ".claude/worktrees/fix-epic-${epicId}"
  # Switch to the correction branch (may already be checked out)
  git checkout "fix/epic-${epicId}-pr-comments" 2>/dev/null || true
else
  WT_PATH=$(/x-git-worktree create \
    --branch "fix/epic-${epicId}-pr-comments" \
    --base develop \
    --id "fix-epic-${epicId}")
  cd "$WT_PATH"
fi
```

> **Veja:** [RULE-018 — Worktree Lifecycle](../../../../rules/14-worktree-lifecycle.md), Seção 5.
```

### 3.2 Adicionar Step Final de Cleanup

Adicionar um novo step (numeração depende da estrutura atual; assumindo último step atual + 1) ao fim do workflow do `x-pr-fix-epic/SKILL.md`:

```markdown
### Step M — Worktree Cleanup (after correction PR merged)

After the correction PR is merged successfully:

```bash
# Return to main repo before removing the worktree
cd "$(git worktree list --porcelain | awk '/^worktree/{print $2; exit}')"

/x-git-worktree remove --id "fix-epic-${epicId}"
echo "[CLEANUP] Removed correction worktree fix-epic-${epicId}"
```

If the correction PR is **not** merged (failure or operator aborted), the worktree is **preserved**. Operator can manually inspect via `cd .claude/worktrees/fix-epic-${epicId}` or remove via `/x-git-worktree remove --force --id fix-epic-${epicId}`.
```

### 3.3 Atualização da Seção "When to Use" / "Idempotency"

Confirmar que a documentação existente sobre idempotência (RULE-010 da skill) menciona o reuso de worktree em runs resumidos. Se ausente, adicionar:

```markdown
## Idempotency (RULE-010)

If `x-pr-fix-epic` is interrupted (network failure, operator abort) and re-invoked with the same `--epic` argument:
- The correction worktree at `.claude/worktrees/fix-epic-${epicId}/` is **reused** (not re-created).
- The correction branch `fix/epic-${epicId}-pr-comments` is reused.
- Steps already completed are detected via execution-state and skipped.

This ensures that resuming after a transient failure does not create duplicate worktrees or branches.
```

## 3.4 Entrega de Valor

- **Valor Principal:** A branch de correção pós-PR fica isolada do main checkout, evitando conflito com qualquer trabalho concorrente. O reuso idempotente preserva a semântica de RULE-010 da skill.
- **Métrica de Sucesso:** Smoke: rodar `x-pr-fix-epic` em um épico de teste, interromper, re-rodar — verificar que o worktree é reusado (não recriado). Após PR merged, worktree é removido.
- **Impacto no Negócio:** Operador pode rodar `x-pr-fix-epic` mesmo enquanto o main checkout está em outra branch (ex: trabalhando em outra feature) — sem conflito.

```gherkin
