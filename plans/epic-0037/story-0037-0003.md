## 3. Descrição

Como **platform engineer** mantenedor do `x-epic-implement`, eu quero migrar a orquestração de stories paralelas do recurso nativo do harness `Agent(isolation:"worktree")` para chamadas explícitas a `/x-git-worktree create|remove`, ganhando visibilidade nos logs (cada criação/remoção de worktree aparece como ação rastreável), permitindo recuperação manual via `/x-git-worktree list` quando subagents falham, e removendo a única dependência do harness-feature `isolation:"worktree"` no codebase, alinhando com a política unificada do épico (RULE-018 / Rule 14).

Esta é a **história de migração comportamental crítica**: é a única story do épico que muda comportamento real de execução (todas as outras adicionam capabilities opt-in ou só editam docs). Por isso, ela carrega o maior risco e tem sequenciamento explícito de testes. Após esta story, **zero ocorrências** de `Agent(isolation:"worktree")` devem permanecer no codebase.

### 3.1 Refatorar Section 1.4a "Parallel Worktree Dispatch"

Editar `java/src/main/resources/targets/claude/skills/core/dev/x-epic-implement/SKILL.md` Section 1.4a (linhas 919–969). Substituir o pattern atual por uma orquestração em **três fases explícitas**:

**Fase A — Pre-dispatch worktree creation**:
```bash
# Para cada story executável no batch paralelo:
for story in $PARALLEL_STORIES; do
  WT_PATH=$(/x-git-worktree create \
    --branch "feat/story-${story}-${slug}" \
    --base develop \
    --id "story-${story}")
  echo "$story:$WT_PATH" >> "$WORKTREES_FILE"
done
```

**Fase B — Dispatch sem `isolation`**:
```python
# Pseudo-código do dispatch (linguagem real é prompt LLM)
for story in PARALLEL_STORIES:
  Agent(
    subagent_type="general-purpose",
    # NÃO passar isolation="worktree" — agora é explícito via WT_PATH
    prompt=f"""
      Story: {story}
      Worktree path: {wt_paths[story]}
      Step 0: cd {wt_paths[story]}
      ... (resto do prompt original)
    """
  )
```

**Fase C — Post-success cleanup**:
```bash
# Após PR merged + auto-rebase:
/x-git-worktree remove --id "story-${story}"
```

**Fase D — Post-failure preservação**:
```bash
# Se SubagentResult.status == FAILED:
echo "[PRESERVED] Worktree story-${story} kept for diagnostics: ${wt_path}"
# NÃO remover; loga path para inspeção manual
```

### 3.2 Refatorar Section 1.4d "Worktree Cleanup"

Editar Section 1.4d (linhas 1018–1026). Substituir a referência ao auto-cleanup do harness por:

```markdown
### 1.4d Worktree Cleanup

After parallel dispatch completes and all SubagentResults are validated, clean up worktree resources via **explicit calls** to `/x-git-worktree`:

- **SUCCESS + merged**: Worktree is removed via `/x-git-worktree remove --id story-{id}` after the story PR is merged AND auto-rebase (Section 1.4e) completes successfully.
- **FAILED stories**: Worktree is **preserved** for diagnostic investigation. The branch and worktree path are logged. Manual cleanup via `/x-git-worktree remove --force --id story-{id}` after triage.
- **Defensive cleanup at end of epic**: After the entire epic execution completes (success or partial failure), call `/x-git-worktree cleanup --dry-run` to list any obsolete worktrees that should be removed. Operator decides whether to invoke `/x-git-worktree cleanup` (without `--dry-run`) based on the listing.

> **Veja:** [RULE-018 — Worktree Lifecycle](../../../../rules/14-worktree-lifecycle.md), Seção 5 (Creator Owns Removal), para a matriz normativa de ownership.
```

## 3.6 Entrega de Valor

- **Valor Principal:** A integração documentada em `x-git-worktree/SKILL.md` vira realidade. Operadores ganham visibilidade total sobre o ciclo de vida de worktrees em execuções de épico, podendo recuperar manualmente quando algo falha. Remove a única dependência de uma "feature mágica" do harness, alinhando o codebase com a política explícita unificada.
- **Métrica de Sucesso:** `grep -rn "isolation.*worktree" java/src/main/resources/targets/` retorna vazio. Smoke manual de `/x-epic-implement` com 2 stories paralelas mostra `/x-git-worktree create` e `/x-git-worktree remove` nos logs, e `.claude/worktrees/` contém os worktrees esperados durante execução.
- **Impacto no Negócio:** Reduz tempo de debugging quando subagents falham (paths preservados são imediatamente inspecionáveis), e elimina o "magic" no fluxo (operadores entendem cada passo).

## 7. Critérios de Aceite (Gherkin)

```gherkin
Cenario: Happy path — épico com 2 stories paralelas, ambas SUCCESS
  DADO que existe epic-0037 com 2 stories sem dependências entre si
  E ambas são executáveis em paralelo
  QUANDO /x-epic-implement 0037 é executado
  ENTÃO /x-git-worktree create é chamado 2 vezes (uma por story)
  E 2 worktrees existem em .claude/worktrees/story-{id}/
  E 2 subagents recebem prompts contendo o cwd correto
  E ambos terminam com SubagentResult SUCCESS
  E /x-git-worktree remove é chamado 2 vezes após PR merge + auto-rebase
  E .claude/worktrees/ está vazio ao fim

Cenario: Mixed — uma story SUCCESS, outra FAILED
  DADO que epic-0037 tem 2 stories paralelas
  E story A irá ter sucesso, story B irá falhar
  QUANDO /x-epic-implement 0037 é executado
  ENTÃO 2 worktrees são criados
  E story A tem worktree removido após merge
  E story B tem worktree PRESERVADO (.claude/worktrees/story-B/ ainda existe)
  E o log contém "WT_PRESERVED: .claude/worktrees/story-B"
  E /x-git-worktree cleanup --dry-run reporta 0 obsoletos (FAILED não é obsoleto)

Cenario: Error — /x-git-worktree create falha em pre-dispatch
  DADO que .claude/worktrees/story-A/ já existe (residual)
  QUANDO /x-epic-implement 0037 tenta criar worktree para story A
  ENTÃO /x-git-worktree create retorna WT_CREATE_FAILED
  E o épico aborta com mensagem instrutiva
  E sugere /x-git-worktree cleanup para resolução

Cenario: Smoke — execution-state.json contém worktreePath
  DADO que /x-epic-implement 0037 está em execução
  QUANDO uma story está IN_PROGRESS
  ENTÃO plans/epic-0037/reports/execution-state.json tem
       stories[story-id].worktreePath populado com path absoluto
  E o path existe no filesystem

Cenario: Boundary — zero stories paralelas (apenas sequenciais)
  DADO que epic-0037 tem todas as stories com dependências (sem batch paralelo)
  QUANDO /x-epic-implement 0037 é executado
  ENTÃO /x-git-worktree create NÃO é chamado (sequencial não usa worktree)
  E execução prossegue sem erro
```
