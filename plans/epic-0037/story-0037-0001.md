## 3. Descrição

Como **platform engineer** mantenedor das skills do `ia-dev-env`, eu quero que a política de worktree (RULE-018) viva como um arquivo de regra de primeira classe em `.claude/rules/14-worktree-lifecycle.md`, eliminando a duplicação inline em `x-git-worktree/SKILL.md` e ganhando autoridade global automática (rules são carregadas em toda sessão), garantindo que todas as skills do épico tenham um único ponto de referência canônico para a política.

Esta é a **história de fundação**: nenhuma outra story do épico pode prosseguir sem o rule file estabelecido, porque todas elas referenciam RULE-018 ao implementar detecção de aninhamento, naming convention, ou ownership de cleanup. O escopo cobre **apenas** a criação do rule file (com conteúdo expandido vs. inline atual) e a substituição do bloco inline por um ponteiro. **Não** modifica comportamento de skill alguma.

### 3.1 Criação do Rule File

Criar `java/src/main/resources/targets/claude/rules/14-worktree-lifecycle.md` no slot 14 (slots 10/11/12 reservados para conditional rules; slot 13 já é `13-skill-invocation-protocol.md`).

Conteúdo do rule file:
- **Seção 1 — Naming Convention**: tabela com padrões `task-XXXX-YYYY-NNN/`, `story-XXXX-YYYY/`, `fix-epic-XXXX/`, `release-X.Y.Z/`, `hotfix-{desc}/`, `{custom}/`.
- **Seção 2 — Branches Protegidas**: `main`, `develop` nunca podem ser checked-out diretamente em worktree.
- **Seção 3 — Invariante de Não-Aninhamento (RULE-002 do épico)**: nenhuma skill cria worktree quando já está em worktree. Mecanismo de detecção: `git rev-parse --show-toplevel` + `git worktree list --porcelain`. Referencia a operação `detect-context` de `x-git-worktree` (introduzida na STORY 2).
- **Seção 4 — Ciclo de Vida**: `criar → usar → branch merged via PR → remove`. Worktrees de stories falhas são preservados para diagnóstico.
- **Seção 5 — Creator Owns Removal (RULE-003 do épico)**: matriz com 5 linhas (epic-implement, story-implement standalone, task-implement standalone, pr-fix-epic, release) declarando quem cria e quem remove em cada cenário.
- **Seção 6 — Quando Usar**: árvore de decisão "se é orquestrador → usa explicit `/x-git-worktree create`; se é skill standalone → expõe flag `--worktree`; se é subagent dentro de orquestrador → detecta e reusa".
- **Seção 7 — Anti-Patterns**: `Agent(isolation:"worktree")` (deprecated), criar worktree em `main`/`develop`, criar nested sem detecção, esquecer cleanup em standalone success.

## 5. Contratos de Dados

### 5.1 Estrutura do Rule File

```markdown
# Rule 14 — Worktree Lifecycle

> **Escopo:** Toda skill que cria branches via `git checkout -b` ou que delega para `/x-git-worktree`.
> **Ownership:** Platform Team.

## 1. Naming Convention

| Contexto | Padrão | Exemplo |
| :--- | :--- | :--- |
| Task | `.claude/worktrees/task-XXXX-YYYY-NNN/` | `.claude/worktrees/task-0037-0003-001/` |
| Story | `.claude/worktrees/story-XXXX-YYYY/` | `.claude/worktrees/story-0037-0003/` |
| Epic Fix | `.claude/worktrees/fix-epic-XXXX/` | `.claude/worktrees/fix-epic-0037/` |
| Release | `.claude/worktrees/release-X.Y.Z/` | `.claude/worktrees/release-2.4.0/` |
| Hotfix | `.claude/worktrees/hotfix-{slug}/` | `.claude/worktrees/hotfix-auth-bug/` |
| Custom | `.claude/worktrees/{identifier}/` | (qualquer) |

## 2. Branches Protegidas

- `main`, `develop` — NUNCA podem ser checked-out diretamente em um worktree.
- `/x-git-worktree create` aborta com `PROTECTED_BRANCH` se `--branch main` ou `--branch develop`.

## 3. Invariante de Não-Aninhamento

> **Regra:** Nenhuma skill DEVE criar um worktree quando já está executando dentro de um worktree.

### Mecanismo de Detecção (canônico)

```bash
# Operation: /x-git-worktree detect-context (ver SKILL.md, Operation 5)
TOPLEVEL=$(git rev-parse --show-toplevel)
if echo "$TOPLEVEL" | grep -q "/.claude/worktrees/"; then
  IN_WORKTREE=true
else
  IN_WORKTREE=false
fi
```

## 4. Ciclo de Vida

```
[criar] ──► [usar] ──► [branch merged via PR] ──► [remove]
                                                      ▲
                                                      │
                              [story falha] ──► [preservar para diagnóstico]
```

## 5. Creator Owns Removal

| Skill | Cria worktree? | Quem remove? | Quando? |
| :--- | :--- | :--- | :--- |
| `x-epic-implement` | Sim, por story (orquestrador) | epic-implement | Após PR da story merged + auto-rebase |
| `x-story-implement` standalone (`--worktree`) | Sim | story-implement | Phase 3 success |
| `x-story-implement` orquestrada | Não (reusa o worktree do epic) | epic-implement | (creator owns) |
| `x-task-implement` standalone (`--worktree`) | Sim | task-implement | Fim da execução |
| `x-task-implement` dentro de story | Não (reusa o worktree da story) | story-implement ou epic-implement | (creator owns) |
| `x-pr-fix-epic` | Sim | pr-fix-epic | Após PR de correção merged |
| `x-release` (post-EPIC-0035) | Sim | release | Após `RESUME-AND-TAG` confirmar tag |

## 6. Quando Usar

```
┌─ Sou orquestrador (lanço subagents)?
│   └─ Sim → Crio worktrees explícitos via /x-git-worktree create antes de despachar
│
├─ Sou skill standalone que pode rodar em paralelo com outras instâncias?
│   └─ Sim → Exponho flag --worktree (opt-in) e crio worktree quando flag presente
│
└─ Sou subagent dentro de um orquestrador?
    └─ Sim → Detecto contexto via /x-git-worktree detect-context e reuso o worktree do orquestrador (NUNCA crio nested)
```

## 7. Anti-Patterns

- ❌ `Agent(isolation:"worktree")` — DEPRECATED. Use `/x-git-worktree create` explícito (ADR-0004).
- ❌ `git checkout -b` direto sem worktree em skills branch-creating — viola este rule file.
- ❌ Criar worktree em `main` ou `develop` — viola Seção 2.
- ❌ Criar nested worktree (worktree dentro de worktree) — viola Seção 3.
- ❌ Esquecer cleanup em standalone success — leak de worktrees em `.claude/worktrees/`.
- ❌ Skill orquestrada removendo worktree do orquestrador — viola Seção 5.
```
