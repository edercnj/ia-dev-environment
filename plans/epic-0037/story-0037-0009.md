# ADR-0004 — Worktree-First Branch Creation Policy for Parallel Skill Execution

**Status:** Accepted
**Date:** 2026-04-XX (data do merge da story)
**Authors:** Platform Team
**Supersedes:** N/A
**Superseded by:** N/A
**Related:** ADR-0003 (Skill Taxonomy), Rule 14 (Worktree Lifecycle)

## Context

Multiple skills in `ia-dev-env` create git branches (`x-git-push`, `x-task-implement`, `x-story-implement`, `x-pr-fix-epic`, `x-release`, `x-epic-implement`). Until EPIC-0037, only `x-epic-implement` used any form of git worktree isolation, and it did so via the harness-native `Agent(isolation:"worktree")` parameter — an "automatic magic" feature that creates a worktree per subagent, with implicit cleanup. Other skills created branches directly in the main checkout via `git checkout -b`.

This caused three problems:
1. **Conflict risk under parallelism**: when multiple instances of branch-creating skills ran concurrently in the same repo (e.g., user invoking `x-story-implement` twice in parallel terminals), they would race on `git checkout -b`, with cryptic git errors and unpredictable state.
2. **Documentation drift**: `x-git-worktree/SKILL.md` claimed integration with `x-epic-implement` that did NOT exist (epic-implement used the harness feature, not the skill). This eroded trust in the skill documentation.
3. **Lack of operator visibility**: with `Agent(isolation:"worktree")`, worktree creation/removal was invisible in execution logs. When subagents failed, operators had no straightforward way to inspect the failed worktree (paths were not exposed by the harness).

We needed a unified policy that:
- Eliminates the conflict risk for ALL branch-creating skills.
- Eliminates the documentation drift.
- Makes worktree lifecycle visible and operable.
- Does not break backward compatibility for existing invocations.

## Decision

We adopt a **Worktree-First Branch Creation Policy** with the following components:

### 1. Explicit `/x-git-worktree create|remove` calls (DEPRECATE harness-native isolation)

All skills that create branches MUST use explicit calls to the `/x-git-worktree` skill (or to its inline `detect_worktree_context` snippet) for worktree management. The harness-native `Agent(isolation:"worktree")` parameter is **DEPRECATED** and removed from `x-epic-implement` (the only previous user) as part of EPIC-0037 STORY 3.

### 2. Opt-in for Standalone Skills, Automatic for Orchestrators

| Skill | Mode |
| :--- | :--- |
| `x-epic-implement` (orchestrator) | **Automatic**: always creates worktrees per parallel story dispatch |
| `x-pr-fix-epic` (orchestrator-like) | **Automatic**: always creates worktree for the correction branch |
| `x-release` (orchestrator-like, post-EPIC-0035) | **Automatic**: always creates worktree for release/hotfix branch |
| `x-story-implement` (standalone or subagent) | **Opt-in via `--worktree`** (standalone) OR auto-detect (when subagent inside epic worktree) |
| `x-task-implement` (standalone or subagent) | Same as story-implement |
| `x-git-push` (utility) | **Opt-in via `--worktree`** flag |

Rationale: orchestrators have no user choice — they always benefit from isolation. Standalone skills serve both interactive use (user wants legacy behavior) and parallel use (user opts in).

### 3. Nesting Invariant (RULE-002 / Rule 14 Section 3)

No skill SHALL create a worktree when already running inside a worktree. The check is mandatory before any `/x-git-worktree create` call. Detection mechanism: `git rev-parse --show-toplevel` + check for `/.claude/worktrees/` substring (canonicalized in `x-git-worktree` Operation 5: `detect-context`).

### 4. Creator Owns Removal (RULE-003 / Rule 14 Section 5)

Whoever creates a worktree is responsible for removing it. Skills running as subagents NEVER remove worktrees they did not create. Failure cases preserve worktrees for diagnostic inspection.

### 5. RULE-018 Promoted to First-Class Rule

The policy is captured in `.claude/rules/14-worktree-lifecycle.md` (slot 14 — slots 10/11/12 reserved for conditional rules; slot 13 is `13-skill-invocation-protocol.md`). This rule file is loaded into every Claude Code session, giving the policy global authority.

## Alternatives Considered

### Alternative A: Keep `Agent(isolation:"worktree")` and just fix docs

**Rejected.** Pros: minimal change, automatic cleanup. Cons: continues to be invisible to operators, perpetuates "magic", does not extend to non-orchestrator skills, leaves the documentation drift in a less-bad-but-still-bad state.

### Alternative B: Hybrid — harness for orchestrators, explicit for standalone

**Rejected.** Pros: pragmatic. Cons: introduces TWO mechanisms in the codebase, requires conditional logic in epic-implement to choose between them, harder to teach new contributors, and the documentation drift would remain partially.

### Alternative C: Per-skill ADRs (one per skill modified)

**Rejected.** The decision is a SINGLE coherent architectural commitment with shared invariants (nesting prevention, creator-owns-removal). Splitting it into per-skill ADRs would fragment what is conceptually one decision.

### Alternative D (chosen): Single ADR + Rule 14 + EPIC-0037

The chosen path. One ADR for the decision and rationale, one rule file for the normative invariants, one epic to implement across all 7 skills.

## Consequences

### Positive

- **Operator visibility**: every worktree create/remove is logged and inspectable via `/x-git-worktree list`.
- **Failure recovery**: failed subagents leave their worktrees behind, allowing manual diagnosis.
- **Single mechanism**: contributors learn one pattern (`/x-git-worktree create` + detect snippet), not two.
- **Documentation truth**: `x-git-worktree/SKILL.md` becomes accurate; the diagram in "Integration with Epic Execution" reflects real flow.
- **Backward compatibility**: standalone skills without `--worktree` flag behave exactly as before.

### Negative

- **Boilerplate**: each skill consumer needs to inline the `detect_worktree_context()` snippet (~6 lines). Mitigation: snippet is canonicalized in `x-git-worktree` Operation 5.
- **Cleanup is now manual for orchestrators**: previously the harness auto-cleaned no-change worktrees. Now `x-epic-implement` must call `/x-git-worktree remove` and run `cleanup --dry-run` defensively. Mitigation: STORY 3 includes explicit cleanup phase.
- **More git operations per epic**: each story creates and removes a worktree explicitly, adding ~2 git commands per story. Mitigation: imperceptible compared to test execution time.

### Per-Skill Consequence Matrix

| Skill | Behavior change | Backward compat |
| :--- | :--- | :--- |
| `x-epic-implement` | `Agent(isolation:"worktree")` REMOVED, replaced with explicit `/x-git-worktree create|remove` | Same end behavior; logs differ |
| `x-story-implement` | New `--worktree` flag; Phase 0 detects context; Phase 3 cleanup conditional | Sem flag = idêntico |
| `x-task-implement` | New `--worktree` flag; branch creation detects context | Sem flag = idêntico |
| `x-git-push` | New `--worktree` flag; Step 1.3 alternate path | Sem flag = idêntico |
| `x-pr-fix-epic` | Always creates worktree (idempotent for resumes) | Comportamento de branch igual; isolamento adicionado |
| `x-release` (post-EPIC-0035) | Always creates worktree for release/hotfix | Comportamento de branch igual |
| `x-git-worktree` | New Operation 5 (`detect-context`); doc drift removed; integration diagram rewritten | Operations 1-4 inalteradas |

## Compliance / Enforcement

- **Code review**: PR reviewers verify presence of `detect_worktree_context()` guard before any `/x-git-worktree create` call.
- **CI grep**: `grep -rn "Agent.*isolation.*worktree" java/src/main/resources/targets/` MUST return empty after EPIC-0037 STORY 3 merges.
- **CI grep**: `grep -rn "git checkout -b" java/src/main/resources/targets/claude/skills/core/{git,dev,pr,ops}/` MUST return only matches in test fixtures or in `x-git-worktree` itself.
- **Smoke test**: `/x-epic-implement` with 2 parallel stories MUST succeed end-to-end without branch conflict.

## Review Triggers

This ADR should be reviewed if any of the following occur:
- Claude Code harness introduces a new isolation mechanism that supersedes `isolation:"worktree"` AND offers operator visibility comparable to explicit calls.
- The cost of explicit `/x-git-worktree` calls becomes a measurable bottleneck (e.g., adds >10% to epic execution time).
- A new branch-creating skill is added to the codebase — the new skill MUST be evaluated against the policy (orchestrator → automatic; standalone → opt-in).
