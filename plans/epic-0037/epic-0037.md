# Épico: Worktree-First Branch Creation Policy

**Autor:** Platform Team
**Data:** 2026-04-11
**Versão:** 1.0
**Status:** Em Refinamento

## 1. Visão Geral

**Chave Jira:** —

Unificar o uso de **git worktrees** em todas as skills do `ia-dev-env` que criam branches, eliminando o risco de conflito quando múltiplas instâncias rodam concorrentemente no mesmo checkout (ex: tasks paralelas dentro de uma story, stories paralelas dentro de um épico, ou múltiplas invocações standalone do usuário). O épico **migra `x-epic-implement` do recurso nativo do harness `Agent(isolation:"worktree")` para chamadas explícitas a `/x-git-worktree create|remove`**, torna **5 skills branch-creating** worktree-aware (`x-story-implement`, `x-task-implement`, `x-git-push`, `x-pr-fix-epic`, `x-release`), promove a **convenção RULE-018 a um arquivo de regra de primeira classe** (`.claude/rules/14-worktree-lifecycle.md`), elimina o **drift de documentação** em `x-git-worktree/SKILL.md` (que descreve uma integração inexistente com `x-epic-implement`), e formaliza a **política dual de isolamento** em **ADR-0004**.

**Motivação imediata:** auditoria das skills core revelou que (a) `x-git-worktree` afirma ser usada por `x-epic-implement` mas **nenhuma skill chama `/x-git-worktree`** — `x-epic-implement` usa exclusivamente `Agent(isolation:"worktree")` (linhas 936–961 do `SKILL.md`); (b) `x-git-push`, `x-task-implement`, `x-story-implement` Phase 0, `x-pr-fix-epic` e `x-release` criam branches via `git checkout -b` direto, sem qualquer isolamento, criando risco real de aninhamento e conflito quando rodam concorrentemente; (c) RULE-018 vive inline em `x-git-worktree/SKILL.md:49-57` sem autoridade global. Este épico fecha essas três lacunas em uma política coerente.

## 3. Definições de Qualidade Globais

### Global Definition of Ready (DoR)

- Source-of-truth de cada skill afetada lida integralmente por quem vai implementar a story
- Branch `feature/epic-0037-*` criada a partir de `develop` atualizado
- `mvn clean verify` verde como baseline antes da story começar
- Layout atual confirmado: EPIC-0036 já mergeado em `develop`; nomes e paths seguem a taxonomia **pós-EPIC-0036** (flat `core/x-*` com `x-epic-implement`, `x-story-implement`, `x-task-implement`, `x-pr-fix-epic`)
- Para STORY 8: confirmado que **EPIC-0035 está completo** (todas as 8 stories merged)

### Global Definition of Done (DoD)

- **Cobertura:** ≥ 95% line, ≥ 90% branch (Rule 05). Zero degradação permitida.
- **Testes Automatizados:** Cada story DEVE ter pelo menos 1 teste (verification, smoke, ou unit) validando o critério de aceite principal. Golden file tests passando após regeneração.
- **Smoke Tests:** `PlatformDirectorySmokeTest` e `AssemblerRegressionSmokeTest` permanecem verdes.
- **Documentação:** `SKILL.md` da skill modificada, `README.md` se aplicável, e referências cruzadas atualizadas.
- **TDD Compliance:** Commits seguem padrão test-first quando aplicável (golden file tests existem antes da regeneração).
- **Build Verde:** Cada story termina com `mvn clean verify` verde antes do merge.
- **Golden Files Consistentes:** Após edição de qualquer `SKILL.md` ou `rules/*.md`, golden files DEVEM ser regenerados no mesmo commit.
- **Source of Truth Compliance:** Zero edições em `.claude/skills/` ou `.claude/rules/` diretamente — toda mudança em `java/src/main/resources/targets/`.
- **Conventional Commits:** `<type>(epic-0037): ...` ou `<type>(story-0037-NNNN): ...`.

## 4. Regras de Negócio Transversais (Source of Truth)

| ID | Título | Descrição |
| :--- | :--- | :--- |
| **[RULE-001]** | Source of Truth Exclusiva (`targets/`) | Toda edição de skills, rules, KPs, agents ou templates DEVE ocorrer em `java/src/main/resources/targets/`. Proibido editar `.claude/`, `.github/` ou outros outputs gerados. Após cada commit que modifica source of truth, golden files DEVEM ser regenerados via `mvn process-resources` + `GoldenFileRegenerator` no mesmo commit (ou commit imediatamente subsequente atômico).<br>Enforcement: CI golden file tests falham se regeneração não ocorreu. |
| **[RULE-002]** | Invariante de Não-Aninhamento de Worktree | Nenhuma skill DEVE criar um worktree quando já está executando dentro de um worktree. A detecção é obrigatória **antes** de qualquer chamada a `/x-git-worktree create`.<br>Mecanismo canônico: `git rev-parse --show-toplevel` + `git worktree list --porcelain` (encapsulado pela operação `detect-context` de `x-git-worktree`).<br>Enforcement: code review verifica presença do guard de detecção em toda chamada a `/x-git-worktree create`. |
| **[RULE-003]** | Creator Owns Removal | Quem cria um worktree é responsável por removê-lo. Skills chamadas como subagent (orquestradas) NUNCA removem worktrees criados pelo orquestrador.<br>Matriz de cleanup:<br>- `x-epic-implement` cria worktree de story → epic remove após PR merge.<br>- `x-story-implement` standalone com `--worktree` → story remove no Phase 3 success.<br>- `x-task-implement` standalone com `--worktree` → task remove ao fim da execução.<br>- `x-pr-fix-epic` cria worktree de fix → pr-fix-epic remove após PR de correção mergeado.<br>- `x-release` cria worktree de release → release remove após `RESUME-AND-TAG` confirmar tag (post-EPIC-0035).<br>Enforcement: testes de cleanup verificam zero worktrees stale após cada cenário. |
| **[RULE-004]** | Backward Compatibility por Default | Nenhuma skill modificada pode quebrar invocações existentes. Worktree-awareness é **opt-in** via flag `--worktree` para skills standalone (`x-story-implement`, `x-task-implement`, `x-git-push`) e **automático** apenas para orquestradores (`x-epic-implement`, `x-pr-fix-epic`, `x-release`) onde não há decisão do usuário a tomar.<br>Enforcement: testes de regressão garantem que invocações sem `--worktree` se comportam idênticas ao baseline pré-épico. |
| **[RULE-005]** | Documentação Reflete Realidade | Toda referência a `Agent(isolation:"worktree")` ou a integrações inter-skill (ex: "x-epic-implement usa x-git-worktree") DEVE refletir o código real. Drift de documentação é tratado como bug bloqueante.<br>Enforcement: STORY 1 deleta a seção "Integration with Epic Execution" atualmente desatualizada; STORY 3 reescreve com diagrama acurado refletindo o fluxo migrado. |
| **[RULE-006]** | RULE-018 Como Rule File de Primeira Classe | A política de worktree (naming convention, branches protegidas, lifecycle, ownership) DEVE viver em `.claude/rules/14-worktree-lifecycle.md` (slot 13 já é skill-invocation-protocol). O conteúdo inline duplicado em `x-git-worktree/SKILL.md:49-57` DEVE ser substituído por um ponteiro `Veja RULE-018`.<br>Enforcement: rule file existe, é referenciado por todas as skills do escopo, e o conteúdo inline foi removido. |
| **[RULE-007]** | Conventional Commits + Rule 08 | Commits do épico seguem formato `<type>(epic-0037): <descrição>` ou `<type>(story-0037-NNNN): <descrição>`.<br>Tipos: `feat`, `fix`, `docs`, `refactor`, `test`, `chore`.<br>Enforcement: hook de lint + code review. |
| **[RULE-008]** | Coordenação com EPIC-0035 | STORY 8 (worktree para `x-release`) **NÃO PODE** começar enquanto EPIC-0035 não estiver completamente mergeado. EPIC-0035 reescreve `x-release/SKILL.md` extensivamente (substituindo `git checkout -b` + `git merge` local por PR-flow via `gh` CLI). STORY 8 deve ser desenhada contra o **novo formato pós-EPIC-0035**, não contra o atual.<br>Enforcement: STORY 8 inicia com Status `Bloqueada (waiting EPIC-0035)`; transição para `Pendente` apenas após verificação de que EPIC-0035 stories 0001–0008 estão todas merged. |
