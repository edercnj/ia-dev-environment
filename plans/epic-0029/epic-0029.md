# Épico: Task-Centric Workflow Overhaul

**Autor:** Architect Agent
**Data:** 2026-04-07
**Versão:** 1.0
**Status:** Pronto

## 1. Visão Geral

**Chave Jira:** —

O workflow atual do `ia-dev-environment` opera com PRs por história — cada story gera um único branch e PR. Tasks dentro de uma história são checkboxes informais sem IDs, sem rastreamento de estado e sem entrega independente. Este épico refatora o workflow para que a **task** se torne a unidade de entrega (1 task = 1 branch = 1 PR), cria skills atômicas para cada etapa do workflow (lint, format, commit, TDD, PR, docs, worktree, plan-task), extrai reviews individuais em skills separadas, e adiciona flag `--auto-approve-pr` para modo de velocidade com branch-mãe. A mudança abrange templates de story, schema de execution-state, 12 skills novas, 12 skills existentes modificadas, e classes Java de checkpoint.

## 3. Definições de Qualidade Globais

### Global Definition of Ready (DoR)

- Spec técnica lida e compreendida
- Templates existentes analisados (formato atual vs desejado)
- Skills existentes lidas (SKILL.md das skills a modificar)
- Classes Java existentes lidas (checkpoint/, assembler/)
- Dependências entre stories resolvidas (stories predecessoras concluídas)

### Global Definition of Done (DoD)

- **Cobertura:** ≥ 95% Line, ≥ 90% Branch
- **Testes Automatizados:** Unitário para Java classes, integração byte-for-byte para golden files, pelo menos 1 teste automatizado por story
- **Smoke Tests:** Obrigatório — verificar que skills geradas aparecem corretamente no output
- **Relatório de Cobertura:** Gerado por `mvn verify`, relatório Jacoco
- **Documentação:** README.md de cada skill nova seguindo template adequado
- **Persistência:** `execution-state.json` serializa/deserializa corretamente com campos de tasks
- **Performance:** Geração completa para todos os 8 perfis em < 60s
- **TDD Compliance:** Commits mostram padrão test-first. Refactoring explícito após green. Testes incrementais via TPP.
- **Double-Loop TDD:** Acceptance tests derivados de Gherkin (outer loop). Unit tests guiados por TPP (inner loop).

## 4. Regras de Negócio Transversais (Source of Truth)

> Regras que se aplicam a múltiplas histórias. Cada história referencia as regras pelo ID.

| ID | Título | Descrição |
| :--- | :--- | :--- |
| **[RULE-001]** | Task como Unidade de Entrega | Cada task gera seu próprio branch (`feat/task-XXXX-YYYY-NNN-desc`) e PR. Tasks são independentemente testáveis e revisáveis. O PR é a entrega mínima, não a story. |
| **[RULE-002]** | Testabilidade Obrigatória | Toda task DEVE seguir um dos padrões válidos (Domain+UnitTest, Port+Adapter+IT, UseCase+AT, Endpoint+APITest, Migration+Smoke, Config+VerificationTest). Anti-patterns proibidos: interface-only, DTO-only, test-only (exceto Layer 4), config-only sem teste. |
| **[RULE-003]** | Approval Gate Humano | Cada task PR requer aprovação humana (AskUserQuestion: APPROVE/REJECT/PAUSE) antes de prosseguir para a próxima task. Exceto em modo `--auto-approve-pr`. |
| **[RULE-004]** | Auto-Approve com Branch-Mãe | Flag `--auto-approve-pr` cria branch-mãe (`feat/story-XXXX-YYYY-desc`) a partir de `develop`. Task PRs são auto-merged na branch-mãe. A branch-mãe NUNCA é auto-merged para `develop` ou `main` — requer review humano. |
| **[RULE-005]** | Git Flow Compliance | Feature branches criadas a partir de `develop`. PRs para `develop` (modo manual) ou parent branch (modo auto-approve). Nunca commit direto em `main` ou `develop`. Naming: lowercase com hyphens. |
| **[RULE-006]** | Task ID Format | IDs seguem `TASK-XXXX-YYYY-NNN` onde XXXX=epic (4 dígitos), YYYY=story (4 dígitos), NNN=sequencial na story (001-999). Branch: `feat/task-XXXX-YYYY-NNN-short-desc`. |
| **[RULE-007]** | Pre-Commit Chain | Todo commit passa pela cadeia: format (`x-format`) → lint (`x-lint`) → compile (`{{COMPILE_COMMAND}}`). Se format modifica arquivos, re-stage automaticamente. |
| **[RULE-008]** | TDD Strict | Ciclos Red-Green-Refactor obrigatórios. RED que passa = ciclo inválido. GREEN com gold-plating = violação. REFACTOR que muda comportamento = novo test primeiro. Ordem TPP obrigatória. |
| **[RULE-009]** | Review Modular | Cada especialista de review tem sua própria skill (x-review-qa, x-review-perf, etc.). O orquestrador x-review apenas delega e consolida. Skills condicionais ativadas por feature gates. |
| **[RULE-010]** | Backward Compatibility | Campo `tasks` no `execution-state.json` é opcional. Épicos antigos sem tasks continuam funcionando — story tratada como single task implícita. Versionamento do schema com campo `version`. |
| **[RULE-011]** | Sizing Constraints | Mínimo 3 tasks por story, máximo 8. Cada task idealmente 50-150 LOC (size M). Tasks < 20 LOC devem ser merged com adjacentes. Tasks > 300 LOC devem ser split. |
| **[RULE-012]** | Cross-Story Task Dependencies | Dependências entre tasks de stories diferentes DEVEM ser refletidas em dependências story-level. Formato: `TASK-XXXX-YYYY-NNN depends on TASK-XXXX-ZZZZ-MMM`. |
| **[RULE-013]** | Batch Approval | Quando múltiplas stories produzem task PRs simultaneamente (parallelism), o usuário recebe prompt consolidado listando todos os PRs pendentes com opções: Aprovar todos / Revisar individualmente / Pausar. |
| **[RULE-014]** | Resume por Task | Resume verifica status de PRs de cada task via `gh pr view`. IN_PROGRESS → PENDING. PR merged → DONE. PR closed → FAILED. BLOCKED reevaluado se deps mudaram. |
| **[RULE-015]** | Value Delivery | Cada story DEVE ter Seção 3.5 (Entrega de Valor) com valor de negócio mensurável. Proibido: "Implementar repositório". Correto: "Persistência com integridade referencial". |
| **[RULE-016]** | Conventional Commits com Task ID | Formato: `<type>(<TASK-XXXX-YYYY-NNN>): <subject> [TDD:TAG]`. TDD tags: `[TDD]`, `[TDD:RED]`, `[TDD:GREEN]`, `[TDD:REFACTOR]`. Subject ≤ 72 chars, imperative mood. |
| **[RULE-017]** | Documentation Sync | Documentação deve ser atualizada em sincronia com mudanças de código. Skill x-docs detecta tipo de doc necessário (API, README, ADR, changelog) e delega. |
| **[RULE-018]** | Worktree Lifecycle | Worktrees criados para tasks/stories paralelas. Naming: `.claude/worktrees/{identifier}/`. Cleanup automático: branch merged, ou > 7 dias sem commits, ou branch inexistente. |
