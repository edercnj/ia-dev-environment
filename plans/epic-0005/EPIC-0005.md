# Épico: Epic Orchestrator — Implementação Automatizada de Épicos Completos

**Autor:** Product Owner / Architect
**Data:** 2026-03-15
**Versão:** 1.0
**Status:** Em Refinamento

## 1. Visão Geral

**Chave Jira:** EPIC-0005

Criar a skill `x-dev-epic-implement`, uma orquestradora que implementa TODAS as histórias de um épico
automaticamente, delegando cada história a subagents isolados para evitar estouro de janela de contexto.
O orchestrator principal funciona como um "gerente de projeto" leve — nunca lê código-fonte, knowledge
packs ou diffs diretamente. Ele lê o implementation-map gerado por `x-story-map`, constrói o grafo de
execução, despacha subagents por story (via lógica do `x-dev-lifecycle`), executa integrity gates entre
fases, gerencia falhas com retry e block propagation, persiste estado em checkpoint files para
resumabilidade, e suporta execução paralela via git worktrees. O escopo inclui: schema de estado de
execução e checkpoint engine, template de relatório de execução, parser de implementation-map, loop de
orquestração com dispatcher sequencial, integrity gates entre fases, failure handling (retry + block
propagation transitiva), resumabilidade (`--resume`), execução parcial (`--phase N`, `--story`),
paralelismo com worktrees (`--parallel`), consolidação final (review + report + PR), dry-run mode,
progress reporting com métricas de execução, e testes E2E do orchestrator. A skill consome internamente
`x-dev-lifecycle`, `x-review`, `x-review-pr`, `x-git-push`, e todos os knowledge packs (lidos apenas
pelos subagents, nunca pelo orchestrator).

## 3. Definições de Qualidade Globais

### Global Definition of Ready (DoR)

- Spec do Epic Orchestrator (este documento) lida e compreendida
- Skills dependentes (`x-dev-lifecycle`, `x-review`, `x-review-pr`, `x-git-push`) estáveis e funcionais
- Templates existentes (`_TEMPLATE-STORY.md`, `_TEMPLATE-EPIC.md`, `_TEMPLATE-IMPLEMENTATION-MAP.md`) versionados
- Estrutura de `docs/stories/epic-XXXX/` com `execution-state.json` schema definido
- Convenção SD-09 de diretórios documentada e seguida
- Regras de dual copy (`resources/skills-templates/core/` + `resources/github-skills-templates/`) compreendidas
- Comportamento do Agent tool com `isolation: "worktree"` compreendido

### Global Definition of Done (DoD)

- **Cobertura:** ≥ 95% Line, ≥ 90% Branch
- **Testes Automatizados:** Unitários, integração (golden file tests). Cenários Gherkin cobertos.
- **TDD Compliance:** Commits mostram padrão test-first. Refactoring explícito após green. Testes incrementais via TPP.
- **Double-Loop TDD:** Acceptance tests derivados do Gherkin (outer loop). Unit tests guiados por TPP (inner loop).
- **Relatório de Cobertura:** Vitest coverage report com thresholds validados
- **Documentação:** SKILL.md, templates atualizados em ambas as cópias (`resources/` + `.claude/` + `.github/`)
- **Backward Compatibility:** Skills existentes permanecem intactas. `x-dev-epic-implement` é puramente aditiva.
- **Checkpoint Integrity:** `execution-state.json` sempre consistente — atualizado após cada story, nunca corrompido por falha parcial.

## 4. Regras de Negócio Transversais (Source of Truth)

> Regras que se aplicam a múltiplas histórias. Cada história referencia as regras pelo ID. Alterações de regra propagam automaticamente para todas as histórias dependentes.

| ID | Título | Descrição |
| :--- | :--- | :--- |
| **[RULE-001]** | Context Isolation | O orchestrator NUNCA lê código-fonte, knowledge packs ou diffs. Todo trabalho pesado é delegado a subagents com contexto isolado. O orchestrator mantém em contexto apenas: conteúdo do implementation-map, tabela de status por story, commit SHAs, contadores de retry, e checkpoint file path. |
| **[RULE-002]** | Checkpoint After Every Story | Após cada story completar (SUCCESS ou FAILED), o `execution-state.json` DEVE ser atualizado antes de qualquer próxima ação. Se o processo morrer entre uma story e o checkpoint, a story é considerada IN_PROGRESS no resume e será re-despachada. |
| **[RULE-003]** | Dependency Satisfaction | Uma story só pode ser despachada se TODAS as suas dependências (conforme implementation-map) têm status SUCCESS. Dependências com status FAILED, BLOCKED, PENDING ou IN_PROGRESS impedem o despacho. |
| **[RULE-004]** | Integrity Gate Mandatory | Entre cada fase, TODOS os testes do projeto devem passar e coverage thresholds (≥ 95% line, ≥ 90% branch) devem ser atendidos. Não há bypass. Falha no gate aciona diagnóstico de regressão para identificar qual story da fase causou a quebra. |
| **[RULE-005]** | Retry Budget | Cada story tem no máximo 2 retries. O subagent do retry recebe o erro anterior como contexto adicional. Após 2 retries sem sucesso, a story é marcada FAILED permanentemente. |
| **[RULE-006]** | Block Propagation Transitiva | Se story A é FAILED e story B depende de A, B é marcada BLOCKED. Se story C depende de B, C também é BLOCKED. A propagação percorre todas as dependências transitivas. O orchestrator continua executando stories não-bloqueadas na mesma fase e fases subsequentes. |
| **[RULE-007]** | Critical Path Priority | Dentro de uma fase com múltiplas stories executáveis, stories no caminho crítico do implementation-map devem ser executadas primeiro (em modo sequencial) ou iniciadas primeiro (em modo paralelo). Isso minimiza o impacto de eventuais falhas no prazo total. |
| **[RULE-008]** | Subagent Result Contract | Todo subagent DEVE retornar um objeto com: `{ status: "SUCCESS" \| "FAILED" \| "PARTIAL", commitSha?: string, findingsCount: number, summary: string }`. O orchestrator rejeita respostas que não sigam este contrato e marca a story como FAILED. |
