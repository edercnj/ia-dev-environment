# Épico: Planejamento Multi-Agente de Histórias com Tasks Detalhadas

**Autor:** Product Owner / Architect
**Data:** 2026-04-07
**Versão:** 1.0.0
**Status:** Em Refinamento

## 1. Visão Geral

**Chave Jira:** —

Implementar um sistema de planejamento multi-agente que ocorre ANTES do desenvolvimento, substituindo as sub-tarefas genéricas atuais (`[Dev]`, `[Test]`, `[Doc]`) por tasks detalhadas geradas por 5 agentes especializados (Architect, QA Engineer, Security Engineer, Tech Lead, Product Owner). Cada agente contribui tasks com critérios de DoD específicos, resultando em histórias verdadeiramente prontas para desenvolvimento com validação de DoR automatizada. Inclui persistência de planos individuais por task para retomada em caso de interrupção, e integração com o ciclo de implementação existente (`x-dev-lifecycle`, `x-dev-implement`, `x-dev-epic-implement`) via modo PRE_PLANNED.

Escopo inclui: 2 novas skills (`x-story-plan`, `x-epic-plan`), 3 novos templates, 4 templates modificados, 3 skills existentes adaptadas, e atualização do gerador Java (SkillGroupRegistry, PlanTemplatesAssembler) com regeneração de golden files.

Escopo exclui: mudanças em definições de agentes (agent definitions), criação de novos agentes, mudanças em knowledge packs, e mudanças em regras (.claude/rules/).

## 3. Definições de Qualidade Globais

### Global Definition of Ready (DoR)

- Templates de saída existem e estão acessíveis em `.claude/templates/` ou `java/src/main/resources/shared/templates/`
- Story tem Gherkin com ≥ 4 cenários cobrindo categorias obrigatórias (degenerate, happy, error, boundary)
- Dependências da story estão concluídas ou são mockáveis
- Contratos de dados definidos com campos, tipos e validações

### Global Definition of Done (DoD)

- **Cobertura:** ≥ 95% Line, ≥ 90% Branch
- **Testes Automatizados:** Unitários + integração para toda lógica de negócio. Cada história DEVE ter pelo menos 1 teste automatizado validando o critério de aceite principal.
- **Smoke Tests:** Obrigatório quando testing.smoke_tests == true. Cada história deve passar no smoke gate.
- **Relatório de Cobertura:** JaCoCo XML com granularidade por classe
- **Documentação:** SKILL.md atualizado, README.md presente em skills novas
- **Persistência:** Artefatos gerados salvos em `plans/epic-XXXX/plans/` com convenção flat
- **Performance:** Geração de golden files em < 60s por profile
- **TDD Compliance:** Commits show test-first pattern. Explicit refactoring after green. Tests are incremental (from simple to complex via TPP — Transformation Priority Premise).
- **Double-Loop TDD:** Acceptance tests derived from Gherkin scenarios (outer loop). Unit tests guided by TPP (inner loop).

## 4. Regras de Negócio Transversais (Source of Truth)

> Regras que se aplicam a múltiplas histórias. Cada história referencia as regras pelo ID.

| ID | Título | Descrição |
| :--- | :--- | :--- |
| **[RULE-001]** | Backward Compatibility | Todas as mudanças DEVEM ser backward compatible. Skills que consomem artefatos pré-existentes DEVEM degradar graciosamente quando novos artefatos não existem. O modo INLINE (comportamento atual) é preservado como fallback. Nenhuma funcionalidade existente pode quebrar. |
| **[RULE-002]** | Padrão de Staleness (mtime) | Todos os novos artefatos DEVEM seguir o padrão de staleness checking via comparação de `mtime` usado em todo o codebase: `mtime(source) <= mtime(artifact)` → reusa; `mtime(source) > mtime(artifact)` → regenera; artifact ausente → gera. |
| **[RULE-003]** | Template Verbatim Copy | Novos templates DEVEM ser copiados verbatim pelo `PlanTemplatesAssembler`. Tokens `{{PLACEHOLDER}}` são preenchidos pelo LLM em runtime, NÃO pelo template engine em build-time. Seções obrigatórias validadas antes da cópia. |
| **[RULE-004]** | Convenção Flat de Arquivos | Novos arquivos de planejamento DEVEM seguir a convenção flat existente: `plans/epic-XXXX/plans/{type}-story-XXXX-YYYY.md`. Subdiretórios por história são proibidos para manter compatibilidade com 40+ referências hardcoded em 7 skills. |
| **[RULE-005]** | Embedding de Prompts de Agente | Instruções de PLANNER mode para subagentes DEVEM ser embutidas nos prompts dentro do SKILL.md da skill orquestradora, NÃO nos arquivos de definição de agentes (`.claude/agents/`). Agentes existentes mantêm seus roles inalterados. |
| **[RULE-006]** | Conteúdo em pt-BR | Todo conteúdo gerado por skills de planejamento DEVE estar em português brasileiro (pt-BR), exceto termos técnicos (cache, handler, endpoint), identificadores de código, e IDs (RULE-NNN, TASK-NNN, story-XXXX-YYYY). |
