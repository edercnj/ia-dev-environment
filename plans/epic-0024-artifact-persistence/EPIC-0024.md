# Épico: Persistência e Padronização de Artefatos de Planejamento e Revisão — ia-dev-environment

**Autor:** Claude Code (Opus 4.6)
**Data:** 2026-04-05
**Versão:** 1.0
**Status:** Em Refinamento

## 1. Visão Geral

**Chave Jira:** —

O gerador `ia-dev-environment` produz skills que orquestram ciclos completos de desenvolvimento assistido por IA — planejamento arquitetural, planos de teste TDD, implementação, reviews paralelos por engenheiros especialistas, e review holístico do Tech Lead. Atualmente, os artefatos produzidos por essas skills (planos de implementação, test plans, reviews, avaliações de segurança) não possuem templates padronizados, não verificam existência prévia antes de regenerar, e não são persistidos de forma consistente. Isso resulta em: (a) outputs com formato variável entre sessões, (b) desperdício de tokens quando sessões são retomadas, (c) impossibilidade de revisão humana sistemática, e (d) perda de trabalho quando execuções são canceladas.

Este épico introduz 12 templates padronizados para todos os tipos de artefatos, um assembler Java para distribuí-los na geração, e modificações em 8 skills para referenciar templates e verificar idempotência antes de regenerar.

**Escopo incluído:**
- 12 novos templates de artefatos (planos, assessments, reviews, dashboards, reports)
- 1 novo assembler Java (`PlanTemplatesAssembler`) seguindo padrão existente
- Modificações em 8 skills para pre-checks de idempotência e referência a templates
- Novo diretório `reports/` para relatórios intermediários de fase
- Dashboard consolidado de reviews e tracking de remediação
- Golden tests para todos os novos templates

**Escopo excluído:**
- Mudanças na lógica de negócio das skills (apenas formato de output e persistence)
- Novos agents ou knowledge packs
- Integração com ferramentas externas de reporting
- Mudanças no `execution-state.json` (schema existente mantido)

## 3. Definições de Qualidade Globais

### Global Definition of Ready (DoR)

- Templates anteriores (Phase 0) disponíveis antes de modificar skills
- `PlanTemplatesAssembler` funcional antes de testar skills com referência a templates
- Spec file `SPEC-artifact-persistence-v1.md` lido e compreendido
- Padrão do `EpicReportAssembler` estudado como referência

### Global Definition of Done (DoD)

- **Cobertura:** ≥ 95% Line, ≥ 90% Branch para código Java novo
- **Testes Automatizados:** Golden tests para todos os profiles incluindo novos templates. Testes unitários para validação de seções obrigatórias. Cada história DEVE ter pelo menos 1 teste automatizado validando o critério de aceite principal.
- **Smoke Tests:** Obrigatório. Cada história deve passar no smoke gate.
- **Relatório de Cobertura:** JaCoCo integrado ao `mvn verify`
- **Documentação:** CLAUDE.md atualizado com catálogo de artefatos ao final do épico
- **Persistência:** Templates copiados verbatim sem renderização de placeholders
- **Performance:** Geração não deve aumentar tempo de build em mais de 5%
- **TDD Compliance:** Commits show test-first pattern. Explicit refactoring after green. Tests are incremental (from simple to complex via TPP — Transformation Priority Premise).
- **Double-Loop TDD:** Acceptance tests derived from Gherkin scenarios (outer loop). Unit tests guided by TPP (inner loop).

## 4. Regras de Negócio Transversais (Source of Truth)

> Regras que se aplicam a múltiplas histórias. Cada história referencia as regras pelo ID. Alterações de regra propagam automaticamente para todas as histórias dependentes.

| ID | Título | Descrição |
| :--- | :--- | :--- |
| **[RULE-001]** | Template obrigatório para artefatos | Todo artefato de planejamento ou avaliação produzido por uma skill DEVE seguir o template padronizado correspondente em `.claude/templates/`. Templates definem seções obrigatórias que a LLM DEVE preencher. Seções condicionais (DB schema, event schema) são incluídas apenas quando o projeto possui a capability correspondente. |
| **[RULE-002]** | Idempotência via staleness check | Antes de gerar qualquer artefato, a skill DEVE verificar se o artefato já existe. Se `mtime(story) <= mtime(plan)`, reutilizar e logar `Reusing existing {artifact} from {date}`. Se story foi modificada após o plano, regenerar. |
| **[RULE-003]** | Templates language-agnostic | Templates DEVEM conter marcadores `{{LANGUAGE}}`, `{{FRAMEWORK}}`, `{{DATABASE}}`, `{{ARCHITECTURE}}` preenchidos em runtime pela LLM. Templates NÃO são renderizados pelo engine durante geração — copiados verbatim. |
| **[RULE-004]** | Dual-target copy | O `PlanTemplatesAssembler` DEVE copiar templates para `.claude/templates/` e `.github/templates/`. DEVE validar seções obrigatórias antes de copiar. |
| **[RULE-005]** | Score numérico parseable | Todo template de review DEVE incluir score no formato `XX/YY` e status `Approved`/`Rejected`/`Partial`, permitindo parsing automático por skills de consolidação. |
| **[RULE-006]** | Dashboard cumulativo | O dashboard consolidado DEVE ser cumulativo: criado por `x-review`, atualizado por `x-review-pr`. Histórico de rounds preservado. |
| **[RULE-007]** | Instrução explícita de template | Skills que delegam a subagentes DEVEM instruí-los: "Read template at `.claude/templates/_TEMPLATE-{TYPE}.md` for required output format". |
| **[RULE-008]** | Implementation plan completo | O template de implementation plan DEVE incluir: (a) class diagram Mermaid, (b) method signatures, (c) affected layers table, (d) TDD strategy com mapeamento UT-N/AT-N, (e) mini-ADRs. Seções condicionais: DB schema, API changes, Event changes. |
| **[RULE-009]** | Modelo profundo para planejamento | Planejamento de implementação DEVE usar `model: opus` nas seções de skills que delegam planejamento a subagentes. |
| **[RULE-010]** | Validação de seções obrigatórias | O assembler DEVE validar seções obrigatórias por template. Se ausente no source, logar warning e não copiar. |
| **[RULE-011]** | Header padronizado | Todo artefato persistido DEVE incluir header com: Story/Epic ID, Data de geração, Autor (role), Template Version. |
| **[RULE-012]** | Fallback graceful | Skills DEVEM funcionar quando templates não estão disponíveis (projetos pré-EPIC-0024). Se template ausente, usar formato inline como fallback e logar warning. |
