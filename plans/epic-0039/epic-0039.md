# Épico: x-release — Interactive Flow, Auto-Versioning, Smart Resume & Observability

**Autor:** Tech Lead (gerado via `/x-story-epic-full`)
**Data:** 2026-04-13
**Versão:** 1.0
**Status:** Em Refinamento

## 1. Visão Geral

**Chave Jira:** —

Evoluir a skill `/x-release` (post-EPIC-0035) de um fluxo dirigido por argumentos para um orquestrador interativo, auto-versionado e resiliente a interrupções. Inclui auto-detecção da próxima versão a partir de Conventional Commits, prompts em cada ponto de pausa com handoff direto para `/x-pr-fix-comments`, smart resume de state files órfãos, paralelização de VALIDATE-DEEP, GitHub Release automático com confirmação, telemetria por fase, e fechamento didático com diagrama Git Flow. O escopo termina em paridade no fluxo de hotfix e regeneração das golden files dos 17 profiles. Fora do escopo: releases multi-módulo, integração Jira/ADO por fase, publicação em registries externos além do GitHub Release, rollback automático pós-tag.

### Global Definition of Ready (DoR)

- Spec da story aprovada com data contracts, Gherkin scenarios e diagrama

### Global Definition of Done (DoD)

- **Cobertura:** ≥ 95% line, ≥ 90% branch (Rule 05 — `jacoco:check` passa)
- **Testes Automatizados:** unit + integration obrigatórios; smoke onde aplicável; cada story com pelo menos 1 teste validando o critério principal
- **Smoke Tests:** `PipelineSmokeTest` e `ExpectedArtifactsTest` continuam verdes; manifest regenerado quando aplicável
- **Relatório de Cobertura:** jacoco HTML + XML em `target/site/jacoco/`
- **Documentação:** `SKILL.md` da `x-release` reflete novos defaults; references atualizadas; CHANGELOG `[Unreleased]` com entry
- **Persistência:** state file v2 com migração explícita (sem upgrade silencioso de v1)
- **Performance:** VALIDATE-DEEP wall-clock reduzida ≥ 40% após paralelização (S04)
- **TDD Compliance:** commits mostram pattern test-first; refactoring explícito após green; testes incrementais via TPP
- **Double-Loop TDD:** acceptance tests derivados dos cenários Gherkin (outer loop); unit tests guiados por TPP (inner loop)
- **Golden Files:** regenerados uma única vez (S14b) após SKILL.md final, byte-for-byte iguais nos 17 profiles
- **Cross-File Consistency:** alterações em `SKILL.md` source propagam para os 17 profiles via `GoldenFileRegenerator`

## 4. Regras de Negócio Transversais (Source of Truth)

| ID | Título | Descrição |
| :--- | :--- | :--- |
| **[RULE-001]** | Source-of-truth: gerador, não output | Toda alteração em skills/templates DEVE ser feita em `java/src/main/resources/targets/claude/`. Edição direta em `.claude/` é proibida (será sobrescrita pelo gerador). |
| **[RULE-002]** | Conventional Commits para auto-versão | Auto-detecção de versão (S01) consome `git log <tag>..HEAD` e classifica por prefixo: `feat!`/`BREAKING CHANGE` → MAJOR; `feat` → MINOR; `fix`/`perf` → PATCH. Commits sem prefixo conhecido são ignorados; quando todos forem ignorados, abortar com `VERSION_NO_BUMP_SIGNAL`. |
| **[RULE-003]** | Schema evolution explícito | Mudanças no state file schema bumpam `schemaVersion`. v1→v2 é breaking — abortar com `STATE_SCHEMA_VERSION` e instruções de migração. Sem upgrade silencioso. |
| **[RULE-004]** | Prompts têm equivalente não-interativo | Todo `AskUserQuestion` introduzido pela skill DEVE ter flags equivalentes (`--no-prompt`, `--version`, `--continue-after-merge`, etc.) para uso em CI sem TTY. |
| **[RULE-005]** | Preservação de codes existentes | A paralelização de VALIDATE-DEEP (S04) DEVE manter os mesmos `VALIDATE_*` error codes da implementação sequencial atual; apenas a ordem de saída pode mudar (sorted by severity). |
| **[RULE-006]** | Telemetria fica no repo | `plans/release-metrics.jsonl` é commitado (não gitignored). Contém uma linha JSON por fase: `{releaseVersion, phase, startedAt, endedAt, durationSec, outcome}`. Append-only. Comparações usam as últimas 5 releases. |
| **[RULE-007]** | Confirmação obrigatória para GitHub Release | Phase 11 PUBLISH SEMPRE pergunta antes de criar GitHub Release (mesmo após primeira execução). Flag `--no-github-release` desliga em CI. |
| **[RULE-008]** | Golden regen consolidado | Regeneração de goldens dos 17 profiles ocorre uma única vez (S14b), após `SKILL.md` finalizado. Stories intermediárias NÃO regeneram goldens individualmente. |
| **[RULE-009]** | Smart Resume substitui STATE_CONFLICT | Quando state file existe e operador invoca `/x-release`, NÃO abortar com `STATE_CONFLICT`. Em vez disso, prompt actionable: resume / abort / start-new. |
| **[RULE-010]** | Cleanup seguro com double-confirm | `/x-release --abort` exige double confirm + dry-run prévio listando recursos a destruir (PRs, branches local + remoto, state file). |
