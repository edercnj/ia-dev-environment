## 3. Descrição

Como **release manager**, eu quero que `/x-release` detecte sozinho a próxima versão a partir dos commits desde a última tag, garantindo que eu não precise calcular `MAJOR.MINOR.PATCH` manualmente nem errar bump.

Hoje a skill exige `<version>` como argumento posicional. Operadores precisam ler `git log` e classificar commits por tipo na cabeça. Após esta story, basta rodar `/x-release` — a próxima versão é detectada e exibida no banner inicial. Override explícito continua disponível via `--version X.Y.Z` para releases fora do padrão (rebrand, downgrade controlado, etc.).

### 3.1 Algoritmo de detecção

- Ler última tag via `git describe --tags --abbrev=0 --match 'v*'`
- Listar commits via `git log <last-tag>..HEAD --format=%s%n%b`
- Classificar cada commit pela primeira linha (`type(scope)?: subject` ou `type!(scope)?: subject`) e pelo body (procurar `BREAKING CHANGE:`)
- Bump rules:
  - `feat!` ou `fix!` ou `BREAKING CHANGE:` no body → MAJOR
  - `feat:` (sem `!`) → MINOR
  - `fix:` ou `perf:` → PATCH
  - `docs:`, `chore:`, `test:`, `refactor:`, `style:`, `build:`, `ci:` → ignorar
- Aplicar maior bump detectado sobre a última versão (ex.: 3.1.0 + MINOR → 3.2.0)
- Se nenhum commit dispara bump → abortar com `VERSION_NO_BUMP_SIGNAL`

### 3.2 CLI surface

- `<version>` posicional torna-se opcional
- Novo flag `--version X.Y.Z` para override explícito (mutuamente exclusivo com auto-detecção)
- Banner de abertura: `"Próxima versão detectada: 3.2.0 (MINOR) — 7 feat, 2 fix, 0 breaking desde v3.1.0"`
- Erro de override inválido (formato não SemVer): `VERSION_INVALID_FORMAT`

### 3.3 Edge cases

- Sem tag prévia (primeira release): assumir `0.0.0` como base; bump produz `0.1.0` (MINOR), `0.0.1` (PATCH) ou `1.0.0` (MAJOR)
- Histórico vazio (zero commits desde tag): `VERSION_NO_COMMITS`
- Apenas commits ignoráveis (docs/chore): `VERSION_NO_BUMP_SIGNAL` com sugestão de `--version` explícito

## 3.5 Entrega de Valor

- **Valor Principal:** elimina cálculo manual de versão e o erro mais comum em releases (bump errado de minor vs patch)
- **Métrica de Sucesso:** 100% das releases pós-implementação rodadas sem `--version` explícito; zero releases com bump inconsistente com os commits
- **Impacto no Negócio:** reduz friction operacional; release manager foca em validar conteúdo, não em contar commits

## 5. Contratos de Dados

### 5.1 Input (CLI)

| Campo | Tipo | M/O | Validações | Exemplo |
| :--- | :--- | :--- | :--- | :--- |
| `<version>` (posicional) | `String` | O | regex `^\d+\.\d+\.\d+(-[a-z0-9.]+)?$` | `3.2.0` |
| `--version <X.Y.Z>` | `String` | O | mesmo regex | `--version 3.2.0` |
| `--last-tag <tag>` (debug) | `String` | O | tag existente | `--last-tag v3.1.0` |

### 5.2 Output (banner + state)

| Campo | Tipo | Sempre presente | Descrição |
| :--- | :--- | :--- | :--- |
| `targetVersion` | `String` | Sim | Versão computada (auto ou override) |
| `previousVersion` | `String\|null` | Não (null em primeira release) | Versão da última tag |
| `bumpType` | `enum{major,minor,patch,explicit}` | Sim | Origem do bump |
| `commitCounts` | `{feat,fix,perf,breaking,ignored}` | Sim | Contagem por tipo desde última tag |

### 5.3 Error Codes

| Exit | Code | Condição |
| :--- | :--- | :--- |
| 1 | `VERSION_NO_BUMP_SIGNAL` | Sem commits versionáveis desde última tag |
| 1 | `VERSION_NO_COMMITS` | `git log <tag>..HEAD` vazio |
| 1 | `VERSION_INVALID_FORMAT` | `--version` não bate com regex SemVer |
| 1 | `VERSION_TAG_NOT_FOUND` | `git describe` falha (sem tag e `--last-tag` inválido) |

## 7. Critérios de Aceite (Gherkin)

```gherkin
Cenario: Sem commits versionáveis desde última tag (degenerate)
  DADO que a última tag é v3.1.0
  E todos os commits desde v3.1.0 são apenas docs/chore
  QUANDO eu rodo /x-release sem argumentos
  ENTÃO o exit code é 1
  E a saída contém VERSION_NO_BUMP_SIGNAL
  E sugere usar --version explicito

Cenario: Auto-detecção MINOR (happy path)
  DADO que a última tag é v3.1.0
  E há 7 commits feat e 2 commits fix desde v3.1.0
  QUANDO eu rodo /x-release sem argumentos
  ENTÃO o banner exibe "Próxima versão detectada: 3.2.0 (MINOR)"
  E o state file inicial registra targetVersion=3.2.0 e bumpType=minor

Cenario: Override explícito tem precedência
  DADO que a auto-detecção sugeriria 3.2.0
  QUANDO eu rodo /x-release --version 4.0.0
  ENTÃO targetVersion=4.0.0 e bumpType=explicit
  E o banner indica "Versão explícita: 4.0.0"

Cenario: Detecção MAJOR via BREAKING CHANGE no body (boundary)
  DADO que há commits feat com "BREAKING CHANGE:" no body
  QUANDO eu rodo /x-release sem argumentos
  ENTÃO bumpType=major

Cenario: Formato de versão inválido (error path)
  QUANDO eu rodo /x-release --version 3.2
  ENTÃO o exit code é 1
  E a saída contém VERSION_INVALID_FORMAT

Cenario: Primeira release sem tag prévia (boundary at-min)
  DADO que o projeto não tem nenhuma tag v*
  E há 5 commits feat desde a raiz
  QUANDO eu rodo /x-release sem argumentos
  ENTÃO previousVersion=null e targetVersion=0.1.0
```

### 7.1 TPP Ordering

Cenários ordenados de degenerate (no commits) → happy path (auto MINOR) → override → boundary (BREAKING via body, primeira release) → error (formato inválido).

### 7.2 Mandatory Categories Coverage

- [x] Degenerate: sem commits versionáveis
- [x] Happy path: auto MINOR
- [x] Error: formato inválido
- [x] Boundary: BREAKING via body, primeira release sem tag
