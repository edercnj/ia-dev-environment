# Épico: Context Window Resilience

**Autor:** Arquiteto de Software
**Data:** 2026-04-08
**Versão:** 1.0
**Status:** Em Refinamento

## 1. Visão Geral

**Chave Jira:** —

Reduzir o consumo de janela de contexto durante a execução de tasks, stories e epics pelos skills de orquestração (`x-dev-epic-implement`, `x-dev-lifecycle`, `x-tdd`). A cadeia de execução atual consome ~94K tokens apenas em instruções de skills, sem contar rules (~14K), knowledge packs (~24K por subagent) e outputs intermediários. Este épico introduz mecanismos de context budget tracking, compressão de instruções via references sob demanda, carregamento seletivo de knowledge packs, compactação de outputs, auditoria de isolamento de subagents e progressive loading com slim mode. O target é redução de ~50% no consumo de tokens da cadeia de skills core.

## 3. Definições de Qualidade Globais

### Global Definition of Ready (DoR)

- Spec de contexto de janela lida e compreendida
- Templates de skills acessíveis em `java/src/main/resources/targets/claude/skills/core/`
- Métricas de baseline coletadas (linhas/tokens por skill)
- Dependências entre stories mapeadas

### Global Definition of Done (DoD)

- **Cobertura:** ≥ 95% Line, ≥ 90% Branch
- **Testes Automatizados:** Testes de integração (`mvn verify -Pintegration-tests`) passando para todos os profiles. Golden files regenerados e byte-for-byte match.
- **Smoke Tests:** Obrigatório. Cada skill modificado deve ser invocável com `--dry-run` sem erros.
- **Relatório de Cobertura:** JaCoCo HTML + XML
- **Documentação:** SKILL.md atualizado, reference files documentados
- **Persistência:** N/A
- **Performance:** Redução mensurável no tamanho dos SKILL.md core (target: -50%)
- **TDD Compliance:** Commits show test-first pattern. Explicit refactoring after green. Tests are incremental (from simple to complex via TPP — Transformation Priority Premise).
- **Double-Loop TDD:** Acceptance tests derived from Gherkin scenarios (outer loop). Unit tests guided by TPP (inner loop).

## 4. Regras de Negócio Transversais (Source of Truth)

| ID | Título | Descrição |
| :--- | :--- | :--- |
| **[RULE-001]** | Preservação Funcional | A separação em `SKILL.md` core + `references/*.md` DEVE preservar a funcionalidade completa do skill. O core contém workflow, flags e fases. As references contêm detalhes de implementação de cada fase, schemas e lógica condicional. |
| **[RULE-002]** | Degradação Graceful de References | O core de cada skill DEVE ser auto-suficiente para execução básica. Se uma reference file não existir, o skill DEVE degradar gracefully com warning (consistente com RULE-012 do projeto). |
| **[RULE-003]** | Documentação de Contexto Mínimo | Skills que invocam outros skills via `Skill` tool DEVEM documentar o contexto mínimo que o skill invocado precisa. O skill invocado DEVE ter um "slim mode" que carrega apenas esse mínimo. |
| **[RULE-004]** | Output por Referência | Outputs intermediários (checkpoint reads, phase reports, review dashboards) DEVEM ser salvos em arquivo e referenciados por path, NUNCA acumulados inline no contexto do orquestrador. |
| **[RULE-005]** | Isolamento de Subagents | Subagents DEVEM receber apenas metadata (story ID, branch, phase, flags). Source code, diffs e conteúdo de KPs NUNCA devem ser passados inline no prompt do subagent — o subagent deve ler esses arquivos por conta própria. |
| **[RULE-006]** | Context Budget Informativo | O campo `context-budget` no frontmatter de skills é informativo para o orquestrador e NÃO afeta o carregamento do skill pelo Claude Code. É consumido apenas por skills orquestradores que decidem entre execução inline vs. delegação a subagent. |
