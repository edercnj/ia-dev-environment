## 3. Descrição

Como **orquestrador** (`x-epic-implement` ou `x-story-implement`), eu quero estender `x-pr-create` com flags `--target-branch`, `--auto-merge`, `--epic-id` para que o orquestrador possa propagar OO-style: `--target-branch epic/0049 --auto-merge merge --epic-id 0049`.

Esta story é **chave para o novo fluxo** — sem ela, RULE-001/002/009 não funcionam.

### 3.1 Novos flags

- `--target-branch <branch>` (default: comportamento atual — `develop` ou parent story branch)
  - Quando passado, vira o `--base` do `gh pr create`
- `--auto-merge <strategy>` (default `none`)
  - Strategies: `merge | squash | rebase | none`
  - Quando != `none`: pós-create, invoca `Skill x-pr-merge --pr <N> --strategy <strategy> --auto`
- `--epic-id <ID>` (M opcional)
  - Adiciona label `epic-XXXX` à PR
  - Adiciona à description metadata

### 3.2 Backward compatibility (RULE-008)

- Sem `--target-branch`: comportamento idêntico ao atual (lógica de `--auto-approve-pr`)
- Sem `--auto-merge`: PR não é auto-mergeada (comportamento atual)
- Sem `--epic-id`: nenhum label `epic-*` é adicionado

### 3.3 Validações

- Se `--target-branch` é `develop` E `--auto-merge != none`: warning "Auto-merge directly to develop is unusual; consider using epic branch"
- Se `--epic-id` passado mas regex `^\d{4}$` falha: error
- `--auto-merge` requer `--target-branch` (sem ele, ambíguo)

## 3.5 Entrega de Valor

- **Valor Principal:** Habilita propagação OO-style de parâmetros (RULE-009); chave para todo o novo fluxo `epic/XXXX`. Sem esta story, S18 e S19 não conseguem entregar o novo modelo.
- **Métrica de Sucesso:** Após S18+S19, 100% das stories executadas em modo "novo flow" usam `--target-branch epic/XXXX` propagado.
