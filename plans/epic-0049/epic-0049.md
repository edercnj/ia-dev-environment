# Épico: Refatoração do Fluxo de Épico — Sequencial Default, Branch Única e Orquestradores Thin

**Autor:** Eder Junior
**Data:** 2026-04-16
**Versão:** 1.0
**Status:** Pronto

## 1. Visão Geral

**Chave Jira:** —

Refatorar o fluxo de execução de épicos em `ia-dev-environment`, mudando os defaults de `x-epic-implement` (paralelo + auto-merge para `develop`) para um modelo mais seguro: **sequencial + auto-merge para uma branch única `epic/XXXX`** com PR final manual para `develop`. Aplicar princípio "thin orchestrator" (UseCase pattern) tanto a `x-epic-implement` quanto a `x-story-implement`, extraindo 15 sub-skills novas (4 públicas + 11 internas com prefixo `x-internal-*`). Adicionar versionamento Git automático às 7 skills de planejamento. **Inclui:** novas Rules 21 (Epic Branch Model) e 22 (Skill Visibility) — Rule 20 já existe (`20-telemetry-privacy.md`), por isso os próximos IDs livres —, atualização de Rules 09/14/19, backward compatibility via `--legacy-flow` + `flowVersion`. **Exclui:** migration helper para épicos v1 mid-flight (`x-internal-epic-migrate-flow` listada para épico futuro).

## 3. Definições de Qualidade Globais

### Global Definition of Ready (DoR)

- Plano de design aprovado pelo product owner
- Branch `epic/0049` criada (será criada manualmente até STORY-0049-0008 estar implementada)
- Spec da spec disponível no working tree
- Branch `develop` no estado limpo no início de cada story
- Para stories de refator (S18/S19): snapshot do comportamento v1 capturado em testes canônicos

### Global Definition of Done (DoD)

- **Cobertura:** ≥ 95% Line, ≥ 90% Branch
- **Testes Automatizados:** Cada nova skill SKILL.md tem golden tests gerados via `GoldenFileRegenerator` para os 17 stacks suportados. Cada extensão de skill existente tem testes adicionais validando os novos flags. Cada Rule nova/atualizada tem audit script no CI.
- **Smoke Tests:** Obrigatório. Smoke gate ao fim do épico verifica fluxo end-to-end com EPIC-9999 fictício.
- **Relatório de Cobertura:** JaCoCo XML + HTML, granularidade por package em `target/site/jacoco/`
- **Documentação:** SKILL.md de cada nova skill segue convenção (frontmatter completo + body com seções padrão). Skills internas têm bloco visível 🔒 INTERNAL SKILL.
- **Persistência:** N/A (ferramenta de geração; sem persistência de runtime)
- **Performance:** Tempo de execução de `x-epic-implement` (modo dry-run) não pode aumentar mais de 10% vs baseline EPIC-0048
- **TDD Compliance:** Commits show test-first pattern. Explicit refactoring after green. Tests are incremental (from simple to complex via TPP — Transformation Priority Premise).
- **Double-Loop TDD:** Acceptance tests derived from Gherkin scenarios (outer loop). Unit tests guided by TPP (inner loop).
- **Audit Rule 13 + Rule 22:** Zero matches em scripts de audit (delegação via `Skill()` correta; visibility frontmatter consistente).

## 4. Regras de Negócio Transversais (Source of Truth)

| ID | Título | Descrição |
| :--- | :--- | :--- |
| **[RULE-001]** | Branch única por épico (`epic/XXXX`) | Toda atividade de um épico (planejamento + execução) acontece em uma única branch `epic/XXXX` criada no nascimento do épico. PR final `epic/XXXX → develop` é o gate manual do épico. Branch é excluída do conjunto de protected branches do `x-git-cleanup-branches`.<br>Aplicação: `x-epic-create`, `x-epic-decompose`, `x-epic-orchestrate`, `x-epic-implement`, `x-story-implement` (quando `--target-branch epic/XXXX` propagado). |
| **[RULE-002]** | Auto-merge default ON, target = `epic/XXXX` | Stories PRs são auto-mergeadas pelo GitHub native auto-merge na branch `epic/XXXX` (não em `develop`). Strategy default: `merge` (preserva commits por task com TDD tags). Strategy configurável via `--auto-merge-strategy <merge\|squash\|rebase>`.<br>Override: `--auto-merge=false` desabilita; `--legacy-flow` força target em `develop`. |
| **[RULE-003]** | Default sequencial em `x-epic-implement` | Execução sequencial de stories é o novo default. Paralelo é opt-in via `--parallel`. Razão: paralelo causa conflitos crônicos quando stories irmãs tocam arquivos sobrepostos.<br>Em modo `--parallel`, worktrees usam `epic/XXXX` como base (não `develop`). |
| **[RULE-004]** | Estratégia de merge: preserva history | Auto-merge default usa `merge commit` (não squash, não rebase). Preserva commits por task com TDD tags (RED/GREEN/REFACTOR), permitindo bisect granular pós-merge final em `develop`.<br>Merge final `epic/XXXX → develop` também é merge commit. |
| **[RULE-005]** | Thin orchestrator (UseCase pattern) | `x-epic-implement` e `x-story-implement` são UseCases puros: arg parsing → sequence de chamadas a sub-skills → agregação. Nenhuma operação shell direta (git/gh/jq/mvn) sobrevive no SKILL.md do orquestrador além de leituras passivas (`Read`, `Glob`).<br>Aplicação cascata: `x-epic-implement` (orquestra stories) → `x-story-implement` (orquestra tasks) → `x-task-implement` (executa TDD via `x-test-tdd`). |
| **[RULE-006]** | Convenção `x-internal-*` para skills internas | Skills não destinadas a invocação direta pelo usuário usam prefixo `x-internal-`. Frontmatter obrigatório: `visibility: internal` + `user-invocable: false`. Body começa com bloco visível 🔒 INTERNAL SKILL. Subdir convention: `internal/{group}/x-internal-{name}/`.<br>Generator filtra skills internas de `/help` e do menu de slash commands. |
| **[RULE-007]** | Skills de planejamento devem versionar | Toda skill que escreve sob `plans/epic-XXXX/` (`x-epic-create`, `x-epic-decompose`, `x-epic-map`, `x-epic-orchestrate`, `x-story-create`, `x-story-plan`, `x-task-plan`) DEVE delegar para `x-planning-commit` no final. Working tree fica limpo após qualquer comando.<br>`--dry-run` propaga e suprime o commit. |
| **[RULE-008]** | Backward compatibility via `flowVersion` + `--legacy-flow` | `execution-state.json` ganha campo top-level `flowVersion: "1" \| "2"`. Auto-detecção: ausente ou `"1"` → força legacy mode (warning visível); `"2"` → novo flow. Flag explícito `--legacy-flow` força legacy.<br>Janela de depreciação: 2 releases após merge do EPIC-0049. Stories órfãs (PR já em `develop`) tratadas como legacy. |
| **[RULE-009]** | Propagação OO-style de parâmetros | Parâmetros `--target-branch`, `--auto-merge`, `--epic-id` propagam-se transparentemente: `x-epic-implement` → `x-story-implement` → `x-pr-create`. Cada nível DEVE aceitar e repassar; nunca derivar implicitamente.<br>Sem flag explícito → comportamento legacy (target `develop`, sem auto-merge). |
| **[RULE-010]** | Skills internas pequenas (token budget) | Sub-skills internas chamadas frequentemente (`x-internal-status-update`, `x-planning-commit`, `x-git-merge`) DEVEM ser projetadas para ≤ 150 linhas SKILL.md. Skills mais complexas (`x-internal-epic-build-plan`) podem ir até 250 linhas.<br>Razão: cada `Skill()` reinjeta o body completo no contexto. |
