## 3. Descrição

Como **Desenvolvedor do gerador ia-dev-env**, eu quero um **template Pebble** em `java/src/main/resources/shared/templates/CLAUDE.md` implementando o contrato formalizado em ADR-0048-B, acompanhado de um teste de sintaxe **`ClaudeMdTemplateSyntaxTest`** que valida que o template é parseable pela engine Pebble e renderiza sem exceção com um contexto dummy, garantindo que STORY-0048-0011 (Bug B fix — `ClaudeMdAssembler` + registro em `AssemblerFactory`) consuma um template pronto, testado em isolamento, e semanticamente completo.

Esta story é **standalone** do tronco de remoção (pode rodar em paralelo a 0003–0008 conforme DAG da spec seção 8). O template precisa estar pronto antes de STORY-0048-0011 porque o assembler depende dele, mas não depende de nenhum mapping restrito a Java ter sido propagado nos maps derivados (o template usa placeholders Pebble que o assembler popula em runtime com valores vindos do `ProjectContext`). A única pré-condição dura é ADR-0048-B em `Accepted` (mergeada via story-0048-0002), pois o schema de placeholders é autoritativo do ADR.

O template é um arquivo markdown com cabeçalhos fixos e placeholders Pebble (`{{ variable }}`, `{% if ... %}`). Seções obrigatórias (por RULE-048-05 e espírito do CLAUDE.md raiz do próprio repo): **Project Overview** (consome `{{PROJECT_NAME}}`, `{{LANGUAGE}}`, `{{FRAMEWORK}}`), **Build** (consome `{{BUILD_COMMAND}}`), **Test** (consome `{{TEST_COMMAND}}`), **Architecture Notes** (consome `{{ARCHITECTURE}}`, opcional bloco `{% if DATABASES %}Databases: {{DATABASES}}{% endif %}`, idem para `INTERFACE_TYPES`), e **Key Rules / Related Skills** (conteúdo estático curto apontando para `.claude/` estrutura). O teste de sintaxe usa `PebbleEngine` já instanciado em `TemplateEngine.java` (ou constrói instância equivalente) para `parseString` ou `getTemplate` e depois `evaluate` com um `Map<String,Object>` mínimo; assert que não lança `PebbleException` e que o output contém os marcadores das seções obrigatórias.

### 3.1 Template `shared/templates/CLAUDE.md`

- Local: `java/src/main/resources/shared/templates/CLAUDE.md`.
- Formato: markdown com placeholders Pebble válidos.
- Placeholders obrigatórios (por ADR-0048-B §3.2 e spec §13): `{{ PROJECT_NAME }}`, `{{ LANGUAGE }}`, `{{ FRAMEWORK }}`, `{{ ARCHITECTURE }}`, `{{ DATABASES }}`, `{{ INTERFACE_TYPES }}`, `{{ BUILD_COMMAND }}`, `{{ TEST_COMMAND }}`.
- Estrutura mínima (cabeçalhos H1/H2 exatos):
  - `# {{ PROJECT_NAME }}` (H1 único)
  - `## Project Overview` — descreve que o projeto é `{{ LANGUAGE }}` com `{{ FRAMEWORK }}`
  - `## Build` — bloco code com `{{ BUILD_COMMAND }}`
  - `## Test` — bloco code com `{{ TEST_COMMAND }}`
  - `## Architecture Notes` — cita `{{ ARCHITECTURE }}`, bloco condicional `{% if DATABASES %}Databases: {{ DATABASES }}{% endif %}` e `{% if INTERFACE_TYPES %}Interfaces: {{ INTERFACE_TYPES }}{% endif %}`
  - `## Key Rules` — 3–5 bullets estáticos pointing para `.claude/rules/` como referência
  - `## Related Skills` — 1 frase estática apontando para `.claude/skills/`
- Comprimento-alvo: 40–120 linhas (suficientemente rico para justificar "executive summary auto-loaded"; suficientemente curto para não duplicar `.claude/README.md`).

### 3.2 Teste `ClaudeMdTemplateSyntaxTest`

- Local: `java/src/test/java/dev/iadev/template/ClaudeMdTemplateSyntaxTest.java`.
- Estratégia: usar `PebbleEngine` (mesma instância/configuração que `TemplateEngine.java` usa em produção — se houver factory, reusar; senão, instanciar `new PebbleEngine.Builder().build()` em setUp).
- Cenários:
  1. **existsAndReadable**: `Files.exists(Path.of("src/main/resources/shared/templates/CLAUDE.md"))` true; `Files.readString` não lança.
  2. **parsesWithoutException**: `engine.getTemplate("shared/templates/CLAUDE.md")` retorna `PebbleTemplate` não-null.
  3. **rendersWithDummyContext_doesNotThrow**: contexto `Map.of("PROJECT_NAME","demo","LANGUAGE","java","FRAMEWORK","spring","ARCHITECTURE","hexagonal","DATABASES","postgresql","INTERFACE_TYPES","REST","BUILD_COMMAND","mvn package","TEST_COMMAND","mvn test")`; `template.evaluate(new StringWriter(), ctx)` não lança.
  4. **renderedOutputContainsMandatorySections**: output da renderização contém substrings `"# demo"`, `"## Project Overview"`, `"## Build"`, `"## Test"`, `"## Architecture Notes"`.
  5. **handlesAbsentOptionalDatabases**: contexto sem `DATABASES` → output não contém `"Databases:"` (bloco `{% if %}` respeitado).
- Todos cenários parametrizados ou individuais conforme conveniente; cobertura do próprio teste não se aplica (é teste — sem classe produtiva nova fora do template).

## 3.5 Entrega de Valor

- **Valor Principal:** Template-fonte pronto e validado para o `ClaudeMdAssembler` de STORY-0048-0011; schema Pebble testado em isolamento remove risco de erro de sintaxe quando o assembler entrar no pipeline.
- **Métrica de Sucesso:** `shared/templates/CLAUDE.md` existe; `ClaudeMdTemplateSyntaxTest` com 5 cenários verdes; coverage do teste ≥ 95%/90% — não há classe produtiva nova, mas o teste cobre 100% das branches condicionais do template (`{% if %}` blocks); `mvn verify` verde.
- **Impacto no Negócio:** Qualidade dos projetos gerados aumenta com CLAUDE.md auto-loaded — Claude Code passa a ter contexto executivo em todo projeto Java gerado, melhorando experiência do desenvolvedor final.

### 7.1 Scenario Ordering (TPP)

Ordem TPP: existência (degenerate file check) → parse (unconditional estrutural) → render happy (unconditional full) → assert conteúdo (validação de output) → branch condicional (boundary / edge case).

### 7.2 Mandatory Scenario Categories

- [x] Degenerate cases (ausência da chave opcional DATABASES)
- [x] Happy path (render com contexto completo)
- [x] Error paths (cobertos indiretamente: parse failure seria exceção — se o template for inválido, cenário 2 falha)
- [x] Boundary values (bloco condicional respeitado sem a chave)

### 7.3 TDD Implementation Notes

- **Double-Loop TDD**: cenário 2 (parse) é o acceptance test externo; cenários 1, 3, 4, 5 são inner-loop unit tests.
- **TPP**: existência → parse → render → conteúdo → branch condicional.
- O teste precede ou acompanha o template no commit history (test-first pattern).
