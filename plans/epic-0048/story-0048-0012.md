## 3. Descrição

Como **Maintainer** do gerador `ia-dev-env`, eu quero um smoke test E2E final `Epic0048EndToEndTest` que invoca o CLI real via `ProcessBuilder` em 2 perfis Java (java-spring + java-quarkus), valida os invariantes de Bug A e Bug B (zero dirs vazios + CLAUDE.md presente e com bytes suficientes), exercita exit codes corretos para linguagens removidas, e remove quaisquer `@Disabled` órfãos deixados pelas stories anteriores — garantindo que o épico entregue um gerador limpo, sem testes skipped, com invariantes validados em nível de processo (não apenas em `@TempDir` interno).

Esta story é a "ponte de validação integrada" entre as stories de fix (0009 Bug A, 0011 Bug B) e a story final (0013 audit + release). Até aqui, os invariantes foram validados em testes de integração (`OutputDirectoryIntegrityTest`, `ClaudeMdRootPresenceTest`) que rodam o `AssemblerPipeline` diretamente — rápido, mas não cobre o caminho CLI completo (parsing de flags, `GenerateCommand.run`, output real em filesystem). O `Epic0048EndToEndTest` fecha essa gap: usa `ProcessBuilder.command("java", "-jar", "<jar>", "generate", "--profile", "java-spring", "--out", tempDir)`, captura exit code + stderr, e assertiona contra os mesmos invariantes no output real no disco. 2 perfis cobrem os dois frameworks suportados (spring + quarkus) — os outros 7 goldens Spring já são cobertos em outros layers.

Simultaneamente, a story elimina débito técnico residual acumulado durante STORY-0048-0003 a STORY-0048-0011: qualquer `@Disabled` que tenha sobrado de testes legados (e.g. testes parametrizados em linguagens removidas que alguém desabilitou em vez de deletar — violação da DoD "Zero Skipped Tests" do épico), mensagens `"skipped profile"` em logs de teste, etc. E adiciona testes negativos adicionais para cobertura de borda da validação de linguagem: config YAML declara `language: python` sem flag CLI `--language` (cenário que poderia contornar `CliLanguageValidator`), flag `--language` vazia (não-NPE), e combinação de valores inválidos com outros params válidos.

### 3.1 Epic0048EndToEndTest — CLI real, 2 perfis, A+B integrados

- Novo arquivo `java/src/test/java/dev/iadev/e2e/Epic0048EndToEndTest.java`.
- Estratégia: `ProcessBuilder` invoca o jar empacotado (pré-condição: `mvn package` antes dos testes — task 001 garante o jar via maven-failsafe ou invocação explícita).
- Alternativa aceitável se `ProcessBuilder` sobre jar for pesado no CI: invocar `picocli.CommandLine.execute(...)` com args split — ainda exercita toda a stack da CLI exceto o `main()` literal. Decisão documentada no PR body.
- Cobertura: 2 perfis Java (`java-spring`, `java-quarkus`) em tempdirs separados, além de ≥3 cenários negativos (linguagens removidas, flag vazia, YAML ambíguo).
- Asserts por perfil:
  1. Exit code == 0
  2. `Files.walk(tempDir).filter(Files::isDirectory).filter(empty)` retorna 0 dirs (invariante Bug A — RULE-048-04)
  3. `Files.exists(tempDir/CLAUDE.md)` == true (invariante Bug B — RULE-048-05)
  4. `Files.size(tempDir/CLAUDE.md) > 100` (conteúdo não-vazio)
  5. `Files.readString(tempDir/CLAUDE.md).contains("{{")` == false (placeholders resolvidos)

### 3.2 Testes negativos adicionais (CliLanguageValidationTest extensions)

- Estender `java/src/test/java/dev/iadev/cli/CliLanguageValidationTest.java` (criado em STORY-0048-0003) com novos cenários:
  - Flag `--language` vazia (`"--language", ""`): deve falhar com mensagem clara, NÃO NPE
  - Flag `--language java` explícita: deve passar (regressão happy)
  - Config YAML declara `language: python` sem flag CLI: deve falhar cedo em `ConfigLoader` (não silenciar fallback)
  - Flag `--language python` + `--framework spring`: deve falhar por linguagem, não por framework (ordem de validação)
  - Linguagem com whitespace (`--language "java "`): trimming consistente ou erro claro (nunca silent acceptance com string suja)

### 3.3 Higienização: @Disabled órfãos + logs "skipped profile"

- Executar: grep `@Disabled` em `java/src/test` e `grep -rn "skipped profile" java/src/test`.
- Para cada hit: decidir ação — (a) se o teste cobre linguagem removida, DELETAR o teste inteiro (não manter desabilitado); (b) se o teste cobre cenário java ainda válido mas quebrado temporariamente, remover `@Disabled` e consertar (bug conhecido → nova issue se não-trivial); (c) se é teste de feature experimental sem relação com épico, documentar em `plans/epic-0048/reports/residual-disabled.md` para triagem futura (permitido APENAS se claramente fora de escopo).
- Meta numérica final: `grep -rn "@Disabled" java/src/test | wc -l` = 0 novos hits introduzidos por este épico (baseline: o que já existia ANTES de STORY-0048-0001 é tolerado mas documentado).

## 3.5 Entrega de Valor

- **Valor Principal:** pipeline completo (CLI → pipeline → filesystem) validado end-to-end contra os 2 bugs fechados (A + B) em um único teste; débito técnico de `@Disabled` órfãos eliminado; cobertura sanitária de edge cases para validação de linguagem aumenta robustez.
- **Métrica de Sucesso:** `Epic0048EndToEndTest` verde (2 perfis × ≥5 asserts cada); `grep @Disabled` em testes NÃO retorna hits novos introduzidos pelo épico; `CliLanguageValidationTest` cresce para ≥8 cenários (incluindo novos negativos); `mvn verify` verde ≤ tempo baseline + 5% (E2E CLI pode adicionar 10-30s — documentar).
- **Impacto no Negócio:** confiança de release — usuário que clona v4.0.0 e roda `ia-dev-env generate --profile java-spring` tem garantia automatizada de que output é limpo (zero dirs vazios) e completo (CLAUDE.md presente); regressão futura em qualquer dos invariantes falha CI imediatamente.

```gherkin

  DADO que o jar foi empacotado via mvn package

  ENTÃO o processo retorna exit code 0

  DADO mesmas pré-condições

  ENTÃO todos os mesmos invariantes acima são satisfeitos
  E o conteúdo de CLAUDE.md contém a string "quarkus" (framework-aware)

Cenario: ERROR — CLI rejeita --language python com mensagem exata RULE-048-06
  DADO mesmas pré-condições

  ENTÃO exit code != 0
  E stderr contém "Language 'python' is not supported. Only 'java' is available (see CHANGELOG v4.0.0 / EPIC-0048)."

  DADO um setup-config.yaml customizado com "language: python"

  ENTÃO exit code != 0
  E stderr contém referência a RULE-048-06
  E nenhum arquivo é gerado (fail-fast antes do pipeline)

Cenario: BOUNDARY — flag --language "" (vazia) é rejeitada sem NPE
  DADO mesmas pré-condições
```
