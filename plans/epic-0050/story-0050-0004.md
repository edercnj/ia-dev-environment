## 3. Descrição

Como **engenheiro de plataforma**, eu quero declarar `model: haiku` em 10 skills que se qualificam para Haiku pela RULE-005 (utilitárias sem raciocínio de design OU knowledge packs read-only), para capturar ~15% da redução de tokens prevista no épico. Haiku é ~2,5× mais barato que Sonnet e suficiente para essas tarefas.

### 3.1 Alvos categorizados

**Utilitárias (git ops, format/lint):**
| Skill | Justificativa |
| :--- | :--- |
| `x-git-worktree` | Git ops — sem raciocínio de design |
| `x-git-commit` | Git commit — apenas invoca git CLI |
| `x-code-format` | Delega para formatter (prettier, black, gofmt) |
| `x-code-lint` | Delega para linter |

**Knowledge Packs (read-only references):**
| Skill | Justificativa |
| :--- | :--- |
| `architecture` | KP consumido como referência textual |
| `coding-standards` | KP consumido como referência |
| `testing` | KP consumido como referência |
| `layer-templates` | Templates de código (copy-paste) |
| `patterns` | Referência de padrões |
| `dockerfile` | Referência de patterns Docker |

### 3.2 Mudança exata

Adicionar `model: haiku` entre `name:` e `description:` no frontmatter de cada arquivo.

## 3.5 Entrega de Valor

- **Valor Principal:** Captura ~15% da redução de tokens via Haiku-eligibility (~1.200 tokens/execução). Estabelece um "tier barato" para trabalho utilitário.
- **Métrica de Sucesso:** 10 arquivos declaram `model: haiku`; invocações de `x-git-commit` e `x-code-format` após o merge rodam em Haiku (telemetria confirma).
- **Impacto no Negócio:** Reduz custo de operações frequentes mas triviais (cada story executa múltiplas chamadas a `x-git-commit` e `x-code-format`).
