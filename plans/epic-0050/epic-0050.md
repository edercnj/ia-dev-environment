# Épico: Model Selection Enforcement & Token Optimization

**Autor:** Eder Junior
**Data:** 2026-04-17
**Versão:** 1.0
**Status:** Pronto

## 1. Visão Geral

**Chave Jira:** —

Reduzir o consumo de tokens em modelos premium (Opus) no projeto `ia-dev-environment` aplicando **seleção determinística de modelo** (Opus / Sonnet / Haiku) em três pontos de enforcement: (a) **frontmatter `model:`** nas skills orquestradoras, utilitárias e knowledge packs; (b) **parâmetro `model:` em `Agent(subagent_type: "general-purpose", ...)`** nas skills que despacham subagents; (c) **parâmetro `model:` em `Skill(skill: "...", ...)`** nas cadeias de orquestração. Complementado por (d) metadata determinístico `Recommended Model` nos 10 agents (eliminando o placeholder "Adaptive") e (e) **CI audit script** que falha PRs introduzindo regressões. **Inclui:** uma Rule nova (Model Selection Strategy), refactors em ~15 SKILL.md alvo, atualização dos 10 agents, script de audit bash e uma medição pós-deploy via hooks de telemetria (EPIC-0040). **Exclui:** migração automática de skills fora da matriz declarada (backward compatibility via Rule 07 do épico). Baseline atual: **84,4% Opus / 0,2% Sonnet / 5,8% Haiku**. Target pós-épico: **≤50% Opus / ≥35% Sonnet / ≥12% Haiku**. Redução estimada: **~22% de tokens por execução de epic completo (~7.850 tokens)**.

## 4. Regras de Negócio Transversais (Source of Truth)

| ID | Título | Descrição |
| :--- | :--- | :--- |
| **[RULE-001]** | Orquestradores declaram `model:` no frontmatter | Toda skill que despacha múltiplos `Agent()` ou `Skill()` DEVE declarar `model:` no frontmatter YAML. Default para orquestradores é `sonnet`. Exceção: skills dedicadas a design arquitetural profundo (`x-arch-plan`) podem usar `opus`.<br>Aplicação: `x-epic-implement`, `x-story-implement`, `x-release`, `x-review`, `x-epic-orchestrate`, `x-pr-fix-epic`, `x-task-implement`, `x-epic-decompose`. |
| **[RULE-002]** | `Agent(...)` com `model:` explícito | Toda invocação `Agent(subagent_type: "general-purpose", ...)` dentro de um skill DEVE passar o parâmetro `model:` explícito. Herança implícita do pai é proibida em delegações para subagent.<br>Aplicação cascata: 5 subagents em `x-story-plan` (Architect=opus, QA/Security/TL/PO=sonnet); subagents em `x-arch-plan` e `x-test-plan`. |
| **[RULE-003]** | `Skill(...)` com `model:` param em orquestradores | Toda invocação `Skill(skill: "...", ...)` dentro de um orquestrador DEVE passar `model:` quando o skill invocado tem perfil de tier diferente do pai. Exceção: skills invocadas diretamente pelo usuário (entry points) — herança vale apenas nesse caso.<br>Aplicação: `x-epic-implement → Skill("x-story-implement", model:"sonnet")`; `x-story-implement → Skill("x-git-commit", model:"haiku")`; `x-review → Skill("x-review-*", model:"sonnet")`. |
| **[RULE-004]** | Agents declaram `Recommended Model` determinístico | Agents em `.claude/agents/*.md` DEVEM declarar `Recommended Model` com valor determinístico (`Opus`, `Sonnet` ou `Haiku`). O valor `Adaptive` é proibido.<br>Matriz default: `architect`=Opus; `product-owner`=Sonnet (mudança de Opus); demais 8 agents de review/implementação = Sonnet. |
| **[RULE-005]** | Haiku-eligibility por categoria | Skills elegíveis para `model: haiku` são: (a) utilitárias sem raciocínio de design (git ops, format, lint), OU (b) knowledge packs read-only consumidos como referência.<br>Lista inicial: `x-git-worktree`, `x-git-commit`, `x-code-format`, `x-code-lint`, `architecture`, `coding-standards`, `testing`, `layer-templates`, `patterns`, `dockerfile`. |
| **[RULE-006]** | CI audit script obrigatório | O CI DEVE executar `scripts/audit-model-selection.sh` em cada PR. O script falha quando detecta: (a) skill de orquestração sem `model:` no frontmatter; (b) `Agent(...)` sem `model:` em skill de orquestração; (c) `Skill(...)` sem `model:` em orquestrador; (d) agent com `Recommended Model: Adaptive`.<br>Exit code 0 = pass; ≥1 = fail. |
| **[RULE-007]** | Backward compatibility (escopo aditivo) | O épico NÃO altera skills fora da matriz declarada. A seleção de modelo é **aditiva**: adiciona `model:` onde não existe; nunca remove nem modifica skills fora do escopo. O audit script cobre apenas os alvos declarados; pode ser expandido em épicos futuros.<br>Skills existentes que já rodam em Opus e não estão na matriz continuam sem mudança. |
