# Épico: Zero-Bypass Lifecycle Enforcement

**Autor:** Eder Celeste Nunes Junior (LLM-assisted)
**Data:** 2026-04-26
**Versão:** 1.0
**Status:** Em Refinamento

> **Status Transitions (Rule 22 — lifecycle-integrity):**
> artifacts lifecycle-controlados (Story/Task) usam o enum canônico
> `Pendente | Planejada | Em Andamento | Concluída | Falha | Bloqueada`.
> O campo Status do Épico aqui é documental e reflete o estado
> agregado das histórias filhas. Transições permitidas do enum:
> `Pendente → Planejada | Em Andamento | Falha | Bloqueada`;
> `Planejada → Em Andamento | Falha | Bloqueada`;
> `Em Andamento → Concluída | Falha | Bloqueada`;
> reabertura `Concluída → Em Andamento` (via `x-status-reconcile --apply`) e
> `Falha → Pendente`; `Bloqueada → Pendente | Planejada | Em Andamento | Falha`.
> Ver [`.claude/rules/22-lifecycle-integrity.md`](../../.claude/rules/22-lifecycle-integrity.md).

## 1. Visão Geral

**Chave Jira:** —

Após análise diagnóstica de EPIC-0054 a 0057, identificou-se que nenhum desses 4 épicos materializou os 6 artefatos obrigatórios de planejamento por story (Phase 1B–1F de `x-story-implement`). A causa raiz: o orquestrador foi sistematicamente bypassado — stories foram implementadas manualmente, abrindo branches, editando código, commitando e criando PRs fora do fluxo controlado. As 10 camadas de enforcement existentes (CI audits, hooks, testes de integridade) assumem implicitamente que o orquestrador está rodando; quando o operador pula o orquestrador inteiramente, nenhuma camada dispara.

EPIC-0059 estabelece **enforcement total** ("zero-bypass") do lifecycle de stories e tasks: nenhum PR pode mergear em `develop` ou `main` sem cadeia completa e auditável de evidência produzida pelos orquestradores canônicos. O escopo cobre:

1. Extensão de audits existentes para exigir 6 artefatos Fase 1 (Phase 0).
2. Defesa upstream: pre-commit hooks, PreToolUse hooks e Stop hooks que alertam o operador durante a sessão (Phase 1).
3. PR-gate intransponível: CI re-roda pre-commit chain, PR template obriga evidência de orquestrador, telemetria comprova execução (Phase 2).
4. Plataforma: branch protection, Rule 26 normativa, anistia formal para EPIC-0054–0057, e conformidade de `taskTracking` (Phase 3).

## 3. Definições de Qualidade Globais

### Global Definition of Ready (DoR)

- Spec da story ≥ 1 página com Gherkin de aceitação
- Predecessoras todas em estado terminal (`Concluída`, `Falha` ou `Bloqueada`)
- Branch `epic/0059` existe e está atualizada com `develop`
- `plans/epic-0059/plans/` tem os 6 artefatos de planejamento da story: `arch-story-XXXX-YYYY.md`, `plan-story-XXXX-YYYY.md`, `tests-story-XXXX-YYYY.md`, `tasks-story-XXXX-YYYY.md`, `security-story-XXXX-YYYY.md`, `compliance-story-XXXX-YYYY.md` (auto-imposto via RULE-059-01 — dogfooding)
- Audit script alvo identificado (existente ou novo) e seu exit code reservado (ver RULE-059-06)
- Baseline file path identificado (se aplicável)

### Global Definition of Done (DoD)

- **Implementação e Merge:** Story implementada via `/x-story-implement` (telemetria em `plans/epic-0059/telemetry/events.ndjson` comprova); PR mergeada em `epic/0059`
- **Artefatos Obrigatórios:** 6 artefatos Fase 1 + 4 artefatos Fase 3 presentes: `arch-story-*.md`, `plan-story-*.md`, `tests-story-*.md`, `tasks-story-*.md`, `security-story-*.md`, `compliance-story-*.md`, `verify-envelope-*.json`, `review-story-*.md`, `techlead-review-story-*.md`, `story-completion-report-*.md`
- **Cobertura:** ≥ 95% line, ≥ 90% branch nos arquivos tocados (Rule 05 — absolute gate, sem grandfathering)
- **Testes Automatizados:** ≥ 1 acceptance test que valida que o gate do bypass coberto dispara; smoke test obrigatório (RULE-059-02)
- **Smoke Tests:** Obrigatório quando `testing.smoke_tests == true`
- **TDD Compliance:** Commits show test-first pattern (test precedes implementation in git log), explicit refactoring after green, tests are incremental (simple to complex via TPP)
- **Double-Loop TDD:** Acceptance tests derived from Gherkin scenarios (outer loop); unit tests guided by Transformation Priority Premise (inner loop)
- **Self-Check:** Audit script / hook adicionado retorna `exit 0` quando rodado em happy-path
- **CHANGELOG:** Atualizado (`### Added` ou `### Changed`)
- **CLAUDE.md:** Épico mencionado no bloco "In progress" / "Concluded"
- **Baseline:** Qualquer mudança em `audits/*-baseline.txt` é imutável após merge (story-0059-0011)

## 4. Regras de Negócio Transversais (Source of Truth)

> Regras que se aplicam a múltiplas histórias. Cada história referencia as regras pelo ID. Alterações de regra propagam automaticamente para todas as histórias dependentes.

| ID | Título | Descrição |
| :--- | :--- | :--- |
| **[RULE-059-01]** | Dogfooding obrigatório (EPIC-0059 via `/x-story-implement`) | Toda story do EPIC-0059 deve ela mesma ser implementada via `/x-story-implement` (não é permitido bypass); falha auto-imposta se Phase 1 wave estiver ausente (6 artefatos obrigatórios em `plans/epic-0059/plans/`). Prova: telemetria em `events.ndjson` exigida pelo audit de merge. |
| **[RULE-059-02]** | Aceitação: prova que o gate dispara | Cada story produz ≥ 1 acceptance test (test smoke obrigatório) que tenta reproduzir o bypass coberto e valida que o gate dispara com erro não-zero. Testa o cenário "operador pula o orquestrador" e verifica que audit rejeita. |
| **[RULE-059-03]** | Proibição de auto-isenção | Nenhuma story pode adicionar `<!-- audit-exempt -->` ou `<!-- baseline-correction -->` no próprio artefato (Rule 22 self-check rejeita). Isenção exigida deve vir de story predecessor ou de humano revisor em PR. |
| **[RULE-059-04]** | Baseline é crítico (CODEOWNERS) | Mudanças em `audits/*-baseline.txt` exigem review humana; arquivo é adicionado a `.github/CODEOWNERS`. Story-0059-0011 configura imutability check: qualquer adição posterior ao merge falha o build. |
| **[RULE-059-05]** | Rule 26 é carregada em toda conversa | Após merge, `Rule 26` é loaded em sistema prompt de TODA conversa; `.claude/rules/` README reordenado numericamente. Story-0059-0010 adiciona a rule e atualiza índice. |
| **[RULE-059-06]** | Padronização de exit codes | Audit scripts novos (e refatorados) seguem contrato uniforme: `0=OK`, `1=violation`, `2=corrupt baseline`, `3=invalid exemption`, `4=enforcement broken`. Story-0059-0001 estende existente com novo scope; stories 0007, 0008 adicionam audits novos. |
| **[RULE-059-07]** | Env var policy: sem escape por variável | Hooks novos respeitam `CLAUDE_TELEMETRY_DISABLED=1` (desativa telemetry) mas **NÃO** respeitam outros bypass envs (ex: `CLAUDE_SKIP_AUDIT=1`). Intencional — o enforcement é **universal**, sem escape hatch baseado em env var. Mesmo operador local só consegue bypass via `--no-verify` (bloqueado por pre-commit hook). |
