## 1. Visão Geral

**Chave Jira:** —

Formalizar o lifecycle dos gates de auditoria do repositório `ia-dev-environment` e promover os scripts de auditoria a artefato gerado pelo `ia-dev-env`. Três entregas integradas: (a) nova Rule 25 "Audit Gate Lifecycle" + catálogo canônico indexando os 8+ gates existentes; (b) criação física dos 3 scripts citados em Rules mas ausentes (`audit-flow-version.sh`, `audit-epic-branches.sh`, `audit-skill-visibility.sh`); (c) novo `ScriptsAssembler` com source-of-truth em `java/src/main/resources/targets/claude/scripts/` + CI workflow template, fazendo com que todo projeto gerado herde os gates de governance que hoje só existem no próprio gerador. Escopo exclui reformulação de Rules existentes, refatoração de `HooksAssembler` e introdução de novos linters Java.

### Global Definition of Ready (DoR)

- Spec `spec-audit-scripts-lifecycle.md` aprovado e versionado.
- Rule 14 (Project Scope Guard) confirmado como aplicável — scripts bash e um assembler Java adicional estão dentro do escopo.
- Layout atual dos 22 assemblers em `AssemblerFactory.java` conhecido (documentado em `CLAUDE.md`).
- Lista dos 5 scripts alvo confirmada (3 criar, 2 mover): `audit-flow-version.sh`, `audit-epic-branches.sh`, `audit-skill-visibility.sh`, `audit-model-selection.sh`, `audit-execution-integrity.sh`.

### Global Definition of Done (DoD)

- **Cobertura:** ≥ 95% Line / ≥ 90% Branch em `ScriptsAssembler` e classes relacionadas (absolute gate, Rule 05).
- **Testes Automatizados:** cada script novo tem fixture + invocação `--self-check` verde; `ScriptsAssembler` tem teste unitário + teste golden; workflow CI template tem smoke via `actionlint` ou equivalente. Cada história DEVE ter pelo menos 1 teste automatizado validando o critério de aceite principal.
- **Smoke Tests:** obrigatório para os 5 scripts (`bash -n` + `shellcheck` + `--self-check` exit 0).
- **Relatório de Cobertura:** JaCoCo XML anexado a cada PR; métricas filtradas aos arquivos tocados.
- **Documentação:** `docs/audit-gates-catalog.md` atualizado com cada novo gate; `CLAUDE.md` referenciando o catálogo; `CHANGELOG.md` com entrada por história; Rule 25 publicada; ADR correspondente em `adr/ADR-NNNN-*.md`.
- **Persistência:** N/A (epic não introduz persistência).
- **Performance:** scripts de auditoria executam em ≤ 30s cada em CI runner `ubuntu-latest` padrão; `ScriptsAssembler` acrescenta ≤ 500ms ao tempo total de `mvn process-resources`.
- **TDD Compliance:** commits mostram test-first pattern; refactoring explícito após green; testes incrementais via TPP.
- **Double-Loop TDD:** acceptance tests derivados dos cenários Gherkin (outer loop); unit tests via TPP (inner loop).

## 4. Regras de Negócio Transversais (Source of Truth)

| ID | Título | Descrição |
| :--- | :--- | :--- |
| **[RULE-001]** | Audit Gate Taxonomy (nova Rule 25) | Todo gate de governance no repo pertence a uma das 4 camadas: <br>(a) **Hook runtime** — roda em eventos Claude Code (`Stop`, `PreToolUse` etc.), localizado em `.claude/hooks/`, gerado por `HooksAssembler`. <br>(b) **CI script** — roda em PR via GitHub Actions, localizado em `scripts/audit-*.sh`, gerado por `ScriptsAssembler` (novo). <br>(c) **Java test** — roda em `mvn verify`, classes em `java/src/test/java/**/*AuditTest.java` ou `**/*Lint.java`. <br>(d) **CI workflow** — YAML em `.github/workflows/`, orquestra os três anteriores. Matriz de decisão: escolher Hook quando gate precisa rodar durante turno do LLM; CI script quando gate é stateless sobre artefatos do repo; Java test quando gate depende de parsing estruturado de código. |
| **[RULE-002]** | Audit Script Naming & Exit Codes | Scripts CI DEVEM seguir `audit-{subject}.sh` (kebab-case, prefixo `audit-` obrigatório). Exit code 0 = sucesso; 1 = violation detectada (com código nomeado impresso em stderr, ex.: `FLOW_VERSION_VIOLATION`, `SKILL_VISIBILITY_VIOLATION`); 2 = erro operacional (dependência ausente, arquivo ilegível); 3 = baseline/exempção corrupta. Flag `--self-check` MANDATÓRIO (exit 0 se script está estruturalmente válido; exit 2 caso contrário). |
| **[RULE-003]** | Generation Parity (Rule 14 aware) | Scripts de auditoria SÃO artefato gerado pelo `ia-dev-env`. Source-of-truth: `java/src/main/resources/targets/claude/scripts/`. Output gerado: `.claude/scripts/*.sh` (com bit executável). Golden files do diretório inteiro sob `src/test/resources/golden/{profile}/.claude/scripts/`. Raiz do próprio `ia-dev-environment` mantém cópia em `/scripts/` para as Rules 19/21/22/23/24 continuarem resolvendo via path — mas a cópia é consequência do assembler, não fonte. |
| **[RULE-004]** | Catalog-before-Add | Nenhum novo gate DEVE ser introduzido em nenhuma Rule sem ser adicionado simultaneamente a `docs/audit-gates-catalog.md`. PR que introduz Rule mencionando gate não catalogado falha em `scripts/audit-skill-visibility.sh` (ampliado para também checar referências órfãs de scripts — escopo de story 0058-0005). |
| **[RULE-005]** | Backward Compatibility | Os dois scripts existentes (`audit-model-selection.sh`, `audit-execution-integrity.sh`) NÃO podem mudar de path durante a migração — as Rules 23 e 24 referenciam `scripts/audit-*.sh` e o baseline `audits/execution-integrity-baseline.txt` assume esse path. Durante Story 0058-0006, mover para source-of-truth e gerar de volta para `/scripts/` mantém o contrato. |
| **[RULE-006]** | CI Fail-Fast | Workflow `.github/workflows/audit.yml` DEVE falhar na primeira violação encontrada (job-level `fail-fast: true`) para maximizar sinal. Self-check suite roda antes da matriz principal — script estruturalmente quebrado aborta sem correr os demais. |
