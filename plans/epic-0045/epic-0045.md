# Épico: CI Watch no Fluxo de PR

**Autor:** Eder Celeste Nunes Junior
**Data:** 2026-04-16
**Versão:** 1.0
**Status:** Em Refinamento
**Épico ID:** EPIC-0045

## 1. Visão Geral

**Chave Jira:** —

Skills orquestradoras (`x-story-implement`, `x-task-implement --worktree`, `x-release`) hoje seguem de `x-pr-create` (que retorna `prNumber`/`prUrl`) diretamente para o APPROVAL GATE (EPIC-0042) e fazem auto-merge, sem aguardar (a) os checks de CI remotos terminarem via `gh pr checks` / `statusCheckRollup` nem (b) o review automático postado pelo `copilot-pull-request-reviewer[bot]`. Consequência observada: quando o menu `PROCEED / FIX-PR / ABORT` do EPIC-0043 oferece `FIX-PR`, a skill `x-pr-fix` é invocada, consulta `gh api .../pulls/{N}/comments` e encontra zero comentários, porque o Copilot ainda não postou (latência típica 30–180s). O objetivo primário do menu interativo — aproveitar feedback automatizado antes do merge — fica inviabilizado.

Este épico entrega: (1) a skill dedicada `x-pr-watch-ci` em `core/pr/` que encapsula polling de `gh pr checks` + detecção de review do Copilot com exit codes estáveis (`SUCCESS` / `CI_PENDING_PROCEED` / `CI_FAILED` / `TIMEOUT` / `PR_ALREADY_MERGED` / `NO_CI_CONFIGURED` / `PR_CLOSED` / `PR_NOT_FOUND`), (2) a Rule 20 formalizando CI-Watch como default em schema v2 (opt-out via `--no-ci-watch`) com no-op em schema v1 (Rule 19), (3) retrofits em `x-story-implement` Phase 2.2 (novo passo entre PR_CREATED e APPROVAL GATE), `x-task-implement --worktree` e `x-release` (flag `--ci-watch`), e (4) golden diff regen + smoke test real contra PR do próprio repo.

Fora de escopo: auto-classificação de comentários (responsabilidade de `x-pr-fix`), auto-merge (responsabilidade do EPIC-0042), retrofits em skills que não abrem PR, dashboards de métricas, integração nativa com providers não-gh (GitLab/Jenkins). Precondição dura: STORY-0045-0003 só inicia após merge do EPIC-0043 em `develop` para evitar conflito determinístico em `x-story-implement/SKILL.md:797-810`.

## 4. Regras de Negócio Transversais (Source of Truth)

> Regras que se aplicam a múltiplas histórias. Cada história referencia as regras pelo ID. Alterações de regra propagam automaticamente para todas as histórias dependentes.

| ID | Título | Descrição |
| :--- | :--- | :--- |
| **[RULE-045-01]** | CI-Watch default em schema v2 | Em epics com `planningSchemaVersion == "2.0"`, a invocação de `x-pr-watch-ci` entre `x-pr-create` e APPROVAL GATE é obrigatória em todos os orquestradores que abrem PR (`x-story-implement`, `x-task-implement --worktree`, `x-release` via flag). Opt-out apenas via flag `--no-ci-watch` (uso em CI/automação). O orquestrador propaga o exit code + JSON final para o prompt do menu `PROCEED/FIX-PR/ABORT` do EPIC-0043. |
| **[RULE-045-02]** | No-op em schema v1 (Rule 19) | Em epics com `planningSchemaVersion == "1.0"` ou ausente, `x-pr-watch-ci` NÃO é invocada. Rule 19 (Backward Compatibility) é respeitada integralmente. Resolver de schema (`SchemaVersionResolver`) decide o fallback conforme a matriz documentada na Rule 19. |
| **[RULE-045-03]** | State-file canônico versionado | Localização: `.claude/state/pr-watch-<PR_NUMBER>.json`. Shape: `{ prNumber, startedAt, lastPollAt, pollCount, checksSnapshot[], copilotReview?, schemaVersion: "1.0" }`. Escrita atômica via `.tmp` + rename. Campo `schemaVersion` é obrigatório; leitura rejeita shapes desconhecidos. Reuso do padrão EPIC-0035 story 0035-0004. |
| **[RULE-045-04]** | Copilot login canônico | Reviewer Copilot é identificado pelo login exato `copilot-pull-request-reviewer[bot]` em `gh api repos/{owner}/{repo}/pulls/{N}/reviews`. Alias humano ou outro bot-name não é reconhecido. Detecção de Copilot desativado no repo: ausência do reviewer em `gh api .../pulls/{N}/requested_reviewers` após 1º poll → downgrade imediato para `CI_PENDING_PROCEED` (exit 10). |
| **[RULE-045-05]** | Exit codes estáveis como contrato público | Códigos `0` (SUCCESS), `10` (CI_PENDING_PROCEED), `20` (CI_FAILED), `30` (TIMEOUT), `40` (PR_ALREADY_MERGED), `50` (NO_CI_CONFIGURED), `60` (PR_CLOSED), `70` (PR_NOT_FOUND) formam contrato público. Mudança de semântica exige MAJOR bump (Rule 08); adição de novo código é MINOR. Orquestradores não podem assumir códigos fora desta tabela. |
| **[RULE-045-06]** | Rule 13 INLINE-SKILL obrigatória | Todos os orquestradores invocam `x-pr-watch-ci` via `Skill(skill: "x-pr-watch-ci", args: "...")` — Rule 13 Pattern 1. Bare-slash em delegação proibido (Rule 13 §Forbidden). Frontmatter `allowed-tools` da `x-pr-watch-ci` inclui `Bash` (para `gh` CLI); frontmatter dos orquestradores retrofitados inclui `Skill`. |
| **[RULE-045-07]** | Menu do EPIC-0043 consome exit code | O orquestrador passa o exit code + JSON final do `x-pr-watch-ci` como contexto para o prompt do menu `PROCEED/FIX-PR/ABORT`. O `description` do `AskUserQuestion` exibe status dos checks (N passed / M failed / K pending) + presença do Copilot review (`present: true/false`). Exit 20 (`CI_FAILED`) força o menu a sugerir `FIX-PR` como opção recomendada; exit 0 (`SUCCESS`) sugere `PROCEED`. Exit 10 (`CI_PENDING_PROCEED`) sinaliza que Copilot não postou mas CI está verde. |
| **[RULE-045-08]** | Atomic, Reversible Commits | Cada story produz pelo menos um commit atômico por task, reversível via `git revert <sha>` sem tocar tasks irmãs (Rule 18). Mensagens seguem Conventional Commits (Rule 08): `feat(task-0045-YYYY-NNN):` para skill nova e retrofits, `docs(task-0045-YYYY-NNN):` para rule/spec/epic, `test(task-0045-YYYY-NNN):` para goldens e smoke. |
