# Épico: Error Resilience & Recovery

**Autor:** Arquiteto de Software
**Data:** 2026-04-08
**Versão:** 1.0
**Status:** Em Refinamento

## 1. Visão Geral

**Chave Jira:** —

Implementar tratamento robusto de erros internos, transientes e sistêmicos durante a execução de tasks, stories e epics. Auditoria dos skills de execução revelou ZERO tratamento para erros transientes (Claude overloaded, GitHub API rate limit, tool timeout), erros de contexto (context window exceeded), ou falhas sistêmicas. Este épico introduz retry com exponential backoff, recovery de subagents com contexto reduzido, circuit breaker para pausar em falhas sistêmicas, degradação graceful sob pressão de contexto, catálogo padronizado de erros, integrity gate obrigatório entre fases, e histórico de erros no checkpoint. O target é recuperação automática de ≥ 80% dos erros transientes e detecção de falhas sistêmicas em ≤ 3 falhas consecutivas.

## 4. Regras de Negócio Transversais (Source of Truth)

| ID | Título | Descrição |
| :--- | :--- | :--- |
| **[RULE-001]** | Retry para Transientes | Erros transientes (Claude overloaded, GitHub 5xx/429, tool timeout) DEVEM ser retried com exponential backoff (2s, 4s, 8s) com máximo de 3 tentativas antes de marcar como falha permanente. |
| **[RULE-002]** | Fail-Fast para Permanentes | Erros permanentes (file not found, invalid ID, compilation error) DEVEM falhar imediatamente com mensagem contextual. NUNCA retry erros permanentes. |
| **[RULE-003]** | Circuit Breaker | O circuit breaker DEVE abrir após 3 falhas consecutivas na mesma fase, pausando a execução e escalando para o usuário via AskUserQuestion. DEVE resetar ao retomar com `--resume`. |
| **[RULE-004]** | Integrity Gate Obrigatório | O integrity gate entre fases DEVE executar SEMPRE, mesmo quando `mergeMode == "no-merge"`. No modo no-merge, o gate executa com merge local temporário. |
| **[RULE-005]** | Registro de Erros | Todo erro DEVE ser registrado no `execution-state.json` com: timestamp, error code, story/task ID, phase, retry count e resolução final. |
| **[RULE-006]** | Degradação Progressiva | Graceful degradation é progressiva: Level 1 (reduzir verbosidade) → Level 2 (forçar delegação) → Level 3 (salvar estado e sugerir --resume). NUNCA pular diretamente para Level 3. |
| **[RULE-007]** | Error Catalog como Reference | O catálogo de erros DEVE ser um reference file do orquestrador, NÃO um knowledge pack separado. |
