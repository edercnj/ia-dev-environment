# Épico: Remoção de Código Deprecated (`forRemoval = true`)

**Autor:** Eder Celeste Nunes Junior
**Data:** 2026-04-16
**Versão:** 1.0
**Status:** Concluído
**Épico ID:** EPIC-0044

## 1. Visão Geral

**Chave Jira:** —

O repositório `ia-dev-environment` contém 6 símbolos Java marcados com `@Deprecated(forRemoval = true)` espalhados por 2 arquivos de produção (`StackMapping.java` e `ResourceResolver.java`). Todos possuem substitutos já implementados e testados (`DatabaseSettingsMapping` e `resolveResourceDir(String)`), mas continuam consumidos por callers legítimos — 1 arquivo de produção para `StackMapping` (`PermissionCollector.java`) e 23 assemblers para `ResourceResolver`, além de 29+ arquivos de teste. Este épico orquestra a migração dos callers remanescentes e a remoção física dos símbolos deprecated sem introduzir regressão funcional, sem quebra de golden files e sem bump de versão MAJOR (mudança interna, API pública preservada).

Entregas principais: (1) migração mecânica de 24 arquivos de produção para as APIs substitutas; (2) atualização ou eliminação de 29+ referências em testes; (3) remoção física dos 6 símbolos deprecated; (4) eliminação dos 40+ warnings `[removal]` do `mvn compile`; (5) golden files regenerados quando o output determinístico dos assemblers for afetado.

Fora de escopo: deprecated em recursos YAML/markdown (não há anotação equivalente), deprecated em outros idiomas (nenhum encontrado), introdução de novas APIs substitutas (os substitutos já existem e estão em uso), integração com Jira (pode ser feita posteriormente via `x-jira-create-epic`), e renomear/alterar a assinatura pública de `DatabaseSettingsMapping`.

## 3. Definições de Qualidade Globais

### Global Definition of Ready (DoR)

- Inventário completo de símbolos deprecated consolidado na spec com caminho, linha, callers em produção e em testes.
- Substituto confirmado existente e com cobertura de teste verde para cada símbolo removido.
- Branch de trabalho criado a partir de `develop` conforme Rule 09, dentro de worktree dedicado conforme Rule 14.
- Comando local `mvn verify` executado com build verde imediatamente antes do início da história (baseline).

### Global Definition of Done (DoD)

- **Cobertura:** ≥ 95% Line, ≥ 90% Branch (Rule 05). Mantida ou superior à baseline pré-história.
- **Testes Automatizados:** `mvn verify` verde. Suites atingidas (ex: `StackMappingTest`, `ResourceResolverTest`, `*AssemblerTest`) permanecem verdes. Cada história DEVE ter pelo menos 1 teste automatizado validando a remoção (inexistência do símbolo + substituto cobrindo o caso equivalente).
- **Smoke Tests:** Obrigatório — build Maven completo sem warnings `[removal]` no escopo da história.
- **Relatório de Cobertura:** Jacoco XML gerado em `java/target/site/jacoco/jacoco.xml`; delta de cobertura ≥ 0 em relação à baseline.
- **Documentação:** `CHANGELOG.md` atualizado em seção `Removed` (Keep a Changelog). Javadoc dos arquivos modificados sem referências a símbolos removidos.
- **Persistência:** N/A — nenhuma alteração de schema ou dado persistido.
- **Performance:** N/A — refactor sem impacto em latência/throughput.
- **TDD Compliance:** Commits show test-first pattern. Cada remoção é precedida de commit de migração de callers (Green) seguido de commit de remoção (Refactor). Tests are incremental (TPP — degenerate → happy → error → boundary).
- **Double-Loop TDD:** Acceptance tests derived from Gherkin scenarios (outer loop). Unit tests guided by TPP (inner loop).

## 4. Regras de Negócio Transversais (Source of Truth)

> Regras que se aplicam a múltiplas histórias. Cada história referencia as regras pelo ID. Alterações de regra propagam automaticamente para todas as histórias dependentes.

| ID | Título | Descrição |
| :--- | :--- | :--- |
| **[RULE-001]** | Migração antes de remoção | Nenhum símbolo `@Deprecated(forRemoval=true)` pode ser removido antes que 100% dos callers (produção + teste) tenham migrado para o substituto.<br>Sequência obrigatória por história: (1) commit "migrate callers"; (2) `mvn verify` verde; (3) commit "remove deprecated"; (4) `mvn verify` verde novamente.<br>Violação dessa ordem gera commit que não compila e inutiliza `git bisect`. |
| **[RULE-002]** | Zero warnings `[removal]` pós-história | Ao final de cada história, `mvn compile` no escopo alterado DEVE emitir zero warnings da categoria `[removal]` referentes aos símbolos removidos pela história.<br>Warnings remanescentes referentes a outros símbolos deprecated de outras histórias são aceitáveis até que suas respectivas histórias sejam concluídas. |
| **[RULE-003]** | Cobertura preservada | Cobertura de linha ≥ 95% e de branch ≥ 90% (Rule 05) DEVE ser mantida ou melhorada em relação à baseline pré-história.<br>Se a remoção derrubar cobertura (ex: teste que exercitava apenas o símbolo deprecated), adicionar teste equivalente cobrindo o substituto antes de remover. |
| **[RULE-004]** | Golden files consistentes | Quando a migração alterar output determinístico de um assembler, golden files DEVEM ser regenerados via `GoldenFileRegenerator`, precedido de `mvn process-resources` (memória do projeto: executar `mvn process-resources` antes da regeneração evita output stale).<br>Diff de golden file DEVE ser revisado manualmente no PR — diff vazio indica que a migração é pura refactor; diff não-vazio exige justificativa. |
| **[RULE-005]** | Conventional Commits `refactor:` | Commits de remoção de deprecated usam prefixo `refactor:` com escopo `story(0044-NNNN)` (Rule 08).<br>Commits de migração de callers usam `refactor:` ou `chore:` conforme natureza.<br>Nenhum bump de versão MINOR/MAJOR — a API pública é preservada via substitutos já existentes. |
| **[RULE-006]** | Git Flow + Worktree | Cada história abre branch `feat/story-0044-NNNN-<slug>` a partir de `develop` (Rule 09), dentro de worktree dedicado em `.claude/worktrees/story-0044-NNNN/` (Rule 14 — prefixos `feat/`/`feature/` são ambos aceitos, mas o tooling `x-git-push`/`x-pr-create` padroniza em `feat/`).<br>PR targeta `develop` com squash merge após aprovação.<br>Branch removido após merge; worktree removido pelo criador. |
| **[RULE-007]** | Teste-primeiro na migração | Ao migrar um caller, a alteração do caller e a alteração do teste correspondente ocorrem no mesmo commit (ou o teste precede o caller). Nunca `@SuppressWarnings("removal")` em código de produção — ele esconde o problema em vez de resolvê-lo.<br>`@SuppressWarnings("removal")` só é aceitável em testes legados que validam o comportamento do símbolo deprecated antes de ele ser removido, e o próprio teste DEVE ser removido ao final da história. |
| **[RULE-008]** | CHANGELOG obrigatório | Toda história atualiza `CHANGELOG.md` na seção `Removed` (Keep a Changelog), listando os símbolos removidos pela história com o caminho do arquivo de origem e o substituto.<br>Exemplo: `- Removed StackMapping.DATABASE_SETTINGS_MAP — use DatabaseSettingsMapping.DATABASE_SETTINGS_MAP`. |
