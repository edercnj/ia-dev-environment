# Épico: Lifecycle Integrity — Propagação de Status e Commits Atômicos de Reports

**Autor:** Eder Celeste Nunes Junior
**Data:** 2026-04-16
**Versão:** 1.0
**Status:** Em Refinamento
**Épico ID:** EPIC-0046

## 1. Visão Geral

**Chave Jira:** —

Duas falhas de integridade no ciclo de vida SDD coexistem hoje no `ia-dev-env`: (GAP-1) skills de planejamento e execução NÃO atualizam o campo `**Status:**` dos artefatos markdown (épicos, stories, tasks) nem a coluna Status do `implementation-map`. Evidência: `EPIC-0024` foi concluído com 16 stories `SUCCESS`+`MERGED` em `execution-state.json`, mas todos os 16 markdowns permanecem `**Status:** Pendente` e o epic segue `**Status:** Em Refinamento` (0% de sync markdown↔checkpoint). (GAP-2) `x-epic-implement` grava `execution-plan-epic-*.md` e `phase-report-epic-*.md` em `plans/epic-XXXX/reports/` sem invocar `x-git-commit` — esses arquivos ficam órfãos e a precondição `VALIDATE_DIRTY_WORKDIR` de `x-release` (SKILL.md:277-308) aborta o release train.

Este épico entrega: (1) nova `Rule 21 — lifecycle-integrity` formalizando propagação de status e commit atômico de reports; (2) matriz de transição `Pendente → Planejada → Em Andamento → Concluída | Falha | Bloqueada` documentada no header dos templates `_TEMPLATE-TASK.md`, `_TEMPLATE-STORY.md`, `_TEMPLATE-EPIC.md`; (3) retrofit de 10 SKILL.md (7 de planejamento, 3 de execução) para propagar status e commitar reports; (4) nova skill `x-status-reconcile` em `core/ops/` (opt-in) para diagnosticar e backfillar épicos legados sem violar Rule 19; (5) helpers Java em `dev.iadev.application.lifecycle` (`StatusFieldParser`, `LifecycleTransitionMatrix`, `LifecycleAuditRunner`); (6) `LifecycleIntegrityAuditTest` CI-blocking que detecta regressões (phases órfãs, escritas em `reports/` sem commit subsequente, flags `--skip-*` no happy path). Status sync é V2-gated via `SchemaVersionResolver` (Rule 19) — épicos v1 não ganham nem perdem comportamento.

Fora de escopo: backfill automático de épicos legados (acionável via `x-status-reconcile --apply` manual); migração de schema de `execution-state.json`; integração Jira de status sync (épico futuro); dashboards de lifecycle. Precondição dura: slot Rule 21 disponível (slot 20 consumido por EPIC-0045 ci-watch); Rule 19 preservada integralmente.

## 4. Regras de Negócio Transversais (Source of Truth)

> Regras que se aplicam a múltiplas histórias. Cada história referencia as regras pelo ID. Alterações de regra propagam automaticamente para todas as histórias dependentes.

| ID | Título | Descrição |
| :--- | :--- | :--- |
| **[RULE-046-01]** | Source-of-truth invariant | Toda alteração de skill, rule, template ou golden DEVE ser feita em `java/src/main/resources/targets/claude/`.<br>`.claude/**` e `src/test/resources/golden/**` são saídas de `mvn process-resources` — edição manual proibida. |
| **[RULE-046-02]** | Planning updates status | Toda skill de planejamento que produza um plan artefact DEVE atualizar `**Status:** Pendente → Planejada` do artefato-fonte no MESMO commit do plano.<br>Aplica-se a: `x-story-plan` (story), `x-task-plan` (task), `x-arch-plan`/`x-test-plan` (story quando associado), `x-epic-create` (epic), `x-epic-decompose` (epic + stories), `x-epic-map` (coluna Planejamento do implementation-map).<br>Rationale: o plan é evidência objetiva de que o DoR foi alcançado — o campo `**Status:**` DEVE refletir isso. |
| **[RULE-046-03]** | Implementation updates status | Toda skill de execução que termine uma etapa DEVE transicionar `**Status:** → Concluída` e atualizar a coluna Status do `implementation-map` (ou `task-implementation-map`) no MESMO commit atômico final:<br>- `x-task-implement` → task (atômico por Rule 18).<br>- `x-story-implement` Phase 3 (promovida a obrigatória) → story + row do map.<br>- `x-epic-implement` Phase 1.7 (promovida de 1.6b e cabeada ao Core Loop) → story no loop + epic ao final. |
| **[RULE-046-04]** | Status transition is non-skippable | Fases que executam status sync NÃO podem ser puladas por flags opt-out no caminho happy (`--skip-verification`, `--skip-status-sync`, etc.).<br>Flag `--skip-status-sync` pode existir APENAS para recovery manual, documentada explicitamente no SKILL.md. `LifecycleIntegrityAuditTest` rejeita uso em caminho happy (detecção: flag referenciada no Core Loop). |
| **[RULE-046-05]** | Reports are atomically committed | Toda skill que grave arquivos em `plans/epic-XXXX/reports/` DEVE invocar `Skill(skill: "x-git-commit", args: "...")` (Rule 13 Pattern 1 INLINE-SKILL) antes de retornar.<br>Reports não commitados = falha da fase que os produziu. Aplica-se sobretudo a `x-epic-implement` (`execution-plan-epic-*.md` linha 264, `phase-report-epic-*.md` linha 706). |
| **[RULE-046-06]** | Clean workdir invariant | Nenhuma skill pode retornar com `git status --porcelain` não-vazio após sua última fase, salvo quando a skill declara explicitamente "dirty on exit by design" no frontmatter (nenhuma skill hoje).<br>Enforcement: smoke test por skill retrofitada + `LifecycleIntegrityAuditTest` detecta ausência de commit após write em `reports/`. |
| **[RULE-046-07]** | Markdown é SoT; state.json é telemetria | `execution-state.json` permanece append-only como telemetria de checkpoint.<br>Source of Truth do status do artefato é o campo `**Status:**` no markdown + coluna Status do `implementation-map`.<br>Reconciliação é opt-in via `x-status-reconcile`, nunca implícita. Rule 19 preservada: épicos legados não recebem upgrade silencioso. |
| **[RULE-046-08]** | Fail loud on status update failure | Se a atualização de status falhar (arquivo ausente, regex não casa, commit rejeitado), a skill aborta com exit ≠ 0 e código `STATUS_SYNC_FAILED`.<br>Nunca swallow. Logs estruturados em stderr incluem: artefato + linha + valor esperado + valor encontrado. Rationale: silêncio é o modo de falha atual — o fix precisa ser auto-detectável. |
