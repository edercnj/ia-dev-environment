## 3. Descricao

Como **desenvolvedor do ia-dev-environment**, eu quero templates de Architecture Decision Records (ADR) para decisoes comuns de banco de dados, para que equipes tenham um ponto de partida estruturado para documentar escolhas como SQL vs NoSQL, estrategia de sharding, ou topologia de read replicas.

### 3.1 Contexto

O arquivo pertence ao knowledge pack `data-modeling` criado na story-0023-0001. Fornece templates de ADR seguindo o formato existente utilizado pelo skill `x-dev-adr-automation`. Cada template inclui as secoes: Context, Decision Drivers, Considered Options, Decision Outcome e Consequences.

### 3.2 Arquivo a Criar

- `targets/claude/skills/knowledge-packs/data-modeling/references/database-adr-templates.md`

### 3.3 Templates a Incluir

| # | Template ADR | Topico Central |
| :--- | :--- | :--- |
| 1 | SQL vs NoSQL database selection | Criterios de escolha entre paradigmas relacional e nao-relacional |
| 2 | Embedded vs Referenced data model (document DBs) | Trade-offs de embedding vs referencing em bancos de documentos |
| 3 | Partitioning/sharding strategy selection | Opcoes de Range, Hash, List e Composite partitioning |
| 4 | Caching layer selection and topology | Estrategias de cache (local, distributed, multi-tier) |
| 5 | Read replica topology (sync vs async, regional) | Trade-offs de replicacao sincrona vs assincrona e distribuicao regional |
| 6 | Distributed transaction strategy (2PC vs Saga vs Outbox) | Comparacao de padroes de transacao distribuida |

### 3.4 Formato de Cada Template

Cada template ADR segue a estrutura:

```
## ADR: [Titulo]

### Context
[Descricao do contexto e problema a ser resolvido]

### Decision Drivers
[Lista de fatores que influenciam a decisao]

### Considered Options
[Opcoes analisadas com pros e contras]

### Decision Outcome
[Template para preencher com a decisao tomada]

### Consequences
[Impactos positivos e negativos da decisao]
```

### 3.5 Restricoes

- Arquivo DEVE ter no maximo 300 linhas (RULE-001)
- Conteudo NAO deve duplicar o que existe no KP `data-management` (`migration-safety-checklist.md`) (RULE-011)
- Templates DEVEM ser genericos o suficiente para aplicar a qualquer linguagem/framework

- **Valor Principal:** Equipes tem templates prontos e estruturados para documentar decisoes arquiteturais de banco de dados
