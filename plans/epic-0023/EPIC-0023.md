**Autor:** Claude (AI Assistant)
**Data:** 2026-04-05
**Versao:** 1.0
**Status:** Em Refinamento

## 1. Visao Geral

**Chave Jira:** --

A base de conhecimento de banco de dados do ia-dev-environment cobre atualmente 5 bancos (PostgreSQL, MySQL, Oracle, MongoDB, Cassandra) e 3 caches (Redis, Dragonfly, Memcached), organizados em 2 categorias (SQL, NoSQL). Este epico expande a cobertura para 17 bancos em 7 categorias (SQL, NoSQL, Graph, Columnar/OLAP, NewSQL/Distributed, Time-Series, Search), cria um novo knowledge pack `data-modeling` com padroes cross-cutting de modelagem de dados, e enriquece o workflow de review com checklist expandido e novo especialista de Data Modeling.

**Escopo incluido:**

- Novo knowledge pack `data-modeling` (schema design patterns, concurrency, test data, ADR templates)
- Infraestrutura Java para 5 novas categorias de banco (graph, columnar, newsql, timeseries, search)
- Knowledge files para 12 novos bancos (Neo4j, Neptune, ClickHouse, Druid, YugaByteDB, CockroachDB, TiDB, InfluxDB, TimescaleDB, Elasticsearch, OpenSearch, EventStoreDB)
- Expansao do checklist Database Engineer no x-review de 8 para 20 items
- Novo especialista Data Modeling no x-review para projetos DDD/hexagonal/CQRS
- 4 novos config profiles para exercitar as novas categorias
- Testes de integracao e smoke tests para validacao end-to-end

**Escopo excluido:**

- Alteracao de knowledge files existentes (PostgreSQL, MySQL, Oracle, MongoDB, Cassandra, caches)
- Mudancas no output gerado para profiles existentes (backward compatibility total)
- Novos skills executaveis (apenas knowledge packs)
- Alteracao de assemblers existentes alem dos pontos de extensao

## 3. Definicoes de Qualidade Globais

### Global Definition of Ready (DoR)

- Plano de implementacao revisado e aprovado
- Estrutura de knowledge files existente compreendida
- RulesConditionals, StackMapping e KnowledgePackSelection analisados
- Testes existentes executando com sucesso antes de qualquer mudanca

### Global Definition of Done (DoD)

- Output gerado identico ao pre-implementacao para todos os 8 perfis existentes
- **Cobertura:** >= 95% Line, >= 90% Branch
- **TDD Compliance:** Commits mostram padrao test-first (teste precede implementacao no git log), refactoring explicito apos green, testes incrementais via TPP
- **Double-Loop TDD:** Acceptance tests derivados dos cenarios Gherkin (outer loop), unit tests guiados por Transformation Priority Premise (inner loop)
- **Rastreabilidade @GK-N bidirecional:** Todo @GK-N mapeia para >= 1 AT-N, todo AT-N referencia um @GK-N valido
- **Commits atomicos por ciclo TDD:** Cada ciclo Red-Green-Refactor produz um ou mais commits atomicos com formato Conventional Commits
- **Test plan antes de implementacao:** Nenhuma historia inicia implementacao sem test plan gerado via `/x-test-plan`
- Novos knowledge files <= 300 linhas cada (RULE-001)
- Novos bancos registrados em version-matrix.md (RULE-006)
- Zero regressao: funcionalidades existentes continuam funcionais

## 4. Regras de Negocio Transversais


| ID       | Titulo                                   | Descricao                                                                                                                                                                                                                                          |
| -------- | ---------------------------------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| RULE-001 | Budget de Tamanho de Arquivo             | Cada arquivo `.md` de knowledge DEVE ter no maximo 300 linhas (~8KB). Se exceder, dividir em SKILL.md + references/. Garante eficiencia de context window da IA.                                                                                   |
| RULE-002 | Padrao de 3 Arquivos por Banco           | Cada banco especifico DEVE conter exatamente 3 arquivos seguindo a convencao: `types-and-conventions.md` (SQL/NewSQL) ou `modeling-patterns.md` (NoSQL/Graph/Columnar/TimeSeries/Search), `migration-patterns.md`, `query-optimization.md`.        |
| RULE-003 | Arquivo de Principios por Categoria      | Cada categoria de banco (graph, columnar, newsql, timeseries, search) DEVE ter um `{category}-principles.md` no subdirectorio `common/`, analogo aos existentes `sql-principles.md` e `nosql-principles.md`.                                       |
| RULE-004 | Fidelidade de Template Variables         | Todos os knowledge files DEVEM usar template variables existentes (`{{DB_TYPE}}`, `{{DB_MIGRATION}}`, `{{CACHE_TYPE}}`). Nenhuma nova template variable sera criada neste epico.                                                                   |
| RULE-005 | Backward Compatibility                   | Todos os knowledge files e outputs gerados existentes permanecem INALTERADOS. Golden files dos 8 perfis de teste DEVEM passar sem modificacao.                                                                                                     |
| RULE-006 | Atualizacao do Version Matrix            | Todo novo tipo de banco DEVE ser adicionado ao `version-matrix.md` com a mesma estrutura de colunas das entradas existentes.                                                                                                                       |
| RULE-007 | Settings File por Banco com CLI          | Todo novo banco que possui ferramentas CLI DEVE ter um `database-{key}.json` correspondente em `targets/claude/settings/`.                                                                                                                         |
| RULE-008 | StackMapping como Single Source of Truth | `StackMapping.DATABASE_SETTINGS_MAP` e a unica fonte de verdade para nomes de bancos suportados. Todos os novos bancos DEVEM ser registrados neste mapa.                                                                                           |
| RULE-009 | RulesConditionals Category Sets          | Os sets `SQL_DB_TYPES`, `NOSQL_DB_TYPES`, `GRAPH_DB_TYPES`, `COLUMNAR_DB_TYPES`, `NEWSQL_DB_TYPES`, `TIMESERIES_DB_TYPES`, `SEARCH_DB_TYPES` em `RulesConditionals` sao a unica fonte de verdade para roteamento de knowledge files por categoria. |
| RULE-010 | Padroes Genericos no KP data-modeling    | Padroes genericos (naming avancado, concurrency, encryption, test data) vivem no KP `data-modeling` e seus `references/`, NAO no diretorio `knowledge/databases/`. Este diretorio e estritamente para conteudo banco-especifico.                   |
| RULE-011 | Sem Duplicacao com Core Rule 11          | O arquivo `knowledge/core/11-database-principles.md` ja cobre principios genericos de naming e migracao. Novo conteudo no KP DEVE referenciar esse arquivo e fornecer apenas o delta (schema design patterns, concurrency, encryption, test data). |
