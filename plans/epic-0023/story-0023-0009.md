## 3. Descricao

Como desenvolvedor do ia-dev-environment, eu quero knowledge files para EventStoreDB seguindo o padrao de 3 arquivos, para que projetos event-sourced recebam padroes de stream design, projections e event versioning.

### 3.1 Contexto

EventStoreDB e posicionado sob `nosql/` (nao uma categoria separada) porque e um store especializado de documentos/streams. Criar uma 7a categoria para um unico banco seria over-engineering. O set `NOSQL_DB_TYPES` em RulesConditionals precisa incluir "eventstoredb".

### 3.2 Arquivos a Criar

**EventStoreDB (3 arquivos):**
- `knowledge/databases/nosql/eventstoredb/modeling-patterns.md` -- Stream design, convencoes de event schema, projections, subscriptions, category streams, naming de aggregate streams
- `knowledge/databases/nosql/eventstoredb/migration-patterns.md` -- Event versioning, upcasting, stream evolution, projection rebuild, snapshot management
- `knowledge/databases/nosql/eventstoredb/query-optimization.md` -- Subscription tuning, projection optimization, scavenge management, persistent subscriptions

**Settings file:**
- `targets/claude/settings/database-eventstoredb.json`

### 3.3 Mudanca Java

Adicionar "eventstoredb" ao `NOSQL_DB_TYPES` em `RulesConditionals.java`. Este set ja contem "mongodb" e "cassandra". O EventStoreDB sera roteado pelo mesmo mecanismo de NoSQL existente, sem necessidade de nova categoria.

### 3.4 Conteudo Esperado dos Knowledge Files

**modeling-patterns.md:**
- Stream design patterns (aggregate streams, category streams, projection streams)
- Event schema conventions (event type naming, metadata, causation/correlation IDs)
- Aggregate stream naming (`{Category}-{AggregateId}`)
- Category streams (`$ce-{Category}`)
- Projections (built-in vs user-defined, fromCategory, fromStream)
- Subscriptions (catch-up, persistent, competing consumers)

**migration-patterns.md:**
- Event versioning strategies (weak schema, upcasting, copy-and-replace)
- Upcasting (transforming old events to new schema on read)
- Stream evolution (splitting, merging, archiving streams)
- Projection rebuild (from $all stream, idempotent rebuilds)
- Snapshot management (snapshot streams, snapshot frequency)

**query-optimization.md:**
- Subscription tuning (batch size, buffer size, checkpoint interval)
- Projection optimization (partitioning, caching, state management)
- Scavenge management (dead event cleanup, disk reclamation)
- Persistent subscriptions (consumer strategies, parking, retries)
- Index optimization (stream index, event type index)

## 3.5 Entrega de Valor

- **Valor Principal:** Projetos event-sourced recebem orientacao completa de EventStoreDB
- **Metrica de Sucesso:** 3 knowledge files criados + 1 settings file + RulesConditionals atualizado
- **Impacto no Negocio:** Equipes usando EventStoreDB recebem padroes de stream design e event versioning, evitando erros comuns de event schema evolution e projection management
