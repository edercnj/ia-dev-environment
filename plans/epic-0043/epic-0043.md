# Épico: Standardize Interactive Gates with Fixed-Option Menus

**Autor:** Eder Celeste Nunes Junior
**Data:** 2026-04-16
**Versão:** 1.0
**Status:** Em Refinamento
**Épico ID:** EPIC-0043

## 1. Visão Geral

**Chave Jira:** —

Skills orquestradoras (`x-release`, `x-story-implement`, `x-epic-implement`, `x-review-pr`) pausam a execução em gates interativos emitindo **mensagens de texto abertas** que descrevem próximos passos. Exemplo observado no release v3.6.0 (PR #392): a Phase 8 APPROVAL-GATE imprime "Skill pausada. Após merge da PR #392, rode `/x-release 3.6.0 --continue-after-merge` para retomar" e termina. O operador precisa memorizar o comando exato de retomada, não tem atalho estruturado para corrigir o PR (exige sair e invocar `/x-pr-fix` manualmente) e não há loop-back automático após a correção. Em algumas skills existe `AskUserQuestion` estruturado, mas apenas sob flags opt-in (`--interactive`, `--manual-task-approval`, `--manual-batch-approval`) — o default continua sendo HALT textual. Não existe ADR nem regra que normalize a forma desses gates.

Este épico entrega três frentes complementares. Primeiro, uma convenção formal (ADR-0005 + Rule 20) que define o padrão obrigatório: menu fixo de 3 opções `PROCEED` / `FIX-PR` / `ABORT`, com `FIX-PR` invocando `x-pr-fix` ou `x-pr-fix-epic` via Rule 13 Pattern 1 INLINE-SKILL e reapresentando o mesmo menu ao concluir (loop-back). Menu passa a ser o comportamento **default**; novo flag `--non-interactive` substitui as flags opt-in atuais para uso em CI/automação. Segundo, retrofit das 4 skills com gate hoje existente para eliminar mensagens abertas e flags opt-in. Terceiro, audit de CI (grep-based, análogo à Rule 13) que rejeita regressões: mensagens `HALT` textuais em SKILL.md sem `AskUserQuestion` emparelhado, e reaparição das flags `--interactive` / `--manual-*`.

Fora de escopo: dashboards de métricas de uso de gates, persistência cruzada de estado entre skills distintas (cada skill mantém seu próprio `*-state.json`), opções além das 3 canônicas (propostas futuras terão que estender a Rule 20), e gates interativos em skills que hoje operam exclusivamente em lote sem ponto de parada (`x-code-format`, `x-code-lint`, `x-test-run`, etc.).

## 3. Definições de Qualidade Globais

### Global Definition of Ready (DoR)

- Source-of-truth em `java/src/main/resources/targets/claude/skills/` respeitado; nenhuma edição em `.claude/skills/`, `.claude/rules/`, `.claude/templates/` ou `src/test/resources/golden/**`
- Taxonomia ADR-0003 observada: nenhuma nova skill — apenas ADR, rule e retrofits em skills existentes
- Padrões de invocação Rule 13 respeitados: o handler de `FIX-PR` usa Pattern 1 INLINE-SKILL (`Skill(skill: "x-pr-fix", args: "<pr>")` ou `Skill(skill: "x-pr-fix-epic", args: "--epic <id>")`); bare-slash proibido fora de `## Triggers` e `## Examples`
- `AskUserQuestion` reconhecido como ferramenta permitida nas skills afetadas (frontmatter `allowed-tools` ajustado onde necessário)
- `planningSchemaVersion: "2.0"` declarado no `execution-state.json` deste épico
- Próximo slot de rule identificado: `20-interactive-gates.md` (slots 13 ocupado, 14 ocupado, 15–19 ocupados — 10/11/12 reservados para regras condicionais)

### Global Definition of Done (DoD)

- **Cobertura:** ≥ 95% Line, ≥ 90% Branch em qualquer helper Java novo (golden-assertion helpers, audit script runners). A convenção em si é prompt-driven — coberta por golden diff dos SKILL.md afetados.
- **Testes Automatizados:** unit tests para helpers Java novos; golden diff tests obrigatórios para cada SKILL.md retrofitado (`x-release`, `x-story-implement`, `x-epic-implement`, `x-review-pr`); golden diff para nova rule `20-interactive-gates.md`; audit script rodando em CI com exit code ≠ 0 em regressão.
- **Smoke Tests:** `mvn process-resources && mvn test` verde; `SkillsAssemblerTest` não regride; novo `RuleAssemblerTest.listRules_includesInteractiveGates` obrigatório.
- **Relatório de Cobertura:** JaCoCo agregado no Maven report padrão.
- **Documentação:** ADR-0005 criado; Rule 20 criada; CLAUDE.md atualizado no bloco "In progress" com referência ao épico enquanto não concluído; CHANGELOG seção Unreleased com uma entrada por story. `.claude/skills/**`, `.claude/rules/**` e `.claude/README.md` apenas regenerados via `mvn process-resources` — nunca editados à mão (RULE-001).
- **Persistência:** Schema de state file unificado — cada skill com gate mantém seu `*-state.json` com os campos canônicos `phase`, `lastPhaseCompletedAt`, `lastGateDecision`, `fixAttempts[]`, `schemaVersion="1.0"`. Escrita atômica via `.tmp` + rename.
- **Performance:** Nenhum impacto em caminho happy path. Overhead do gate interativo = ida-e-volta ao usuário (independente da implementação da skill).
- **TDD Compliance:** commits mostram test-first — golden diff precede ou acompanha cada diff de SKILL.md. Refactoring explícito após green. Tasks pequenas (Doc/Test size S–M) via TPP.
- **Double-Loop TDD:** acceptance tests derivados dos cenários Gherkin (outer loop). Unit tests de helpers Java guiados por TPP (inner loop).
- **Backward Compatibility:** skills retrofitadas devem respeitar `--non-interactive` com semântica idêntica ao HALT textual atual (para não quebrar CI/automação existente). Flags antigas (`--interactive`, `--manual-task-approval`, `--manual-contract-approval`, `--manual-batch-approval`) emitem warning de deprecação uma vez e ativam `--non-interactive` inverso; remoção dura em épico futuro após período de depreciação. Legacy epics (`planningSchemaVersion == "1.0"`) não regridem (Rule 19).
- **Rule 13 Audit:** grep padrão Rule 13 continua retornando 0 em `java/src/main/resources/targets/claude/skills/core/` após retrofits.
- **Rule 20 Audit:** novo audit (STORY-0043-0006) retorna 0 matches para HALT textual aberto em SKILL.md e 0 matches para flags removidas.

## 4. Regras de Negócio Transversais (Source of Truth)

| ID | Título | Descrição |
| :--- | :--- | :--- |
| **[RULE-001]** | Source-of-Truth Invariant | Toda alteração de skill/rule/template/golden é feita em `java/src/main/resources/targets/claude/` ou `java/src/main/resources/shared/`. `.claude/skills/**`, `.claude/rules/**`, `.claude/templates/**` e `src/test/resources/golden/**` são SAÍDAS geradas por `mvn process-resources` — hand-edits são sobrescritos no próximo build. |
| **[RULE-002]** | Fixed-Option Menu Canônico | Todo gate interativo DEVE oferecer EXATAMENTE 3 opções canônicas: `PROCEED` (avança o fluxo assumindo pré-condições humanas cumpridas), `FIX-PR` (invoca `x-pr-fix` ou `x-pr-fix-epic` via INLINE-SKILL e reapresenta o mesmo menu ao retornar), `ABORT` (encerra a skill com cleanup — possivelmente com confirmação dupla). Labels e `header` das opções seguem o glossário definido em ADR-0005 — skill-specific nuances são permitidas apenas dentro dos descriptions. |
| **[RULE-003]** | Default Interactive, Opt-out via `--non-interactive` | O menu estruturado é o comportamento DEFAULT. Skills exporão um novo flag `--non-interactive` com semântica idêntica ao HALT textual legado (imprime próximos passos + exit 0 sem persistir decisão). Flags antigas (`--interactive`, `--manual-task-approval`, `--manual-contract-approval`, `--manual-batch-approval`) são MANTIDAS durante depreciação: emitem warning uma vez e ativam `--non-interactive` implicitamente. Remoção dura fica para épico futuro. |
| **[RULE-004]** | FIX-PR Loop-Back Obrigatório | A opção `FIX-PR` invoca uma skill de correção via Rule 13 Pattern 1 INLINE-SKILL (`Skill(skill: "x-pr-fix", args: "<pr>")` para gates de PR único, `Skill(skill: "x-pr-fix-epic", args: "--epic <id>")` para gates que consolidam múltiplos PRs). Ao retornar da skill de correção, o controle VOLTA ao mesmo menu — nunca pula para próxima phase nem aborta. Há um guard-rail de no máximo 3 invocações consecutivas de `FIX-PR` por gate; ao atingir o limite, o menu incorpora uma 4ª opção temporária `FORCE-PROCEED` que pula o loop e avança assumindo responsabilidade humana (registrado no state file). |
| **[RULE-005]** | Rule 13 Invocation Patterns | Retrofits DEVEM usar INLINE-SKILL (`Skill(skill: "...", args: "...")`) para delegar a `x-pr-fix` / `x-pr-fix-epic` dentro do handler `FIX-PR`. Bare-slash em delegação é proibido (Rule 13 §Forbidden); permitido apenas em `## Triggers` e `## Examples`. Frontmatter `allowed-tools` de cada skill retrofitada DEVE incluir `Skill` e `AskUserQuestion`. |
| **[RULE-006]** | Atomic, Reversible Commits | Cada story produz pelo menos um commit atômico por task, reversível via `git revert <sha>` sem tocar tasks irmãs (Rule 18). Mensagens seguem Conventional Commits (Rule 08): `docs(task-0043-YYYY-NNN): ...` para ADR/rule, `feat(task-0043-YYYY-NNN): ...` para retrofits de SKILL.md quando introduzem comportamento (menu default), `test(task-0043-YYYY-NNN): ...` para audit script e golden regen. |
| **[RULE-007]** | State File Schema Uniforme | Todo gate interativo que persista estado usa o shape canônico: `{ phase, lastPhaseCompletedAt, lastGateDecision ∈ {PROCEED, FIX_PR, ABORT, FORCE_PROCEED}, fixAttempts[] (histórico ordenado com {at, delegateSkill, prNumber, outcome}), schemaVersion: "1.0" }`. Escrita atômica via `.tmp` + rename. Localização canônica: `plans/<skill-domain>/<id>/state.json` (ex.: release usa `plans/release-state-{VERSION}.json`, mantido por compat; novos estados seguem o layout `plans/<domain>/<id>/state.json`). |
| **[RULE-008]** | Audit Enforcement | CI de cada PR DEVE rodar o audit da Rule 20 (grep-based, análogo ao audit da Rule 13): zero matches para mensagens `HALT` textuais em SKILL.md sem `AskUserQuestion` emparelhado no mesmo arquivo; zero matches para flags removidas/depreciadas em code paths não-documentação. O audit é obrigatório em `mvn test` após STORY-0043-0006. |
