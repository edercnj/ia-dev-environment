## 3. Descrição

Como **engenheiro operando skills orquestradoras** (x-release, x-story-implement, x-epic-implement, x-review-pr), eu quero uma convenção formal de gates interativos — publicada como ADR e regra de projeto — para que cada gate tenha o MESMO shape (3 opções canônicas, loop-back após fix, state persistente) e eu não precise lembrar comandos de retomada nem convenções diferentes por skill.

Hoje o padrão é misto: `x-release` tem `AskUserQuestion` estruturado apenas sob `--interactive`; `x-story-implement` sob `--manual-task-approval` / `--manual-contract-approval`; `x-epic-implement` sob `--manual-batch-approval`. O default em todos é um HALT textual que descreve próximos passos e exige o operador memorizar flags de retomada. Esta história cria o contrato que elimina essa variabilidade: publica o ADR-0005 (racional, trade-offs, alternativas) e a Rule 20 (regra normativa com audit command), e fica publicamente referenciado em CLAUDE.md para servir de gate de aceitação nas retrofits subsequentes.

### 3.1 ADR-0005 — Interactive Gates Convention

**Arquivo:** `adr/ADR-0005-interactive-gates-convention.md`

Estrutura seguindo template padrão de ADR existente (ver `adr/ADR-0002-skill-delegation-protocol.md`):

- **Status:** Accepted | 2026-04-16 | Eder Celeste Nunes Junior
- **Context:** inventário dos gates existentes, inconsistência atual, custo cognitivo do operador, incidente do release v3.6.0 (PR #392)
- **Decision:** 3 opções canônicas (`PROCEED`, `FIX-PR`, `ABORT`); loop-back após FIX-PR; default interativo + `--non-interactive` para CI; state file schema uniforme; glossário dos labels/headers usados por `AskUserQuestion`
- **Consequences:**
  - Positivas: UX uniforme; eliminação de flags opt-in; auditável por grep
  - Negativas: trabalho de migração dispersado em 4 skills; breaking change para scripts que dependiam do HALT textual (mitigado com `--non-interactive`)
- **Alternatives considered:** menu por-skill (rejeitado — fragmenta); manter HALT + opt-in (rejeitado — custo cognitivo não se paga); 4ª opção `FORCE-PROCEED` emergencial após 3 fixes (rejeitado — introduzir 4ª opção viola RULE-002 "EXATAMENTE 3 opções" e cria incerteza sobre o shape do menu; substituído por auto-ABORT com código `*_FIX_LOOP_EXCEEDED` e orientação textual para retomada manual via `--non-interactive`)
- **References:** Rule 13, Rule 20 (criada nesta story), EPIC-0039 prior art, EPIC-0042 para estilo

### 5.2 AskUserQuestion Glossary (canônico)

Cada gate interativo expõe **exatamente 3 opções** (Rule 002). O 2º slot ("loop-back") admite variantes de label contextuais, enumeradas exaustivamente abaixo — nenhum outro label é permitido. O 1º slot (PROCEED) e o 3º slot (ABORT) têm labels canônicos invariantes; suas `description` podem ser contextualizadas por skill (ex.: PROCEED em `x-review-pr` significa "re-disparar auto-remediation" — a semântica "avançar" depende do estado do gate).

| Slot | Opção canônica | `header` (≤12 chars) | `label` (1–5 words) | `description` |
| :--- | :--- | :--- | :--- | :--- |
| 1 (PROCEED) | PROCEED | `"Proceed"` | `"Continue (Recommended)"` | `"Avança o fluxo a partir do estado atual do gate. Semântica concreta é skill-specific: em x-release é 'PR merged, continue'; em x-review-pr após retry esgotado é 'Retry remediation (+2 loops)'; em x-story-implement Phase 2.2.9 é 'Approve task PR'. A description DEVE ser explícita sobre a ação efetiva no contexto da skill."` |
| 2 (LOOP-BACK, variante PR) | FIX-PR | `"Fix PR"` | `"Run x-pr-fix and retry"` | `"Invoca x-pr-fix no PR atual; ao retornar, reapresenta este menu"` |
| 2 (LOOP-BACK, variante sem PR) | REJECT | `"Reject"` | `"Regenerate and retry"` | `"Retorna ao passo anterior com feedback para regenerar o artefato (contrato, arquitetura); ao regenerar, reapresenta este menu. Usado apenas em gates que não têm PR associado (ex.: x-story-implement Phase 0.5 contract gate)"` |
| 3 (ABORT) | ABORT | `"Abort"` | `"Cancel the operation"` | `"Encerra a skill com cleanup; confirmação dupla para release/epic gates"` |

**Nota sobre exhausted-retry gates (x-review-pr):** o slot 1 PROCEED aqui significa "re-disparar auto-remediation (+2 retries)" — não há variante nova de nome, apenas `description` contextualizada. A RULE-002 proíbe novos nomes de slot; labels são fixos, `description` é flexível. Não existe variante `RETRY-REMEDIATION` — o que a story-0043-0005 descrevia como tal é o próprio PROCEED com description apropriada.

### 5.3 Error Codes (introduzidos pela Rule 20)

| Código | Condição | Mensagem (pt-BR) |
| :--- | :--- | :--- |
| `GATE_SCHEMA_INVALID` | state file não satisfaz §5.1 | `"State file inválido para gate em {path}: {campo} ausente ou mal-formado"` |

### 6.1 Máquina de Decisão do Gate (canônica)

```mermaid
stateDiagram-v2
    [*] --> MENU: gate alcançado (default interativo)
    [*] --> NON_INTERACTIVE: --non-interactive presente

    MENU --> PROCEED: slot 1 (PROCEED)
    MENU --> LOOP_BACK: slot 2 (FIX-PR / REJECT)
    MENU --> ABORT: slot 3 (ABORT)

    LOOP_BACK --> DELEGATE: Skill(skill: "x-pr-fix" | "x-pr-fix-epic") OR regenerate OR retry-remediation
    DELEGATE --> GUARD: retorno do delegate
    GUARD --> MENU: fixAttempts.size < 3 (loop-back)
    GUARD --> AUTO_ABORT: fixAttempts.size == 3 (GATE_FIX_LOOP_EXCEEDED)

    PROCEED --> [*]: phase avança
    ABORT --> [*]: skill encerra com cleanup
    AUTO_ABORT --> [*]: skill encerra com código GATE_FIX_LOOP_EXCEEDED e orientação manual
    NON_INTERACTIVE --> [*]: HALT textual + exit 0
```
