## 1. Visão Geral

**Chave Jira:** —

Ao executar um épico com `/x-dev-epic-implement`, cada história gera um PR individual que recebe comentários automatizados de revisão (Copilot, CodeRabbit, ou reviewers humanos). Atualmente, a correção desses comentários exige intervenção manual: o desenvolvedor precisa abrir cada PR, ler os comentários, classificá-los, e aplicar correções individualmente. Para um épico com 16 PRs (como o EPIC-0024), isso resulta em 60+ comentários distribuídos em múltiplos PRs — um processo tedioso e propenso a omissões.

Este épico introduz a skill `/x-fix-epic-pr-comments` que automatiza o ciclo completo: (a) descobre todos os PRs de um épico via `execution-state.json`, (b) coleta e classifica comentários de todos os PRs em batch, (c) gera um relatório consolidado de findings, (d) aplica correções para comentários actionable, (e) cria um único PR com todas as correções, e (f) responde aos comentários originais nos PRs.

**Escopo incluído:**
- 1 nova skill `/x-fix-epic-pr-comments` com SKILL.md completo
- Source template Java para distribuição na geração
- Registro no assembler e pipeline de geração
- Relatório consolidado de findings em markdown
- Integração com `x-fix-pr-comments` (reutilização da lógica de classificação)
- Hook opcional no `x-dev-epic-implement` como fase pós-execução
- Golden tests para todos os profiles

**Escopo excluído:**
- Mudanças na lógica de classificação do `x-fix-pr-comments` (reutiliza as-is)
- Suporte a comentários de revisão manual (apenas automated review comments)
- Integração com ferramentas externas de tracking além de GitHub
- Modificações no `execution-state.json` schema

## 3. Definições de Qualidade Globais

### Global Definition of Ready (DoR)

- Skill `x-fix-pr-comments` existente e funcional como referência
- Padrão de `execution-state.json` com campos `prNumber` e `prUrl` por história documentado
- Formato de classificação de comentários (actionable/suggestion/question/praise) definido
- Pelo menos 1 épico anterior executado com PRs mergeados como caso de teste (EPIC-0024)

### Global Definition of Done (DoD)

- **Cobertura:** ≥ 95% Line, ≥ 90% Branch para código Java novo
- **Testes Automatizados:** Golden tests para todos os profiles. Testes unitários para parsing de comments e classificação.
- **Smoke Tests:** Obrigatório. Cada história deve passar no smoke gate.
- **Relatório de Cobertura:** JaCoCo integrado ao `mvn verify`
- **Documentação:** CLAUDE.md atualizado com a nova skill ao final do épico
- **Performance:** Processamento de 100 comentários de 20 PRs em < 5 minutos
- **TDD Compliance:** Commits show test-first pattern. Explicit refactoring after green.
- **Double-Loop TDD:** Acceptance tests derived from Gherkin scenarios (outer loop). Unit tests guided by TPP (inner loop).

## 4. Regras de Negócio Transversais (Source of Truth)

> Regras que se aplicam a múltiplas histórias. Cada história referencia as regras pelo ID. Alterações de regra propagam automaticamente para todas as histórias dependentes.

| ID | Título | Descrição |
| :--- | :--- | :--- |
| **[RULE-001]** | Checkpoint como fonte de PRs | A skill DEVE ler `execution-state.json` do épico para descobrir todos os PRs. Cada story entry com `prNumber != null` é um PR candidato. Stories sem PR (status FAILED/BLOCKED antes da criação de PR) são ignoradas. |
| **[RULE-002]** | Classificação consistente com x-fix-pr-comments | A classificação de comentários DEVE seguir as mesmas heurísticas do `x-fix-pr-comments`: actionable, suggestion, question, praise, resolved. Não introduzir novas categorias. |
| **[RULE-003]** | PR único para todas as correções | Todas as correções DEVEM ser consolidadas em um único PR targeting `main`. O PR DEVE referenciar todos os PRs de origem no body com formato `Fixes comments from #N, #M, ...`. Branch name: `fix/epic-{epicId}-pr-comments`. |
| **[RULE-004]** | Relatório persistido antes das correções | O relatório consolidado de findings DEVE ser gerado e salvo em `plans/epic-XXXX/reports/pr-comments-report.md` ANTES de iniciar qualquer correção. Isso garante auditabilidade mesmo se as correções falharem. |
| **[RULE-005]** | Deduplicação cross-PR | Quando o mesmo comentário aparece em múltiplos PRs (ex: golden files idênticos), a correção DEVE ser aplicada uma única vez. O relatório DEVE agrupar duplicatas e indicar quantos PRs são afetados. |
| **[RULE-006]** | Fallback sem execution-state.json | Se `execution-state.json` não existir, a skill DEVE aceitar uma lista explícita de PRs via flag `--prs 143,144,145`. Isso permite uso standalone fora do contexto de épicos. |
| **[RULE-007]** | Dry-run obrigatório | A skill DEVE suportar `--dry-run` que gera apenas o relatório consolidado sem aplicar correções. Isso permite revisão humana antes das correções. |
| **[RULE-008]** | Respostas em pt-BR | Todas as respostas aos comentários de PR DEVEM seguir os templates de resposta definidos no `x-fix-pr-comments` (em pt-BR). Respostas incluem commit SHA quando a correção é aplicada. |
| **[RULE-009]** | Verificação pós-correção | Após aplicar todas as correções, a skill DEVE executar compile + test. Se qualquer teste falhar, as correções que causaram falha DEVEM ser revertidas individualmente (não reverter todas). |
| **[RULE-010]** | Idempotência | Se executada novamente no mesmo épico, a skill DEVE detectar o PR de correção existente (branch `fix/epic-{epicId}-pr-comments`) e perguntar se deve atualizar ou criar novo. Comentários já respondidos DEVEM ser ignorados. |
