# Épico: PR por Story no Orquestrador de Épicos

**Autor:** Arquiteto de Software
**Data:** 2026-04-05
**Versão:** 1.0
**Status:** Em Refinamento

## 1. Visão Geral

**Chave Jira:** —

Refatoração do skill `x-dev-epic-implement` para que cada story implementada gere seu próprio Pull Request targeting `main`, com ciclo completo de review (especialistas + tech lead) por story. Atualmente, todas as stories de um épico são mergeadas em uma única branch épica e um único mega-PR é criado ao final — tornando impossível a revisão humana em épicos com muitas stories (10, 15, 30+). A mudança elimina a branch épica, delega PRs ao `x-dev-lifecycle`, e introduz enforcement de dependências via PR merge status.

## 3. Definições de Qualidade Globais

### Global Definition of Ready (DoR)

- Especificação técnica das mudanças revisada e aprovada
- Seções afetadas do SKILL.md original identificadas com números de linha
- Dependências entre stories validadas (sem circularidade)
- Data contracts das mudanças definidos (schema execution-state.json, SubagentResult, flags CLI)

### Global Definition of Done (DoD)

- **Cobertura:** N/A — mudanças são em arquivos SKILL.md (markdown), não código de produção
- **Testes Automatizados:** Validação manual de consistência interna do SKILL.md (seções referenciadas existem, schemas são consistentes, exemplos refletem o novo modelo)
- **Smoke Tests:** Leitura completa do SKILL.md modificado verificando que não há referências órfãs a branch épica, Sections 1.4b/1.4c removidas, ou status REBASING
- **Relatório de Cobertura:** N/A
- **Documentação:** SKILL.md é a própria documentação — deve ser auto-consistente
- **Persistência:** Schema do `execution-state.json` documentado com todos os campos novos e removidos
- **Performance:** N/A
- **TDD Compliance:** Commits mostram padrão test-first quando aplicável. Refatoração explícita após green. Testes incrementais (do simples ao complexo via TPP).
- **Double-Loop TDD:** Critérios de aceite Gherkin como outer loop. Validações unitárias como inner loop.

## 4. Regras de Negócio Transversais (Source of Truth)

> Regras que se aplicam a múltiplas histórias. Cada história referencia as regras pelo ID.

| ID | Título | Descrição |
| :--- | :--- | :--- |
| **[RULE-001]** | Isolamento de Contexto de Subagents | Cada subagent recebe apenas metadados (story ID, branch, phase, flags). Nunca passar source code, knowledge packs ou diffs. O prompt template deve conter apenas referências a arquivos, não conteúdo inline. |
| **[RULE-002]** | Persistência Atômica de Checkpoint | O checkpoint (`execution-state.json`) é atualizado após cada transição de estado. Se o orquestrador crashar, o checkpoint reflete o último estado completo. Nunca deixar uma story em estado `IN_PROGRESS` sem atualização posterior. |
| **[RULE-003]** | Ordem de Dependências via PR Merge | Stories só podem ser executadas quando TODAS as dependências estão com PR merged na `main`. A verificação usa `gh pr view {prNumber} --json state` e exige `state == "MERGED"`. |
| **[RULE-004]** | Auto-merge Opcional | A flag `--auto-merge` permite merge automático após reviews aprovarem (`gh pr merge {prNumber} --merge`). Sem a flag, o orquestrador aguarda merge manual com polling configurável. |
| **[RULE-005]** | Pre-flight é Advisory | A análise de conflitos (Phase 0.5) emite warnings mas NÃO bloqueia execução paralela. Stories com `code-overlap-high` recebem warning mas continuam elegíveis para dispatch paralelo. |
| **[RULE-006]** | Integridade na Main | Os integrity gates rodam na `main` após merge dos PRs de cada fase. O orquestrador captura o SHA da `main` antes de cada fase e faz diff após todos os PRs da fase serem merged. |
| **[RULE-007]** | Prioridade por Caminho Crítico | Stories no caminho crítico têm prioridade na execução e no merge automático. `sortByCriticalPath()` ordena stories pela proximidade ao caminho crítico. |
| **[RULE-008]** | Rastreabilidade Epic-PR | Cada PR de story inclui referência ao épico no body ("Part of EPIC-{epicId}") para rastreabilidade. O relatório de execução contém tabela com links para todos os PRs. |
| **[RULE-009]** | Backward Compatibility | A flag `--single-pr` (nova, opcional) permite manter o comportamento legacy de 1 único PR para todo o épico. Default é per-story PR. Quando `--single-pr` é set, todo o fluxo antigo (branch épica, rebase-before-merge, mega-PR) é preservado. |
| **[RULE-010]** | Report Incremental | O relatório de execução do épico (`epic-execution-report.md`) é atualizado incrementalmente conforme cada story completa, não apenas ao final. Template usa `{{PR_LINKS_TABLE}}` com tabela per-story. |
