# Epico: Reestruturacao do diretorio de recursos para organizacao target-first

**Autor:** Claude (AI Assistant)
**Data:** 2026-04-04
**Versao:** 1.0
**Status:** Em Refinamento

## 1. Visao Geral

**Chave Jira:** --

O diretorio `java/src/main/resources/` possui 26 pastas top-level com naming inconsistente, conteudo duplicado entre targets (Claude, GitHub Copilot, Codex) e organizacao que nao reflete a estrutura dos YAMLs de configuracao. Recursos Claude nao tem prefixo, GitHub usa prefixo `github-*`, Codex usa `codex-*`. Pastas de conhecimento compartilhado (languages, frameworks, databases, patterns, protocols) estao misturadas com templates target-specific no mesmo nivel hierarquico.

Este epico reorganiza os recursos em 3 grupos claros: `targets/` (templates por assistente de IA), `knowledge/` (base de conhecimento compartilhada alinhada com secoes YAML) e `shared/` (templates cross-cutting). Resultado: de 26 pastas top-level para 4 (`META-INF`, `targets`, `knowledge`, `shared`).

**Escopo incluido:**

- Reorganizacao fisica das pastas de recursos (mover arquivos)
- Atualizacao de todos os assemblers Java que referenciam os diretorio antigos
- Atualizacao de probes do `ResourceResolver` e constantes de path
- Atualizacao de testes unitarios e golden files
- Evolucao do `ResourceResolver` com novo metodo `resolveResourceDir`

**Escopo excluido:**

- Alteracao do conteudo dos arquivos de recurso (apenas mover, nao editar)
- Mudancas no output gerado (a saida deve ser byte-for-byte identica)
- Refatoracao de logica de negocio dos assemblers
- Deduplicacao de conteudo entre agents Claude e GitHub Copilot (formatos diferem)

## 3. Definicoes de Qualidade Globais

### Global Definition of Ready (DoR)

- Estrutura proposta revisada e aprovada
- Mapeamento assembler-to-resource completo e validado
- Testes existentes executando com sucesso antes de qualquer mudanca

### Global Definition of Done (DoD)

- Output gerado byte-for-byte identico ao pre-migracao para todos os perfis
- **Cobertura:** >= 95% Line, >= 90% Branch
- **TDD Compliance:** Commits mostram padrao test-first (teste precede implementacao no git log), refactoring explicito apos green, testes incrementais via TPP
- **Double-Loop TDD:** Acceptance tests derivados dos cenarios Gherkin (outer loop), unit tests guiados por Transformation Priority Premise (inner loop)
- **Rastreabilidade @GK-N bidirecional:** Todo @GK-N mapeia para >= 1 AT-N, todo AT-N referencia um @GK-N valido
- **Commits atomicos por ciclo TDD:** Cada ciclo Red-Green-Refactor produz um ou mais commits atomicos com formato Conventional Commits
- **Test plan antes de implementacao:** Nenhuma historia inicia implementacao sem test plan gerado via `/x-test-plan`
- Zero regressao: funcionalidades existentes continuam funcionais
- Nenhum assembler referencia paths antigos
- `resource-config.json` (native-image) atualizado se necessario

## 4. Regras de Negocio Transversais

| ID | Titulo | Descricao |
| :--- | :--- | :--- |
| RULE-001 | Output Identico | A saida gerada pelo pipeline para todos os perfis de configuracao DEVE ser byte-for-byte identica antes e depois da reestruturacao. Qualquer divergencia e um bug bloqueante. |
| RULE-002 | Probe Path Consistency | Todo assembler que usa `ResourceResolver.resolveResourcesRoot(probe)` ou `resolveResourceDir(path)` DEVE ter seu probe atualizado para o novo path. Paths antigos nao devem existir no classpath apos migracao. |
| RULE-003 | Depth-Free Resolution | O novo metodo `resolveResourceDir` DEVE retornar o diretorio resolvido diretamente, sem aritmetica de profundidade (`depth`). Assemblers nao devem precisar calcular quantos niveis subir. |
| RULE-004 | Backward Compatibility do ResourceResolver | O metodo legado `resolveResourcesRoot(probe, depth)` DEVE continuar funcionando durante a migracao. Remocao so ocorre na STORY-008 (cleanup). |
| RULE-005 | Sem Duplicidade de Conteudo | Nenhum arquivo de recurso deve existir em mais de um local, EXCETO quando os formatos diferem entre targets (ex: agents Claude vs GitHub Copilot que tem rendering diferente). |
| RULE-006 | Alinhamento com YAML Config | Pastas em `knowledge/` DEVEM refletir as secoes do YAML de configuracao: `languages` (language), `frameworks` (framework), `databases` (data), `patterns` (architecture), `protocols` (interfaces), `security` (security), `infrastructure` (infrastructure), `cloud-providers` (cloud). |
| RULE-007 | Build Verde por Story | Cada story DEVE resultar em build verde (`mvn test` passing) ao final. Nao e aceitavel quebrar o build entre stories. |
