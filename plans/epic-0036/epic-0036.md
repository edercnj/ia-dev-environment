# Épico: Skill Taxonomy and Naming Refactor

**Autor:** Eder Celeste Nunes Junior
**Data:** 2026-04-12
**Versão:** 1.0
**Status:** Em Refinamento

## 1. Visão Geral

**Chave Jira:** —

Reorganização da árvore de skills sob `java/src/main/resources/targets/claude/skills/` em 10 categorias funcionais (`plan/`, `dev/`, `test/`, `review/`, `security/`, `code/`, `git/`, `pr/`, `ops/`, `jira/`) e renomeação de 19 skills para um esquema verbal consistente `x-{subject}-{action}`. O output gerado (`.claude/skills/`) permanece flat — a hierarquia existe apenas no source of truth. O `SkillGroupRegistry` hardcoded em Java é excluído e substituído por derivação do filesystem. Um guard script de CI garante que nomes antigos não reapareçam após as migrações.

## 3. Definições de Qualidade Globais

### Global Definition of Ready (DoR)

- ADR-0003 revisado e status "Proposed" confirmado
- Todas as 10 categorias definidas e validadas (D2 do ADR)
- Tabela de renames completa com 19 entradas (seções 2.1, 2.2, 2.3 do staging doc)
- Checklist de atualização por rename documentado (seção 3 do staging doc)
- Dependências entre stories validadas sem ciclos
- `mvn clean verify` passando na branch develop antes de iniciar qualquer story

### Global Definition of Done (DoD)

- **Cobertura:** ≥ 95% Line, ≥ 90% Branch
- **Testes Automatizados:** Unit tests para assemblers, golden file tests para output gerado, integration tests para SkillsAssembler com nova estrutura de diretórios
- **Smoke Tests:** Invocação manual de pelo menos 1 skill renomeada em projeto de teste
- **Relatório de Cobertura:** JaCoCo por módulo
- **Documentação:** CLAUDE.md, README.md, regras, templates, cross-skill references, ADR atualizados na mesma PR do rename
- **Persistência:** N/A (sem banco de dados)
- **Performance:** Tempo de assembly não deve degradar > 10% com a nova estrutura hierárquica
- **TDD Compliance:** Commits mostram padrão test-first. Refactoring explícito após green. Testes incrementais via TPP (Transformation Priority Premise).
- **Double-Loop TDD:** Acceptance tests derivados de cenários Gherkin (outer loop). Unit tests guiados por TPP (inner loop).

## 4. Regras de Negócio Transversais (Source of Truth)

> Regras que se aplicam a múltiplas histórias. Cada história referencia as regras pelo ID. Alterações de regra propagam automaticamente para todas as histórias dependentes.

| ID | Título | Descrição |
| :--- | :--- | :--- |
| **[RULE-001]** | Prefixo x- Obrigatório | Todas as skills usam o prefixo `x-` para evitar colisão com skills built-in do Claude Code. O prefixo `run-` é eliminado. Skills sem prefixo `x-` são tratadas como violação. |
| **[RULE-002]** | SoT Hierárquico, Output Flat | O source of truth (`targets/claude/skills/`) é organizado em subpastas por categoria. O output gerado (`.claude/skills/`) permanece flat — `SkillsAssembler` achata a hierarquia em tempo de assembly. Referências textuais a `skills/{name}/SKILL.md` continuam válidas sem alteração. |
| **[RULE-003]** | Taxonomia de 10 Categorias | 10 categorias: `plan`, `dev`, `test`, `review`, `security`, `code`, `git`, `pr`, `ops`, `jira`. Edge cases explícitos: `x-review-security` → review (não security), `x-code-audit` → review, `x-threat-model` → plan, `x-perf-profile` → ops. `core/lib/` e `knowledge-packs/` ficam fora do escopo. |
| **[RULE-004]** | Esquema Verbal de Nomenclatura | Naming segue `x-{subject}-{action}` com sufixos de ação reservados: `-create`, `-plan`, `-decompose`, `-map`, `-implement`, `-orchestrate`, `-update`, `-generate`, `-eval`, `-scan`. O eixo `epic → story → task` fica inequívoco: create = documento, plan/decompose/map = breakdown, implement/orchestrate = execução. |
| **[RULE-005]** | Hard Rename sem Aliases | Claude Code não suporta aliases de skills nativamente. Renames são atômicos por cluster em uma única PR. Sem aliases, stubs duplicados ou lógica customizada no assembler. |
| **[RULE-006]** | Checklist de Atualização por Rename | Para cada rename, 8 categorias de superfícies devem ser atualizadas na mesma PR: (1) diretório SoT, (2) referências cross-skill, (3) regras, (4) templates, (5) outros targets (Copilot, Codex), (6) documentação do projeto, (7) testes, (8) verificação pós-merge. |
| **[RULE-007]** | Golden File Regeneration | Após renames, executar `mvn process-resources` seguido de `GoldenFileRegenerator` para regenerar golden files. Reviewers devem inspecionar diff por sanidade estrutural. |
| **[RULE-008]** | Documentação como Deliverable | Atualizações de documentação (CLAUDE.md, README.md, ADRs, rules, templates, cross-skill refs) são deliverables de primeira classe. "Atualizar docs depois" é proibido — cada story inclui subsecção explícita de docs obrigatórias. (D6 do ADR-0003) |
| **[RULE-009]** | Guard Script de CI | Script de CI deve falhar o build se qualquer dos 19 nomes antigos aparecer fora de locais permitidos (staging doc, ADR-0003, CHANGELOG.md, release notes, o próprio script, diretórios de build). |
