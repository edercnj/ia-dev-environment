# Épico: File-Conflict-Aware Parallelism Analysis

**Autor:** Eder Celeste Nunes Junior
**Data:** 2026-04-13
**Versão:** 1.0
**Status:** Concluído (docs entregues; release deferida)
**Épico ID:** EPIC-0041

## 1. Visão Geral

**Chave Jira:** —

A análise de paralelismo atual em `x-epic-map` é puramente topológica: agrupa stories/tasks em fases com base apenas no DAG de `Blocked By`. Stories sem dependência declarada são consideradas paralelizáveis mesmo quando ambas escrevem nos mesmos arquivos (e.g., `SettingsAssembler.java`, golden files, `CLAUDE.md`), gerando conflitos de merge recorrentes durante execução em worktrees paralelos.

Este épico introduz **análise de paralelismo consciente de footprint de arquivos** em três níveis (epic, story, task). As skills de planejamento (`x-task-plan`, `x-story-plan`) passam a emitir um **File Footprint** estruturado (`write:` / `read:` / `regen:`). Uma nova skill `/x-parallel-eval` (com flag `--scope=epic|story|task`) consome esses footprints e produz uma **matriz de colisão** que demove pares conflitantes para execução serial dentro da mesma fase. `x-epic-map` é estendido com Step 8.5 que invoca essa análise e anota "Restrição de Paralelismo" no mapa. As skills de execução (`x-dev-epic-implement`, `x-dev-story-implement`) executam o gate antes de disparar worktrees paralelos e **degradam para serial com aviso** (não-bloqueante) em caso de colisão. Os mapas existentes dos épicos 0036–0040 (todos "Em Refinamento") são re-avaliados retroativamente, gerando patches `.diff` para revisão humana.

Fora de escopo: análise estática a nível de classe/método, integração com IDE, lock-file distribuído entre worktrees em runtime.

## 3. Definições de Qualidade Globais

### Global Definition of Ready (DoR)

- Heurísticas de colisão de arquivos catalogadas (exclusive-write, append-only, golden-regen, hotspots conhecidos)
- Formato canônico do bloco **File Footprint** definido (Markdown estruturado parseável por regex)
- Política de degradação confirmada com o usuário (degrade + warn, não abortar)
- Compatibilidade backward com plans/maps existentes garantida (footprint ausente = bypass do gate, não erro)

### Global Definition of Done (DoD)

- **Cobertura:** ≥ 95% Line, ≥ 90% Branch em `dev.iadev.parallelism.*`
- **Testes Automatizados:** Unit tests para parser de footprint e detector de colisão; integration tests para `/x-parallel-eval` com fixtures de epics reais; golden tests para o output de `x-epic-map` com Step 8.5 ativo
- **Smoke Tests:** Re-rodar `x-epic-map` em epic-0040 deve produzir map idêntico ao atual + nova seção "8.5 Restrições de Paralelismo" com 0 colisões reais detectadas
- **Documentação:** ADR-0005 + CLAUDE.md + arquivo Source-of-Truth em `java/src/main/resources/targets/claude/README.md` com seção "Parallelism Analysis"; `.claude/README.md` apenas regenerado/verificado após a atualização (output gerado por `mvn process-resources`, não editar à mão — RULE-007)
- **Performance:** `/x-parallel-eval --scope=epic` para um épico de 15 stories em < 3s
- **Backward Compatibility:** Maps gerados antes desta release continuam válidos; ausência de File Footprint nos plans dispara warning, não erro
- **TDD Compliance:** Commits show test-first pattern. Refactoring explícito após green. Tests incrementais via TPP.
- **Double-Loop TDD:** Acceptance tests derivados de cenários Gherkin (outer loop). Unit tests guiados por TPP (inner loop).

## 4. Regras de Negócio Transversais (Source of Truth)

| ID | Título | Descrição |
| :--- | :--- | :--- |
| **[RULE-001]** | File Footprint Estruturado | Task plans DEVEM conter uma seção `## File Footprint` (heading canônico para escopo de task). Story plans DEVEM conter uma seção `## Story File Footprint` (heading canônico para agregação no escopo de story, resultante da união dos footprints das tasks da story). Ambos compartilham o mesmo schema: 3 sub-seções `### write:` (arquivos modificados), `### read:` (arquivos lidos como contexto), `### regen:` (arquivos regenerados via mvn/golden). Paths são relativos à raiz do repo. Wildcards (`**`) são permitidos para diretórios inteiros. Headings fora desta lista são rejeitados pelo parser para garantir parsing determinístico (regex-based). |
| **[RULE-002]** | Granularidade por Arquivo | Footprints rastreiam apenas paths de arquivo. Granularidade a nível de classe/método/símbolo está fora de escopo nesta versão. |
| **[RULE-003]** | Categorias de Conflito | (a) **Hard conflict** — duas tasks/stories listam o mesmo path em `write:`. (b) **Regen conflict** — uma lista path em `write:` e outra em `regen:` (golden file regeneration corre serializada). (c) **Soft conflict** — overlap apenas em `read:` (não é colisão; ignorado). Regras (a) e (b) bloqueiam paralelismo. |
| **[RULE-004]** | Hotspots Conhecidos | Arquivos historicamente conflitantes (`SettingsAssembler.java`, `HooksAssembler.java`, `CLAUDE.md`, `.gitignore`, `CHANGELOG.md`, `pom.xml`, golden files em `src/test/resources/golden/**`) DEVEM ser tratados como **exclusive-write**: qualquer task que os toque força serialização com qualquer outra que também os toque, mesmo cross-fase. |
| **[RULE-005]** | Degrade with Warning | Em caso de colisão detectada antes da execução, a skill executora (`x-dev-epic-implement`, `x-dev-story-implement`) DEVE rebaixar paralelismo para serial dentro da fase E logar um aviso visível com a lista de pares conflitantes e o motivo. NÃO aborta a execução. |
| **[RULE-006]** | Backward Compatibility | Plans/maps gerados antes desta release não possuem File Footprint. `/x-parallel-eval` DEVE tratar footprint ausente como "footprint desconhecido" e emitir warning, mas NÃO bloquear a execução. Re-geração com `--force` é o caminho recomendado de migração. |
| **[RULE-007]** | Source of Truth: Resources | Toda alteração de skill/rule é feita em `java/src/main/resources/targets/claude/`. Hand-edits em `.claude/` são perdidos no próximo `mvn process-resources`. |
| **[RULE-008]** | Output Determinístico | Saídas de `/x-parallel-eval` (matriz, recomendações, patches `.diff`) DEVEM ser determinísticas dado o mesmo input — ordenação por path/ID alfabético, sem timestamps embutidos no corpo, para permitir golden tests. |
