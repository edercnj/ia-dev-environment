## 3. Descrição

Como **engenheiro de plataforma**, eu quero que a análise de conflitos do Phase 0.5 emita warnings sem bloquear execução paralela por default, mas ofereça um modo `--strict-overlap` que restaura o comportamento bloqueante para stories com overlap alto, garantindo flexibilidade entre throughput e segurança.

Atualmente, Phase 0.5 (Sections 0.5.1-0.5.5) detecta overlaps de arquivos entre stories de uma mesma fase e particiona as stories em "Parallel Batch" e "Sequential Queue". Stories com `code-overlap-high` ou `unpredictable` são forçadas a executar sequencialmente. Com per-story PRs, conflitos são resolvidos no nível do PR (auto-rebase + resolução automática via story-0021-0009), não no nível de worktree merge. A análise de overlap passa a ser informativa por default, mas o modo `--strict-overlap` permite restaurar o particionamento bloqueante como opt-in para épicos com alto risco de conflito.

### 3.1 Alteração de Section 0.5.4 (Generate Adjusted Execution Plan)

- **Modo default (sem `--strict-overlap`):** Remover particionamento em "Parallel Batch" e "Sequential Queue". Todas as stories ficam no batch paralelo, independente do nível de overlap. O output muda de plano de execução ajustado para relatório de warnings.
  - Stories com `code-overlap-high`: emitir WARNING, não demover para sequencial
  - Stories `unpredictable`: emitir WARNING, não demover para sequencial
- **Modo strict (`--strict-overlap`):** Preservar particionamento para stories com overlap alto. Stories com `code-overlap-high` ou `unpredictable` são demovidas para sequential queue (comportamento original). Stories com `no-overlap`, `config-only` e `code-overlap-low` ficam no batch paralelo.

### 3.2 Alteração de Section 0.5.5 (Integration with Core Loop)

- **Modo default:** O core loop trata todas as stories como paralelo-elegíveis. Se preflight analysis existe, warnings são logados mas não afetam dispatch.
- **Modo strict:** O core loop consome o particionamento: stories no batch paralelo são despachadas simultaneamente; stories na sequential queue são despachadas uma por vez após o batch paralelo.
- Adicionar nota: "Com per-story PRs, conflitos de overlap são resolvidos automaticamente via auto-rebase (story-0021-0009). O modo `--strict-overlap` é recomendado para épicos com muitas stories editando os mesmos arquivos."
- `--sequential` tem precedência sobre `--strict-overlap` (se ambos set, todas as stories são sequenciais)

### 3.3 Formato do relatório advisory

- O arquivo `preflight-analysis-phase-N.md` continua sendo gerado
- Tabela de overlaps continua com classificações (config-only, code-overlap-low, code-overlap-high)
- **Modo default:** Seção "Adjusted Execution Plan" muda para "Advisory Warnings". Lista de warnings com recomendação: "Auto-rebase será executado após merge do primeiro PR da fase (story-0021-0009)"
- **Modo strict:** Seção "Adjusted Execution Plan" mantida com "Parallel Batch" e "Sequential Queue"

## 3.5 Entrega de Valor

- **Valor Principal:** Maximização do throughput de stories — análise de conflitos não bloqueia mais execução paralela, acelerando a implementação de épicos grandes
- **Métrica de Sucesso:** Todas as stories de uma fase são despachadas em paralelo independente do nível de overlap, com warnings logados para visibilidade
- **Impacto no Negócio:** Redução do tempo total de implementação de épicos, com manutenção da visibilidade de riscos de conflito

3. story-0042-0004 (unpredictable — sem plano de implementação)
