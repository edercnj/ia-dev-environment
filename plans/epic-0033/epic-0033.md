# EPIC-0033: Fix Skill Delegation Chain and Subagent Observability

## Status: Pendente

## Summary

The skill orchestration chain (`x-dev-epic-implement` -> `x-dev-story-implement` -> `x-tdd` -> `x-commit` -> `x-format`/`x-lint`) has fundamental issues that prevent proper delegation and provide zero visibility to users during execution. This epic addresses 4 problems discovered through deep analysis of the delegation architecture.

## Business Value

- **Reliability:** Fix broken delegation chain so skills actually invoke each other via Skill tool instead of ambiguous text instructions
- **Observability:** Give users real-time visibility into what's happening during multi-minute skill executions
- **Consistency:** Consolidate duplicate skill naming (x-dev-lifecycle vs x-dev-story-implement)
- **Developer Experience:** Reduce confusion and debugging time when skill orchestration fails silently

## Problem Statement

### Problem 1: Broken Skill Tool Chain

The delegation chain requires each skill to invoke the next via the `Skill` tool. However, inner skills lack `Skill` in their `allowed-tools`:

| Skill | Has `Skill` in allowed-tools? | Tries to invoke |
|-------|-------------------------------|-----------------|
| x-dev-story-implement | Yes | x-tdd, x-pr-create, x-review |
| x-tdd | **NO** | /x-commit (RED/GREEN/REFACTOR) |
| x-commit | **NO** | /x-format, /x-lint |
| x-format | NO (leaf) | — |
| x-lint | NO (leaf) | — |

The `/x-commit` syntax in x-tdd looks like a Skill invocation, but the tool is unavailable. The "read Slim Mode section" instruction creates ambiguity about whether to invoke via Skill tool or read-and-follow inline.

### Problem 2: Zero Visibility in Claude Code Task List

No skill uses `TaskCreate`/`TaskUpdate`. All tracking is via custom `execution-state.json`. The user sees an empty task list during 30+ minute executions with no indication of:
- Current phase (0, 1, 2, 3)
- Active task in execution loop
- TDD cycle progress (RED/GREEN/REFACTOR)
- Whether execution is progressing or stuck

### Problem 3: Planning Subagents Are Black Boxes

Phase 1 planning subagents (1B: Implementation Plan, 1D: Event Schema, 1E: Security fallback, 1F: Compliance) receive pure text prompts with no task tracking. Up to 5 subagents launch in parallel with zero progress reporting.

### Problem 4: Naming Inconsistency

Two nearly identical skills exist:
- `x-dev-lifecycle` — Referenced by x-dev-epic-implement subagent prompt
- `x-dev-story-implement` — Newer version with `AskUserQuestion` in allowed-tools

The epic orchestrator tells subagents to invoke `x-dev-lifecycle`, but the canonical name should be `x-dev-story-implement`.

## Scope

### In Scope
- Add `Skill` to `allowed-tools` of x-tdd and x-commit
- Standardize all inter-skill invocations to use explicit `Skill(skill: ..., args: ...)` syntax
- Add `TaskCreate`/`TaskUpdate` at Phase and Task granularity (Level 2)
- Add task tracking to planning subagent prompts
- Consolidate naming to `x-dev-story-implement`
- Update all cross-references in x-dev-epic-implement
- Regenerate affected golden files

### Out of Scope
- Changes to x-format or x-lint internals (they are leaf skills)
- Changes to execution-state.json schema (complementary to TaskCreate, not replaced)
- Changes to the Java generator code (only skill templates are modified)
- Level 3 TDD-cycle-level task tracking (deferred)

## Cross-Cutting Rules

- CR-01: All `allowed-tools` changes must be mirrored in both source templates and golden files
- CR-02: TaskCreate descriptions must follow pattern: `"{Phase/Task}: {description}"`, with optional trailing context `— {skill}` when useful (e.g., `"Phase 2: Task T-001"` or `"Task: Implement RED phase — /x-tdd"`). Phase/Planning task descriptions may omit the skill suffix.
- CR-03: Skill invocations must use explicit `Skill(skill: "name", args: "...")` syntax, never ambiguous `/name` shorthand in non-user-facing instructions
- CR-04: execution-state.json tracking is complementary to TaskCreate, not replaced by it
