## Summary

The skill orchestration chain (`x-dev-epic-implement` -> `x-dev-story-implement` -> `x-test-tdd` -> `x-git-commit` -> `x-code-format`/`x-code-lint`) has fundamental issues that prevent proper delegation and provide zero visibility to users during execution. This epic addresses 3 problems discovered through deep analysis of the delegation architecture (the original 4th problem — naming inconsistency — was resolved by EPIC-0032).

## Business Value

- **Reliability:** Fix broken delegation chain so skills actually invoke each other via Skill tool instead of ambiguous text instructions
- **Observability:** Give users real-time visibility into what's happening during multi-minute skill executions via native Claude Code task list
- **Developer Experience:** Reduce confusion and debugging time when skill orchestration fails silently

## Problem Statement

### Problem 1: Broken Skill Tool Chain

The delegation chain requires each skill to invoke the next via the `Skill` tool. However, inner skills lack `Skill` in their `allowed-tools` (verified against `origin/develop` HEAD post EPIC-0032):

| Skill | Has `Skill` in allowed-tools? | Tries to invoke (ambiguous `Invoke /x-` syntax) |
|-------|-------------------------------|-----------------|
| x-dev-story-implement | Yes | x-test-tdd, x-pr-create, x-review, x-review-pr |
| x-dev-epic-implement | Yes | x-pr-fix-epic-comments (Phase 4) |
| x-review | Yes | x-review-qa, x-review-perf, x-review-db, x-review-obs, x-review-devops, x-review-data-modeling, x-review-security, x-review-api, x-review-events |
| x-test-tdd | **NO** | /x-git-commit (RED/GREEN/REFACTOR + Slim Mode) |
| x-git-commit | **NO** | /x-code-format, /x-code-lint |
| x-code-format | NO (leaf) | — |
| x-code-lint | NO (leaf) | — |

Plus the reference files have additional routes:
- `x-dev-story-implement/references/planning-phases.md` has **1** additional ambiguous route at line 20 (Phase 1A: `/x-dev-architecture-plan`). Phases 1B test-plan, 1C task-decomposer, and 1E threat-model already use explicit `Skill(skill: ...)` syntax correctly.
- `x-dev-story-implement/references/verification-phase.md` has **3** routes (`x-dev-arch-update` @ L48, `/x-review` @ L52-54, `x-review-pr` @ L83). The first two are synchronized duplicates of entries in `x-dev-story-implement/SKILL.md` and must be edited in both files.
- `x-owasp-scan/SKILL.md` has **1** route at line 273 (`/x-dependency-audit --scope vulnerabilities`).

The `/x-git-commit` syntax in x-test-tdd looks like a Skill invocation, but the tool is unavailable in its allowed-tools. The "read Slim Mode section" instruction creates ambiguity about whether to invoke via Skill tool or read-and-follow inline. Even in skills that DO have `Skill` in allowed-tools (x-dev-story-implement, x-review), the body text uses `Invoke /x-foo` instead of the canonical `Skill(skill: "x-foo", args: "...")` syntax, leaving the LLM to guess the delegation mechanism.

**Diagnostic count:** **13 logical delegation routes identified across 7 files (24 physical locations total)**. Some logical routes span multiple lines within the same file — e.g., `x-test-tdd → x-git-commit` appears at 4 locations (RED @ 195, GREEN @ 231, REFACTOR @ 264, Slim Mode reference @ 433). Other routes are synchronized duplicates between `SKILL.md` and `references/` files (e.g., `x-dev-arch-update` and `/x-review` exist in both `x-dev-story-implement/SKILL.md` and `verification-phase.md`). **1 of the 24 references** (`/x-test-contract-lint` in `x-dev-story-implement/SKILL.md:269`) points to a **non-existent skill** and must be removed as an orphan. Before this epic, only `x-epic-plan` and `planning-phases.md` Phase 1B/1C/1E use canonical `Skill(skill: ...)` syntax.

### Problem 2: Zero Visibility in Claude Code Task List

No skill uses `TaskCreate`/`TaskUpdate`. All tracking is via custom `execution-state.json`. The user sees an empty task list during 30+ minute executions with no indication of:
- Current phase (0, 1, 2, 3)
- Active task in execution loop
- TDD cycle progress (RED/GREEN/REFACTOR)
- Whether execution is progressing or stuck

### Problem 3: Planning Subagents Are Black Boxes

Phase 1 planning subagents (1B: Implementation Plan, 1D: Event Schema, 1E: Security fallback, 1F: Compliance) receive pure text prompts with no task tracking. Up to 5 subagents launch in parallel with zero progress reporting.

## Scope

- Add `Skill` to `allowed-tools` of `x-test-tdd` and `x-git-commit`

- STORY-0033-0004 (naming consolidation) — already done by EPIC-0032

## Cross-Cutting Rules

- CR-01: All `allowed-tools` changes must be mirrored in both source templates and golden files
- CR-02: TaskCreate descriptions must follow pattern: `"{Phase/Task}: {description}"`, with optional trailing context `— {skill}` when useful (e.g., `"Phase 2: Task T-001"` or `"Task: Implement RED phase — x-test-tdd"`). Phase/Planning task descriptions may omit the skill suffix.
- CR-03: Skill invocations must use explicit `Skill(skill: "name", args: "...")` syntax in skill body text. The bare slash shorthand `/x-foo` is only permitted in the user-facing `## Triggers` and `## Examples` sections where the user types the command.
- CR-04: execution-state.json tracking is complementary to TaskCreate, not replaced by it.
- CR-05 (NEW): The new rule `10-skill-invocation-protocol.md` is authoritative. Any delegation added after this epic MUST follow one of its 3 permitted patterns (INLINE-SKILL, SUBAGENT-SKILL, SUBAGENT-RESEARCH).
