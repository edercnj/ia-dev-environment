# STORY-0033-0001: Add Skill to allowed-tools and Standardize 13 Delegation Routes

## Status: Pendente

## Parent Epic: EPIC-0033

## Depends On: — (no dependencies; STORY-0033-0004 is VOID per EPIC-0032 reconciliation)

## Summary

Fix the delegation chain in `core/` skills by (1) adding `Skill` to the `allowed-tools` frontmatter of `x-test-tdd` and `x-git-commit` (currently missing, blocking delegation at runtime), (2) replacing **13 logical delegation routes across 7 files (24 physical locations)** that use ambiguous `Invoke /x-foo` syntax with explicit `Skill(skill: "x-foo", args: "...")` invocations, (3) removing **1 orphan reference** to the non-existent `/x-test-contract-lint` skill, and (4) creating the formal rule `.claude/rules/10-skill-invocation-protocol.md` documenting the canonical delegation patterns.

## User Story

As a **developer using skill orchestration**, I want every inter-skill invocation in the `core/` templates to be an explicit `Skill(...)` tool call with matching `allowed-tools`, so that every step of the chain `x-dev-epic-implement → x-dev-story-implement → x-test-tdd → x-git-commit → x-code-format/x-code-lint` (plus `x-review` fan-out) shows up as a real tool call in Claude Code and delegation never falls through to inline execution.

## Background

Post-EPIC-0032 audit against `origin/develop` HEAD found:

**Skills with `Skill` in `allowed-tools`:** `x-dev-epic-implement`, `x-dev-story-implement`, `x-review` — but their body text still uses `Invoke /x-foo` instead of `Skill(skill: "x-foo", ...)`, leaving the LLM to guess the delegation mechanism.

**Skills MISSING `Skill` in `allowed-tools`:** `x-test-tdd` and `x-git-commit`. Even if the body text pointed to the right tool, the runtime would reject the call.

## Data Contracts

### Input: `x-test-tdd` frontmatter (before)
```yaml
allowed-tools: Bash, Read, Write, Edit, Grep, Glob
```

### Output: `x-test-tdd` frontmatter (after)
```yaml
allowed-tools: Bash, Read, Write, Edit, Grep, Glob, Skill
```

### Input: `x-git-commit` frontmatter (before)
```yaml
allowed-tools: Bash, Read, Grep, Glob, Write, Edit
```

### Output: `x-git-commit` frontmatter (after)
```yaml
allowed-tools: Bash, Read, Grep, Glob, Write, Edit, Skill
```

### Delegation syntax (before — forbidden by Rule 10 after this story)
```markdown
6. Stage test files and delegate commit to `/x-git-commit`:

   /x-git-commit --task TASK-XXXX-YYYY-NNN --type test --subject "add test for [description]" --tdd RED
```

### Delegation syntax (after — canonical INLINE-SKILL pattern)
```markdown
6. Stage test files and delegate commit to `x-git-commit` via the Skill tool:

       Skill(skill: "x-git-commit", args: "--task TASK-XXXX-YYYY-NNN --type test --subject \"add test for [description]\" --tdd RED")
```

### `x-review` Phase 2 (before — bullet list)

/x-review-qa {STORY_ID}

## Risks

| Risk | Mitigation |
|------|------------|
| Recursion between skills after adding `Skill` to `allowed-tools` (e.g., `x-test-tdd` → `x-git-commit` → ...?) | Chain is strictly directional; Rule 10 explicitly prohibits backward invocation. Functional test Cenário A validates end-to-end before merge. |
| `x-review` Phase 2 parallel dispatch stresses context windows when >4 specialists run simultaneously | Specialists are context-isolated by design (each runs its own skill). If OOM surfaces in testing, fall back to batches of 4 per message. |
| The orphan `/x-test-contract-lint` reference (L269) may have been intentional but broken by an earlier refactor | Investigate in T-005g: if git blame shows the reference was added alongside a skill that was later renamed, restore the correct name. Otherwise remove. |
| Synced edits (T-005h, T-005i) can drift between `SKILL.md` and `references/` files | T-005n self-check re-reads both files after the edit and compares literal phrasing. |
| Large diff in golden files (~28k generated files across 18 profiles) | Regenerate **once** at the end of the story (T-006), not per-edit. PR review focuses on the template files; golden delta is mechanical consequence. |
| Rule 10 conflicts with existing rules | Rule 10 is about invocation syntax; Rule 07 (pre-commit chain ordering) is about chain order; Rule 09 (branching) is unrelated. Review during T-003. |
