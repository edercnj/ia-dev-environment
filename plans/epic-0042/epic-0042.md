# Épico: Merge-Train Automation + Auto PR-Fix Hook

**Autor:** Eder Celeste Nunes Junior
**Data:** 2026-04-15
**Versão:** 1.0
**Status:** Em Refinamento
**Épico ID:** EPIC-0042

## 1. Visão Geral

**Chave Jira:** —

O fechamento de épicos neste projeto hoje é manual: após o último PR de task ser aprovado, o operador precisa fazer merge sequencial de 3–5 PRs encadeados, rebasear cada PR seguinte sobre `develop` atualizada, resolver conflitos em goldens (`src/test/resources/golden/**`) via `git checkout --ours` + regen, e empurrar (`push --force-with-lease`) um a um. A dor paralela: após o Tech Lead Review em `/x-story-implement` devolver GO, comentários não-blocker espalhados pelos PRs das tasks precisam ser varridos manualmente com `/x-pr-fix <pr>` antes do Step 3.7 (Story-Level PR). Esse trabalho manual consome 1–2 horas por fechamento de épico e produz conflitos recorrentes que já apagaram trabalho em sessões anteriores.

Este épico entrega duas automações complementares. Primeiro, uma nova skill `x-pr-merge-train` em `core/pr/` que consome uma lista de PRs vinda de `--prs`, `--epic` ou `--pattern`, valida todos (VETO em draft, CI vermelha, base errada, approval ausente ou merge conflict), faz merge da base via `gh pr merge --squash --auto`, dispara workers de rebase/regen/push em paralelo por onda respeitando `--max-parallel` e um pré-check de sobreposição de arquivos, e persiste progresso em `plans/merge-train/<id>/state.json` para `--resume`. Segundo, um hook de Step 3.6.5 em `x-story-implement` que roda `x-pr-fix` automaticamente após TL=GO, gated em single-pass, com abort duro em `PR_FIX_COMPILE_REGRESSION`.

Fora de escopo: merge cross-epic sem IMPLEMENTATION-MAP declarado, resolução semântica de conflitos em código de produção (apenas goldens são reconciliados automaticamente), integração com plataformas CI além de GitHub, dashboards de métricas, retry automático de `x-pr-fix` após regressão de compilação.

## 3. Definições de Qualidade Globais

### Global Definition of Ready (DoR)

- Source-of-truth em `java/src/main/resources/targets/claude/skills/` respeitado; nenhuma edição em `.claude/skills/`, `.claude/templates/` ou `src/test/resources/golden/**`
- Taxonomia ADR-0003 observada: a nova skill nasce em `core/pr/x-pr-merge-train/SKILL.md`
- Padrões de invocação Rule 13 confirmados por delegação: INLINE-SKILL para `x-git-worktree` e `x-git-commit`; SUBAGENT-GENERAL para workers de rebase (múltiplos `Agent(...)` como siblings em UMA mensagem)
- Bloco canônico de regeneração de goldens em `README.md:810-818` identificado e referenciado literalmente nos prompts dos subagents (RULE-005)
- Ponto de inserção do Step 3.6.5 em `x-story-implement` validado: entre os atuais Step 3.6 (TL Review + Dashboard Update) e Step 3.7 (Story-Level PR); sem gate v1/v2 porque a Phase 3 é compartilhada entre os dois schemas

### Global Definition of Done (DoD)

- **Cobertura:** ≥ 95% Line, ≥ 90% Branch em qualquer helper Java novo introduzido (parsers de state.json, geradores de prompt de subagent, reconciliador de overlap de arquivos). A skill em si é prompt-driven — coberta por golden diff.
- **Testes Automatizados:** unit tests para os helpers Java; `SkillsAssemblerTest.listCoreSkills_includesMergeTrain` obrigatório; golden diff tests para `x-pr-merge-train/SKILL.md` e para o diff de `x-story-implement/SKILL.md`. Cada story entrega ao menos um teste automatizado validando o critério de aceite principal.
- **Smoke Tests:** `mvn process-resources && mvn test -Dtest=SkillsAssemblerTest` deve listar `x-pr-merge-train` e todos os golden diffs devem passar.
- **Relatório de Cobertura:** JaCoCo por módulo, agregado no Maven report padrão.
- **Documentação:** ADR-0005 (Merge-Train Automation Architecture + PR-Fix Hook); CLAUDE.md atualizado com referência à nova skill; CHANGELOG seção Unreleased com uma entrada por story; exemplos de uso embutidos no próprio SKILL.md. `.claude/skills/**` e `.claude/README.md` apenas regenerados via `mvn process-resources` — nunca editados à mão (RULE-001).
- **Persistência:** `plans/merge-train/<id>/state.json` com `phase`, `lastPhaseCompletedAt`, `prsMergedOk[]`, `prsFailed[]`, `worktreeOwnership`, `schemaVersion="1.0"`; escrita atômica via arquivo `.tmp` + rename.
- **Performance:** train de 4 PRs sem overlap completa em < 1,5× do tempo do rebase+smoke mais lento sozinho, com `--max-parallel 4`.
- **TDD Compliance:** commits mostram test-first — golden + assert diff precedem ou acompanham cada diff de SKILL.md. Refactoring explícito após green. Incrementos via TPP nos cenários Gherkin.
- **Double-Loop TDD:** acceptance tests derivados dos cenários Gherkin (outer loop). Unit tests dos helpers Java guiados por TPP (inner loop).
- **Backward Compatibility:** `x-story-implement` sem Step 3.6.5 continua funcional caso `Skill(skill: "x-pr-fix")` falhe — degradação graceful: logar warning, não abortar Step 3.7. Legacy epics (planningSchemaVersion 1.0) não sofrem regressão (Rule 19).
- **Rule 13 Audit:** `grep -rnE "Invoke\s+\`?/?x-[a-z-]+" java/src/main/resources/targets/claude/skills/core/ --include=SKILL.md` exclui Triggers/Examples e retorna 0 no diff da release.

## 4. Regras de Negócio Transversais (Source of Truth)

| ID | Título | Descrição |
| :--- | :--- | :--- |
| **[RULE-001]** | Source-of-Truth Invariant | Toda alteração de skill/rule/template/golden é feita em `java/src/main/resources/targets/claude/` ou `java/src/main/resources/shared/`. `.claude/skills/**`, `.claude/rules/**`, `.claude/templates/**` e `src/test/resources/golden/**` são SAÍDAS geradas por `mvn process-resources` — hand-edits são sobrescritos no próximo build. |
| **[RULE-002]** | Rule 13 Invocation Patterns | `x-pr-merge-train` DEVE usar INLINE-SKILL (`Skill(skill: "...", args: "...")`) para delegar a `x-git-worktree` e `x-git-commit`, e SUBAGENT-GENERAL (múltiplos `Agent(...)` como siblings em UMA mensagem) para os workers de rebase/regen/push. Bare-slash em delegação é proibido; permitido apenas em `## Triggers` e `## Examples`. |
| **[RULE-003]** | Rule 14 Worktree Lifecycle | Creator owns removal: `x-pr-merge-train` cria seu próprio worktree (se `TRAIN_OWNS_WORKTREE`) e o remove apenas em sucesso. Non-nesting: Phase 0 DEVE invocar `x-git-worktree detect-context` antes de qualquer `create`. Em `phase=FAILED`, o worktree é PRESERVADO para diagnóstico humano (Rule 14 §4). |
| **[RULE-004]** | Goldens-Ours-Then-Regen | Durante resolução de conflitos no subagent de rebase, arquivos em `golden/**` e `java/src/test/resources/golden/**` DEVEM ser resolvidos com `git checkout --ours` seguido de regeneração canônica. Conflitos em código (`src/main/**`, `src/test/**` fora de golden) abortam o rebase com `CODE_CONFLICT_NEEDS_HUMAN` — nenhum merge semântico automático é tentado. |
| **[RULE-005]** | Goldens Regen Verbatim | O bloco de comando de regeneração de goldens em `README.md:810-818` DEVE ser embutido byte-a-byte no prompt do subagent de rebase. Paraphrasing ou "limpeza" do bloco é proibido — a reprodutibilidade determinística da regen em máquinas distintas depende de fidelidade literal. |
| **[RULE-006]** | Atomic, Reversible Commits | Cada story produz pelo menos um commit atômico por task, reversível via `git revert <sha>` sem tocar tasks irmãs (Rule 18). Mensagens seguem Conventional Commits (Rule 08): `feat(task-0042-YYYY-NNN): ...` ou `docs(task-0042-YYYY-NNN): ...` conforme o layer. |
| **[RULE-007]** | PR-Fix Hook Is Single-Pass | Step 3.6.5 em `x-story-implement` roda EXATAMENTE UMA VEZ por ciclo de story e SOMENTE quando TL=GO. Em qualquer resultado com `failed > 0` cujo motivo seja regressão de compilação, a Step 3.6.5 aborta com `PR_FIX_COMPILE_REGRESSION`, pula Step 3.7 e devolve controle ao operador humano. Não há retry automático nem reinvocação do TL. |
