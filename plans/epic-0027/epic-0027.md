# Épico: Migração para Git Flow Branching Model

**Autor:** Product Owner / Tech Lead
**Data:** 2026-04-07
**Versão:** 1.0
**Status:** Concluído

## 1. Visão Geral

**Chave Jira:** —

Migração do modelo de branching do projeto de trunk-based development (tudo mergeando em `main`) para Git Flow, introduzindo a branch `develop` como branch de integração, `release/*` para estabilização de releases, `hotfix/*` para correções urgentes a partir de `main`, e alterando o comportamento padrão de merge de PRs para `--no-merge` (PRs criados mas nunca mergeados automaticamente). Todos os artefatos em `.claude/` e `.github/` são gerados pelo `ia-dev-env`, portanto as alterações devem ser feitas no código-fonte do gerador (assemblers Java, templates e resources), e não editando manualmente os arquivos gerados.

## 3. Definições de Qualidade Globais

### Global Definition of Ready (DoR)

- Especificação técnica validada e aprovada
- Dependências da história resolvidas (histórias bloqueadoras concluídas)
- Templates de output (SKILL.md) existentes lidos e compreendidos
- Impacto nos 8 profiles de configuração mapeado
- Golden files de referência identificados para validação

### Global Definition of Done (DoD)

- **Cobertura:** ≥ 95% Line, ≥ 90% Branch
- **Testes Automatizados:** Unitários para lógica de template, integração para pipeline de geração, smoke tests para output completo. Cada história DEVE ter pelo menos 1 teste automatizado validando o critério de aceite principal.
- **Smoke Tests:** Obrigatório. Cada história deve passar no smoke gate com geração completa para pelo menos 1 profile.
- **Relatório de Cobertura:** JaCoCo com thresholds configurados no Maven
- **Documentação:** SKILL.md gerado atualizado, CLAUDE.md gerado consistente, README do projeto alinhado
- **Persistência:** Golden files atualizados para os 8 profiles
- **Performance:** Geração completa em < 30s para qualquer profile individual
- **TDD Compliance:** Commits show test-first pattern. Explicit refactoring after green. Tests are incremental (from simple to complex via TPP — Transformation Priority Premise).
- **Double-Loop TDD:** Acceptance tests derived from Gherkin scenarios (outer loop). Unit tests guided by TPP (inner loop).

## 4. Regras de Negócio Transversais (Source of Truth)

> Regras que se aplicam a múltiplas histórias. Cada história referencia as regras pelo ID. Alterações de regra propagam automaticamente para todas as histórias dependentes.

| ID | Título | Descrição |
| :--- | :--- | :--- |
| **[RULE-001]** | Estrutura de Branches Git Flow | O modelo de branching DEVE seguir: `main` (produção, recebe merges apenas de `release/*` e `hotfix/*`), `develop` (integração, recebe merges de `feat/*` e `fix/*`), `feat/*` ou `feature/*` (criadas de `develop`, merge para `develop`), `release/*` (criadas de `develop`, merge para `main` E `develop`), `hotfix/*` (criadas de `main`, merge para `main` E `develop`). |
| **[RULE-002]** | Proibição de Merge Direto em Main | Feature branches NUNCA devem ter `main` como target de PR. As únicas branches permitidas a mergear em `main` são `release/*` e `hotfix/*`. PRs de `feat/*` para `main` DEVEM ser rejeitados. |
| **[RULE-003]** | Default No-Merge | O modo de merge padrão no `x-dev-epic-implement` DEVE ser `no-merge`. PRs são criados mas NÃO mergeados automaticamente. Usuários devem opt-in explícito via `--auto-merge` ou `--interactive-merge`. |
| **[RULE-004]** | Develop como Base Default | Todos os branches de feature e fix DEVEM usar `develop` como base para criação de PR (`gh pr create --base develop`). Exceções: `hotfix/*` (target `main`) e `release/*` (target `main`). |
| **[RULE-005]** | Workflow de Release Branch | Releases DEVEM seguir: 1) Criar `release/X.Y.Z` de `develop`, 2) Apenas bug fixes na release branch, 3) Merge `release/X.Y.Z` para `main` via PR, 4) Tag `vX.Y.Z` em `main`, 5) Merge `release/X.Y.Z` de volta para `develop`, 6) Deletar release branch. |
| **[RULE-006]** | Hotfix com Dual Merge | Hotfix branches DEVEM ser mergeadas em AMBAS `main` (fix imediato) E `develop` (incluir fix no desenvolvimento). Se uma `release/*` branch existir, o hotfix mergea na release branch em vez de `develop`. |
| **[RULE-007]** | Retrocompatibilidade | Artefatos gerados para projetos trunk-based DEVEM continuar funcionando. O modelo de branching DEVE ser configurável via YAML config (`branching-model: gitflow` vs `branching-model: trunk`), com `gitflow` como default. |
| **[RULE-008]** | CI/CD Branch Awareness | Pipelines CI/CD gerados DEVEM diferenciar: CI trigga em `develop`, `release/*`, `hotfix/*` e PRs para `develop`/`main`. CD Staging trigga em push para `develop`. CD Production trigga em push para `main` ou tags de versão. |
| **[RULE-009]** | Schema Execution State | O `execution-state.json` DEVE incluir campo `baseBranch` (default: `develop`) para rastrear qual branch as stories targetam. Habilita resume correto após interrupção. |
| **[RULE-010]** | Branch Protection Guidance | Artefatos gerados DEVEM incluir documentação recomendando branch protection: `main` (require PR, status checks, reviews, no direct push), `develop` (require PR, status checks, no direct push). |
