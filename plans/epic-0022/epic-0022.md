**Autor:** Claude Code (Opus 4.6)
**Data:** 2026-04-05
**Versão:** 1.0
**Status:** Em Refinamento

## 1. Visão Geral

**Chave Jira:** —

Expansão das capacidades de segurança do gerador ia-dev-environment de **review-oriented** (checklists e knowledge packs de referência) para **execution-oriented** (skills que executam scans automatizados, integração com quality gates, framework de pentest multi-ambiente, avaliação de hardening, e personas de segurança especializadas). O épico produz 13 novas skills executáveis, 4 novos agentes de segurança, 1 novo knowledge pack (OWASP ASVS), 3 references ausentes no security KP, e 1 rule condicional de anti-patterns de segurança. Componentes existentes (security-engineer, x-dependency-audit, x-threat-model, x-codebase-audit, security KP, compliance KP) permanecem inalterados.

## 3. Definições de Qualidade Globais

### Global Definition of Ready (DoR)

- Spec file `SPEC-security-program-v1.md` lido e compreendido
- Componentes de segurança existentes mapeados (sem duplicação)
- Estrutura de diretórios do gerador compreendida (core vs conditional vs knowledge-packs)
- Templates de skill/agent/rule existentes analisados como referência de formato

### Global Definition of Done (DoD)

- **Cobertura:** ≥ 95% Line, ≥ 90% Branch para código novo
- **Testes Automatizados:** Unitários para domain/model e assembler; integração para golden file parity. Cada história DEVE ter pelo menos 1 teste automatizado validando o critério de aceite principal.
- **Smoke Tests:** Golden file parity tests passam para todos os profiles existentes (backward compatibility) e para profile com security flags habilitadas.
- **Relatório de Cobertura:** JaCoCo por módulo, threshold enforced no build.
- **Documentação:** SKILL.md de cada nova skill documentado; agents com checklist completo.
- **Persistência:** N/A (gerador de arquivos, sem banco de dados).
- **Performance:** Geração completa de ambiente com all-security-flags em < 10s.
- **TDD Compliance:** Commits show test-first pattern. Explicit refactoring after green. Tests are incremental (from simple to complex via TPP — Transformation Priority Premise).
- **Double-Loop TDD:** Acceptance tests derived from Gherkin scenarios (outer loop). Unit tests guided by TPP (inner loop).

## 4. Regras de Negócio Transversais (Source of Truth)

> Regras que se aplicam a múltiplas histórias. Cada história referencia as regras pelo ID.

| ID | Título | Descrição |
| :--- | :--- | :--- |
| **[RULE-001]** | Skill Idempotency | Toda skill de segurança executável DEVE ser idempotente — executar duas vezes produz o mesmo output sem efeitos colaterais. Resultados escritos em `results/security/` com filenames datados (`{skill-name}-{YYYY-MM-DD}.md`). |
| **[RULE-002]** | Tool Abstraction Layer | Skills de scanning NÃO DEVEM hardcodar uma única ferramenta. Cada skill define tabela de seleção (build-tool \| primary \| alternative \| install cmd). Fallback automático. Quando nenhuma ferramenta disponível: finding INFO com comandos de instalação. |
| **[RULE-003]** | SARIF Output | Todas as skills de scanning produzem findings em SARIF 2.1.0 como formato intermediário para CI. Report Markdown gerado a partir do SARIF. Permite upload para GitHub Advanced Security, GitLab SAST, SonarQube. |
| **[RULE-004]** | Environment Parameterization | Skills que interagem com sistemas em execução aceitam `--env local\|dev\|homolog\|prod`. Default: local. Produção requer `--confirm-prod`. Prod: APENAS scans passivos. Homolog: ativos sem destrutivos. Dev/local: tudo permitido. |
| **[RULE-005]** | Security Scoring Model | Todas as avaliações produzem score 0-100. Fórmula: 100 - (CRITICAL×15 + HIGH×8 + MEDIUM×3 + LOW×1). Cap em 0. Grades: A(90-100), B(75-89), C(60-74), D(40-59), F(0-39). |
| **[RULE-006]** | Persona Non-Interference | Agentes de segurança TÊM escopos não-sobrepostos. Cada agent declara `## Scope` com inclusões e exclusões. security-engineer: code review. pentest-engineer: exploitation. appsec-engineer: SDLC. devsecops-engineer: pipeline. compliance-auditor: regulatory. |
| **[RULE-007]** | Skill References Security KP | Toda nova skill referencia seções do security KP e compliance KP para orientação de remediação. Skills produzem findings; KPs fornecem correções. |
| **[RULE-008]** | Progressive Severity | Classificação via CVSS 4.0: CRITICAL ≥ 9.0, HIGH ≥ 7.0, MEDIUM ≥ 4.0, LOW < 4.0. Quando CVSS indisponível: matriz impacto × probabilidade do x-threat-model. |
| **[RULE-009]** | CI Pipeline Integration | Toda skill de scanning inclui seção `## CI Integration` com snippets YAML copy-paste ready para GitHub Actions, GitLab CI e Azure DevOps. |
| **[RULE-010]** | Conditional Generation | Novas skills seguem padrão `SkillsSelection`. Inclusão condicional via flags de `SecurityConfig`. Método `selectSecurityScanningSkills()` avalia flags. |
| **[RULE-011]** | Skill Composability | Skills orquestradoras (x-pentest, x-security-dashboard, x-security-pipeline) invocam skills atômicas via subagent. NUNCA duplicam lógica de scanning. |
| **[RULE-012]** | Agent Checklist Format | Novos agentes seguem formato de `security-engineer.md`: persona, role, recommended model, responsibilities, checklist numerado, output format, regras de conduta. |
| **[RULE-013]** | ASVS Level Mapping | Verificações OWASP mapeiam para ASVS L1/L2/L3. Skills referenciam nível ASVS coberto. |
| **[RULE-014]** | Backward Compatibility | Componentes existentes (security-engineer, 06-security-baseline, security KP, x-dependency-audit, x-threat-model) permanecem inalterados. Novos estendem, não substituem. |
| **[RULE-015]** | Template Engine Compatibility | Skill templates usam `{{PLACEHOLDER}}` (Mustache-style): `{{LANGUAGE}}`, `{{FRAMEWORK}}`, `{{BUILD_TOOL}}`, `{{PROJECT_NAME}}`. |
