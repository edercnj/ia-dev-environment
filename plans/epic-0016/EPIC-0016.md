## 1. Visao Geral

**Chave Jira:** —

Este epico implementa 6 melhorias estruturais ao ia-dev-environment, adicionando camadas de governance, rastreabilidade e inteligencia adaptativa ao ciclo de desenvolvimento assistido por IA. As features cobrem desde a geracao de artefatos de compliance (CONSTITUTION.md) ate a otimizacao automatica do lifecycle baseada na complexidade da story.

O escopo abrange modificacoes no config model, novos assemblers, novos templates Pebble, novas skills para Claude Code, um novo stack profile para fintech PCI-DSS, e integracao com skills existentes (x-dev-lifecycle, x-story-create, x-test-run).

**Escopo incluido:**

- CONSTITUTION.md: novo artefato de governance com invariants, CWE mappings, architecture boundaries, naming conventions e compliance requirements — gerado condicionalmente via campo `compliance` no config YAML
- Skill x-spec-drift-check: deteccao de divergencias entre spec (story) e codigo gerado, em modos standalone e inline
- Quality Gate no x-story-create: avaliacao de qualidade de scenarios Gherkin e data contracts antes da gravacao em disco, com scoring e threshold configuravel
- Matriz de Rastreabilidade no x-test-run: correlacao bidirecional requisito -> teste -> execucao, com output em Markdown para auditorias PCI-DSS
- Profile java-spring-fintech-pci: novo stack profile herdando de java-spring, com knowledge pack PCI-DSS, skill x-review-compliance e rules de seguranca
- Scope Assessment Adaptativo: classificacao automatica de stories em tiers SIMPLE/STANDARD/COMPLEX para otimizar fases do x-dev-lifecycle

**Escopo excluido:**

- Modificacoes em profiles existentes (os 10 profiles atuais nao sao alterados)
- Implementacao de runtime enforcement de CONSTITUTION.md (ArchUnit rules etc.) — apenas geracao do artefato
- Integracoes com ferramentas externas de compliance (SonarQube, Snyk etc.)
- Dashboards ou UIs para visualizacao de drift/traceability

## 3. Definicoes de Qualidade Globais

### Global Definition of Ready (DoR)

- Story possui data contract completo com tipos explicitos e obrigatoriedade (M/O) para todos os campos
- Story possui minimo de 4 acceptance criteria em formato Gherkin com valores concretos
- Todas as dependencias (Blocked By) estao em status DONE ou nao existem
- Nenhum cenario Gherkin utiliza linguagem vaga ("funciona corretamente", "resultado esperado", "configurado")
- Diagrama de sequencia Mermaid presente com componentes reais do sistema

### Global Definition of Done (DoD)

- Todos os acceptance tests (ATs) passando com status GREEN
- Nenhum golden file de profile existente quebrado (non-regression)
- Golden file do novo profile fintech-pci validado byte-a-byte (quando aplicavel)
- x-spec-drift-check sem FAILs criticos no codigo entregue
- Documentacao atualizada (CLAUDE.md, README sections, architecture doc)
- **Cobertura:** >= 95% Line, >= 90% Branch
- **TDD Compliance:** Commits mostram padrao test-first (teste precede implementacao no git log), refactoring explicito apos green, testes incrementais via TPP
- **Double-Loop TDD:** Acceptance tests derivados dos cenarios Gherkin (outer loop), unit tests guiados por Transformation Priority Premise (inner loop)
- **Rastreabilidade @GK-N bidirecional:** Todo @GK-N mapeia para >= 1 AT-N, todo AT-N referencia um @GK-N valido
- **Commits atomicos por ciclo TDD:** Cada ciclo Red-Green-Refactor produz um ou mais commits atomicos com formato Conventional Commits
- **Test plan antes de implementacao:** Nenhuma historia inicia implementacao sem test plan gerado via `/x-test-plan`
- Zero regressao: funcionalidades existentes continuam funcionais

## 4. Regras de Negocio Transversais

| ID | Titulo | Descricao |
| :--- | :--- | :--- |
| RULE-001 | Golden file obrigatorio | Toda modificacao de template Pebble ou adicao de novo profile requer atualizacao ou criacao do golden file correspondente em `src/test/resources/golden/`. Testes de integracao devem validar byte-a-byte. |
| RULE-002 | Registro de profile em integration tests | Novos profiles devem ter golden files registrados nos integration tests. O `mvn verify` deve incluir validacao do novo profile sem quebrar os 10 profiles existentes. |
| RULE-003 | Non-regression de profiles existentes | Nenhum dos 10 profiles existentes (go-gin, java-quarkus, java-spring, kotlin-ktor, python-click-cli, python-fastapi, rust-axum, typescript-nestjs, java-picocli-cli, typescript-commander-cli) pode ter seu comportamento ou output alterado por qualquer story deste epico. |
| RULE-004 | Estrutura padrao de skills | Toda nova skill deve seguir a estrutura `SKILL.md` com frontmatter YAML (name, description, version, user-invocable). Skills internas usam `user-invocable: false`. |
| RULE-005 | Backward compatibility no config YAML | Alteracoes no schema do config YAML devem ser backward-compatible. Novos campos sao sempre opcionais com defaults sensiveis. Configs existentes sem o novo campo devem continuar funcionando sem alteracao. |
| RULE-006 | Assemblers como primeira classe | Novos assemblers devem ser registrados no `AssemblerPipeline` como cidadaos de primeira classe — nao como side-effects de outros assemblers. Cada assembler possui responsabilidade unica. |
| RULE-007 | Compliance no frontmatter | Stories que implementam requisitos PCI-DSS devem declarar `compliance: pci-dss` no frontmatter. Isso ativa verificacoes adicionais no quality gate e no scope assessment. |
| RULE-008 | Cobertura minima JaCoCo | Toda story deve manter cobertura >= 95% linha e >= 90% branch. Build falha abaixo desses thresholds. |
| RULE-009 | Outputs acionaveis | Outputs de skills (drift check, quality gate, traceability) devem ser itemizados com severidade (PASS/WARN/FAIL), localizacao precisa e sugestao de correcao. Mensagens genericas sem detalhes sao proibidas. |
| RULE-010 | Constitution concreta | CONSTITUTION.md gerado deve conter invariants executaveis, mapeamentos CWE com exemplos de codigo correto/incorreto, e boundaries arquiteturais especificos. Documentacao ornamental sem exemplos concretos e proibida. |
