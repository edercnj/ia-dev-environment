# Épico: Task-First Planning & Execution Architecture

**Autor:** Platform Team
**Data:** 2026-04-11
**Versão:** 1.0
**Status:** Em Refinamento
**Épico ID:** EPIC-0038

## 1. Visão Geral

**Chave Jira:** —

Inversão do paradigma de planejamento e execução no gerador `ia-dev-env`:
mover de **top-down** (Epic → Story → Task-como-sub-seção) para
**bottom-up** (Task atômica e testável → Story como agregação de valor →
Epic como agrupador com RULEs transversais).

A motivação vem de sintomas observados durante a execução do EPIC-0034
(Remoção de Targets Não-Claude), onde:
(a) `x-task-plan` existia mas estava órfã (nunca invocada);
(b) tasks sofriam drift entre stories porque não tinham arquivo próprio;
(c) TDD acontecia no nível da story, não da task, tornando o ciclo
Red-Green-Refactor honesto impossível em operações atômicas;
(d) decisões de refactor (como o inlining revertido de
`PlatformContextBuilder.countActive` no story-0034-0004) eram tomadas
sem visibilidade de impacto per-task;
(e) reviews de PR rejeitavam observações corretas do Copilot por
premissas erradas sobre o que cada task produzia (ver PR #274 vs PR #282
— o mesmo erro sobre `.github/workflows/` foi cometido duas vezes).

Este épico promove **tasks a cidadãs de primeira classe** com arquivo próprio
(`task-TASK-NNN.md`), contratos I/O explícitos, DoD per-task, grafo de
dependências (`task-implementation-map-STORY-*.md`) e **refactor
comportamental** do skill existente `x-task-implement` (atualmente
chamada `x-dev-implement`, renomeada pelo EPIC-0036 per ADR-0003). O
refactor afeta a taxonomia de skills e o fluxo de execução, mas é
estritamente backward-compatible para épicos existentes (0025-0037) via
flag `planning_schema_version` em `execution-state.json`.

**Importante:** a skill `x-dev-implement` **já executa uma task individual
com TDD loop** (per ADR-0003 §"Primary cluster" rename rationale). Este
épico NÃO cria uma nova skill — ele refatora a existente para honrar os
novos contratos I/O e task-implementation-map definidos aqui.

**Não-objetivos:** este épico NÃO renomeia skills (isso é o EPIC-0036,
pré-requisito duro — ver seção 6), NÃO introduz task-as-PR stacked (isso
fica como v3), NÃO migra retroativamente épicos antigos.

- [ ] Primeiro épico dogfood candidato identificado (o próximo épico após este shipar)
- [ ] Decisão Task=commit vs Task=PR confirmada (v1 = commit, per ADR a ser escrito em story-0038-0009)

### Global Definition of Done (DoD)

- **Cobertura:** ≥ 95% line, ≥ 90% branch (Rule 05). Degradação máxima permitida: 2pp vs. baseline pré-épico.
- **Testes Automatizados:** cada story DEVE ter pelo menos 1 teste automatizado validando o critério de aceite principal. Novas skills (`x-task-plan` refatorada, `x-task-implement` refatorada pós-EPIC-0036) DEVEM ter integration tests.
- **E2E Integration:** teste de integração end-to-end do novo fluxo (plan epic → plan story → plan task → impl task → impl story → impl epic) verde em story-0038-0010.
- **Dogfood:** o próximo épico após este shipar DEVE usar v2 do schema com sucesso. Validação documentada.
- **Relatório de Cobertura:** JaCoCo report gerado via `mvn clean verify` e anexado ao PR de cada story.
- **Documentação:** CLAUDE.md raiz, `.claude/rules/*.md` e templates atualizados refletindo o novo fluxo.
- **Backward Compatibility:** épicos 0025-0037 continuam executando sem erro via legacy loader. Smoke test em story-0038-0008 valida isso.
- **Performance:** `x-story-plan` tempo total não pode aumentar > 20% vs. baseline (overhead de invocar x-task-plan per task deve ser compensado por paralelização).
- **TDD Compliance:** commits seguem padrão test-first. Refactoring explícito após green. Cada task do épico é Red→Green→Refactor honesto (este épico é o primeiro a ser dogfood do próprio schema que está criando).
- **Double-Loop TDD:** cenários Gherkin (outer loop) validam fluxo end-to-end. Unit tests (inner loop) validam cada skill individualmente.
- **Build Verde:** cada story termina com `mvn clean verify` verde ANTES do merge. Breakage intermediário entre stories é proibido.
- **Grep Sanity Check:** ao final do épico, `grep -rn "task embedded in story"` e variações retornam zero hits em código de produção.

## 4. Regras de Negócio Transversais (Source of Truth)

> Regras que se aplicam a múltiplas histórias. Cada história referencia as regras pelo ID.

| ID | Título | Descrição |
| :--- | :--- | :--- |
| **[RULE-TF-01]** | Task Testability | Toda task DEVE ser independently testable OU declarar explicitamente que precisa mock de outra task OU que é coalescível (mutually recursive) com outra task. Sem justificação escrita no `task-TASK-NNN.md`, a task não pode ser commitada.<br>Rationale: TDD honesto no nível atômico só é possível se cada task passa pelo ciclo Red-Green-Refactor isoladamente.<br>Enforcement: `x-task-plan` validation gate; `x-task-implement` rejects tasks without testability declaration. |
| **[RULE-TF-02]** | I/O Contracts Are Mandatory | Toda task DEVE declarar inputs (pré-condições) e outputs (pós-condições) no `task-TASK-NNN.md`. Outputs DEVEM ser verificáveis via grep, assert, ou teste automatizado.<br>Rationale: contratos explícitos previnem drift e permitem topological sort automático.<br>Enforcement: template `_TEMPLATE-TASK.md` exige as seções; x-task-plan falha se ausentes. |
| **[RULE-TF-03]** | Topological Execution | A ordem de execução de tasks dentro de uma story DEVE respeitar o `task-implementation-map-STORY-*.md`. Paralelismo é opt-in baseado em análise automática de dependências.<br>Rationale: execução em ordem errada invalida contratos I/O e quebra TDD.<br>Enforcement: `x-story-implement` (pós-EPIC-0036) lê o map e ordena; falha se map ausente (em schema v2). |
| **[RULE-TF-04]** | Task Commits Are Atomic | Cada task produz EXATAMENTE um commit (Conventional Commits com escopo `task(TASK-XXXX-YYYY-NNN)`). Coalesced groups produzem um commit único com múltiplas tasks listadas no body via `Coalesces-with:` footer.<br>Rationale: bisect-ability e review per-task.<br>Enforcement: `x-task-implement` commit template; pre-commit hook valida formato. |
| **[RULE-TF-05]** | Backward Compatibility | Épicos com `planning_schema_version == "1.0"` (ou ausente) DEVEM continuar executando sem erro via legacy loader. Novos épicos iniciados APÓS este épico mergeado DEVEM usar v2.<br>Enforcement: smoke test em story-0038-0008 executa um épico legacy (escolha: epic-0029 ou similar) e valida exit success. |

## 6. Coordenação com o EPIC-0036 (pré-requisito duro)

Este épico **DEPENDE** do EPIC-0036 (Skill Taxonomy and Naming Refactor)
estar 100% mergeado em develop antes de iniciar story-0038-0001. A
dependência é dura, não soft:

- **EPIC-0036 já tem ADR-0003 e `plans/epic-0036/skill-renames.md` commitados
  em develop**, mas as stories 0036-0001..0006 de execução ainda não foram
  aplicadas ao codebase. A skill `x-dev-implement` (que EPIC-0038 precisa
  refatorar) ainda não foi renomeada para `x-task-implement`.
- EPIC-0036 já **definiu** os nomes finais (ADR-0003 §D3 + staging doc §2.1).
  Portanto este épico NÃO tem incerteza sobre nomes — os 10 stories do
  EPIC-0038 já usam os nomes finais.
- Se este épico começar antes do EPIC-0036 completar, a execução vai
  referenciar skills que ainda têm os nomes antigos (`x-dev-implement`,
  `x-dev-story-implement`, etc.), causando falhas de resolução.
- Por isso, global DoR do EPIC-0038 inclui o check explícito: "EPIC-0036
  stories 0036-0001..0006 mergeadas em develop". Pre-flight em
  `x-story-implement` valida.

**Se ADR-0003 for alterado** (mudança nos nomes finais após esta spec ser
escrita), este épico é re-baselined: spec, epic file e stories são
atualizados ANTES do start.

## 7. Dogfooding Strategy

Este épico é **o primeiro a ser dogfood do próprio schema que está
criando**. Isso é intencional:

- **Fase de execução das stories 0001-0010:** stories usam schema v1 (legacy)
  porque o v2 ainda não existe quando começamos.
- **Pós-merge do story-0038-0010:** o próximo épico planejado (escolha a
  confirmar) será o primeiro a usar schema v2 end-to-end.
- **Validação:** story-0038-0010 AC inclui "dogfood bem-sucedido" — rodar
  `x-epic-orchestrate <next-epic-id>` (nome pós-EPIC-0036) em v2 e verificar
  que produz todos os artefatos esperados (task-TASK-NNN.md × N,
  task-implementation-map, etc.).

Alternativa rejeitada: usar v2 DENTRO do próprio épico-0038 (auto-hospedagem
recursiva). Rejeitada porque (a) bootstrap problem — o `x-task-plan`
callable refatorado e o comportamento task-first de `x-task-implement` só
existem após stories 0003 e 0005 shiparem, (b) risco de loop dependente.
