# Épico: Telemetria de Execução de Skills

**Autor:** Eder Celeste Nunes Junior
**Data:** 2026-04-13
**Versão:** 1.0
**Status:** Em Refinamento

## 1. Visão Geral

**Chave Jira:** —

Instrumentar as skills do `ia-dev-environment` com captura estruturada de telemetria (entry/exit, fases internas e chamadas a tools) para permitir análise de performance, identificação de gargalos e detecção de regressões cross-epic. A captura é híbrida: hooks passivos (`PreToolUse`/`PostToolUse`/`SessionStart`/`Stop`/`SubagentStop`) cobrem automaticamente toda e qualquer execução, enquanto skills-chave (implementação, planejamento, criação de épicos/histórias) emitem marcadores semânticos de fase. Eventos são persistidos em NDJSON per-epic (`plans/epic-XXXX/telemetry/events.ndjson`) e indexados globalmente em `.claude/telemetry/index.json`. Duas skills de análise (`/x-telemetry-analyze`, `/x-telemetry-trend`) produzem relatórios em Markdown com tabelas, diagramas Mermaid Gantt e tendências P50/P95. Fora de escopo: exportação para Grafana/OpenTelemetry externos, UI web, e telemetria runtime do binário gerado.

## 3. Definições de Qualidade Globais

### Global Definition of Ready (DoR)

- Contrato de evento (JSON Schema) definido e versionado (`schemaVersion`)
- Política de privacidade/scrubbing documentada em rule numerada
- Layout de storage (per-epic NDJSON + global index) aprovado
- Feature flag de telemetria (`CLAUDE_TELEMETRY_DISABLED`) especificada e testada
- Impacto em performance medido e documentado (overhead ≤ 50ms por tool call)

### Global Definition of Done (DoD)

- **Cobertura:** ≥ 95% Line, ≥ 90% Branch (Rule 05) em todos os pacotes `dev.iadev.telemetry.*`
- **Testes Automatizados:** Unit tests para tipos Java, integration tests para hooks shell, acceptance tests para skills `x-telemetry-analyze` e `x-telemetry-trend`. Cada história DEVE ter pelo menos 1 teste automatizado validando o critério de aceite principal.
- **Smoke Tests:** Obrigatórios. Fluxo ponta-a-ponta deve gerar NDJSON válido e o `x-telemetry-analyze --epic EPIC-0040` deve produzir relatório sem erros.
- **Relatório de Cobertura:** JaCoCo XML/HTML per-module, reportado em `plans/epic-0040/reports/`
- **Documentação:** ADR-0004 criado, CLAUDE.md root e `.claude/README.md` com seção "Telemetria"
- **Persistência:** Zero eventos perdidos em falha; hooks fail-open (`set +e`, timeout 5s)
- **Performance:** Overhead de captura < 50ms por tool call; skill de análise processa 10k eventos em < 5s
- **Privacidade:** Zero PII nos NDJSONs commitados (validado via fuzz tests com AWS keys, JWT, emails)
- **TDD Compliance:** Commits show test-first pattern. Explicit refactoring after green. Tests are incremental (from simple to complex via TPP — Transformation Priority Premise).
- **Double-Loop TDD:** Acceptance tests derived from Gherkin scenarios (outer loop). Unit tests guided by TPP (inner loop).

## 4. Regras de Negócio Transversais (Source of Truth)

| ID | Título | Descrição |
| :--- | :--- | :--- |
| **[RULE-001]** | Event Schema Versioning | Todo evento DEVE incluir `schemaVersion` (SemVer). Mudanças de schema requerem bump de versão: MAJOR para breaking (remoção/renomeação de campo obrigatório), MINOR para novo campo opcional, PATCH para docs/comentários. Leitores DEVEM suportar N-1 (backward compatibility). |
| **[RULE-002]** | NDJSON Append-Only | Arquivos de eventos são write-once, append-only, uma linha JSON válida por evento. Proibido: edição in-place, reordenação, deleção parcial. Rotação por sessão (`sessions/{sessionId}.ndjson`) é opcional e NÃO substitui o `events.ndjson` canônico. |
| **[RULE-003]** | Zero PII | Eventos NÃO podem conter: conteúdo de arquivos, comandos completos (apenas nome da tool), credenciais (AWS keys, JWT, tokens), emails, paths absolutos fora do repo. `TelemetryScrubber` aplica whitelist de campos + regex de mascaramento. Fuzz tests validam. |
| **[RULE-004]** | Hook Fail-Open | Hooks de telemetria NÃO podem bloquear a execução da skill ou tool. `set +e` obrigatório; timeout 5s; erros vão para stderr apenas. Ausência de `.claude/telemetry/` ou falha de escrita não quebra a sessão. |
| **[RULE-005]** | Context Resolution Order | `epicId`/`storyId`/`taskId` são resolvidos em ordem: (1) env `CLAUDE_TELEMETRY_CONTEXT`, (2) branch Git (`feature/epic-(\d+)`), (3) `plans/epic-*/execution-state.json` com `currentPhase != null`, (4) fallback `"unknown"`. Nunca aborta por falta de contexto. |
| **[RULE-006]** | Feature Flag Opt-Out | Variável `CLAUDE_TELEMETRY_DISABLED=1` desativa TODA captura (hooks e markers). Ausência da variável = telemetria ativa (opt-out, não opt-in). Skills NÃO devem ler essa flag diretamente — apenas os hooks verificam. |
| **[RULE-007]** | Storage Layout Imutável | Paths canônicos: `plans/epic-XXXX/telemetry/events.ndjson` (per-epic, commitado) e `.claude/telemetry/index.json` (global, gitignored — é cache). Skills de análise NÃO podem escrever em outros paths sem flag explícita (`--out`). |
| **[RULE-008]** | Source of Truth: Resources | Toda alteração de skill/hook/rule é feita em `java/src/main/resources/targets/claude/` (source of truth). Hand-edits em `.claude/` são perdidos no próximo `mvn process-resources`. Golden tests (`GoldenFileRegenerator`) validam consistência. |
