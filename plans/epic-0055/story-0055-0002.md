## 3. Descrição

Como **CI/CD engineer**, eu quero implementar **4 camadas de enforcement** para o phase gate contract, garantindo que:

- Violações de `TaskCreate` por fase sejam detectadas durante CI (fail fast)
- Violações de gates PRE/POST sejam auditadas antes de merge
- Stop hook detecte gate failures no final de turno LLM (defesa em profundidade)
- PreToolUse hook bloqueie tentativas de pular fases (ex: chamar Phase 3 antes de Phase 2 completar)

Esta story implementa:
1. `scripts/audit-task-hierarchy.sh` — CI audit (exit 25)
2. `scripts/audit-phase-gates.sh` — CI audit (exit 26)
3. `.claude/hooks/verify-phase-gates.sh` — Stop hook
4. `.claude/hooks/enforce-phase-sequence.sh` — PreToolUse hook
5. `audits/task-hierarchy-baseline.txt` — Grandfather list para deprecação

### 3.1 CI Audit: `scripts/audit-task-hierarchy.sh`

Bash script rodando em CI (via GitHub Actions, etc):

**Input:**
- `--skills-root java/src/main/resources/targets/claude/skills/`

**Validações:**
1. Para cada SKILL.md sob `core/dev/`, `core/plan/`, `core/ops/`, `core/review/`, `core/pr/` (orquestradores):
   - Detectar todas as seções `## Phase N` numeradas
   - Verificar que **cada** Phase N contém ≥ 1 `TaskCreate(...)` call
   - Verificar que `subject:` segue regex canônico (seção 5.3 Rule 25)
   - Verificar que `activeForm` é < 40 chars

2. Para cada `TaskCreate(...)`  detectado:
   - Validar que existe um `TaskUpdate(..., status: "completed")` posterior no mesmo SKILL.md
   - Ou contém comentário `<!-- audit-exempt: <reason> -->`

3. Falhar com exit 25 (`TASK_HIERARCHY_VIOLATION`) se qualquer falha
4. Saída JSON com detalhes: `{ violations: [ { skill: ..., phase: ..., reason: ... } ] }`

**Baseline:**
- Arquivo `audits/task-hierarchy-baseline.txt` lista skills preexistentes (grandfather).
- Novos violations em skills fora do baseline = fail (imutável após merger, per Rule 24 Camada 3).

### 3.2 CI Audit: `scripts/audit-phase-gates.sh`

Bash script validando gates:

**Validações:**
1. Para cada `## Phase N` numerada:
   - Verificar que existe `Skill(skill: "x-internal-phase-gate", args: "--mode pre"`
   - Verificar que existe `Skill(skill: "x-internal-phase-gate", args: "--mode post"`
   - Ou contém comentário `<!-- phase-no-gate: <reason> -->`

2. Falhar com exit 26 (`PHASE_GATE_VIOLATION`) se qualquer phase sem gates

### 3.3 Stop Hook: `.claude/hooks/verify-phase-gates.sh`

Hook registrado em `settings.json` como:

```json
{
  "hooks": {
    "Stop": [".claude/hooks/verify-phase-gates.sh"]
  }
}
```

**Comportamento (executado no Stop event, fim do turno LLM):**

1. Detectar branch atual (via `git rev-parse --abbrev-ref HEAD`)
2. Se branch é `epic/*` ou `feature/*` (ativo, não merge):
   - Ler `execution-state.json` se existir em `plans/epic-XXXX/`
   - Verificar `taskTracking.phaseGateResults[]`
   - Se qualquer entry tem `passed=false`:
     - Log WARNING stderr com details
     - Exit 2 (signal para user)
   - Se `phaseGateResults` vazio ou `taskTracking.enabled=false`:
     - Skip (legacy mode)

### 3.4 PreToolUse Hook: `.claude/hooks/enforce-phase-sequence.sh`

Hook registrado como:

```json
{
  "hooks": {
    "PreToolUse": [".claude/hooks/enforce-phase-sequence.sh"]
  }
}
```

**Comportamento (executado ANTES de `Skill(...)` call):**

1. Parse `Skill(skill: "x-...", args: "...")` from the PreToolUse event context
2. Se skill-alvo é um orquestrador (listado em Anexo B):
   - Ler `execution-state.json.taskTracking.phaseGateResults[]`
   - Detectar última phase com `passed=true`
   - Se skill-alvo é `x-story-implement` e última phase é Phase 1:
     - Verificar que Phase 2 é a próxima a ser invocada
     - Se tentativa é chamar "gate de Phase 3" sem Phase 2 completar:
       - Block com exit 2 + mensagem visível

Esse hook previne "LLM pula Phase 2 e tenta invocar Phase 3 diretamente".

## 3.5 Entrega de Valor

- **Valor Principal:** 4-layer enforcement que previne inlining de sub-skills, pulo de fases, e ausência de artefatos. CI falha fast antes de humano revisar PR (defesa automática).
- **Métrica de Sucesso:** Operador tenta comitar SKILL.md sem `TaskCreate` em Phase 1; CI audit falha com `TASK_HIERARCHY_VIOLATION` antes de PR ser criada.
- **Impacto no Negócio:** Reduz risk de inlining violations atingindo production (hoje: detectadas no Stop hook depois). Backward compatibility via baseline permite rollout gradual.
