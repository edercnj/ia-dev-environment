## Visão Geral

A arquitetura de granularidade de tasks do Claude Code colapsou após EPIC-0049 (Thin Orchestrators). Historicamente, orquestradores (`x-epic-implement`, `x-story-implement`, `x-task-implement`, `x-release`, `x-epic-orchestrate`, `x-review`, `x-review-pr`, `x-pr-merge-train`) criavam listas de tasks visíveis ao operador durante execução. Hoje, apenas `x-review` emite `TaskCreate`/`TaskUpdate` — os demais delegam via `Skill(...)` sem rastrear fases, impedindo o operador de ver:

- Se a onda de planejamento completou seus 5 planos (arch, impl, test, tasks, security, compliance)
- Se os 9 especialistas de review terminaram
- Se o verify gate da Rule 24 foi executado
- Se cada ciclo de TDD seguiu Red → Green → Refactor
- Em qual fase a execução travou

EPIC-0055 introduz:

1. **Rule 25** — Invariantes de hierarquia de tasks + gates por fase
2. **`x-internal-phase-gate`** — Skill que valida transições de fase (PRE/POST gates)
3. **4 camadas de enforcement** — Normativa, Stop hook, PreToolUse hook, CI audit
4. **Retrofit de 8 orquestradores** — `x-epic-implement`, `x-story-implement`, `x-task-implement`, `x-release`, `x-epic-orchestrate`, `x-review`, `x-review-pr`, `x-pr-merge-train`
5. **Backward compatibility via Rule 19** — Campo `execution-state.json.taskTracking` (default false para legacy)

Resultado: lista visível de tasks em **4 níveis hierárquicos** (Epic › Story › Phase › Wave/Cycle), com **gates formais** bloqueando transições de fase até evidências em disco + TaskComplete local.

---

## Objetivos Primários

- Restabelecer visibilidade de granularidade de tasks ao operador (4 níveis aninhados)
- Implementar phase gates formais com enforcement em 4 camadas
- Manter backward compatibility com epics legados (Rule 19)
- Zero regressão de performance (overhead < 2% do wall-clock)

## Regras Críticas

### REGRA-001 — Emissão obrigatória de `TaskCreate` por fase

**Todos** os 8 orquestradores **DEVEM** emitir `TaskCreate(subject: "...")` ao entrar em cada `## Phase N` numerada (seção 4.2 da spec).

Padrão: `subject` usa separador `›` para hierarquia:  
```
"story-0060-0001 › Phase 1 › Arch plan"
"TASK-0060-0001-003 › Step 2 › Cycle 1 › Red"
```

Profundidade máxima: 4 níveis (Root / Level2 / Level3 / Level4).

**Enforcement:** CI audit `scripts/audit-task-hierarchy.sh` (exit 25).

### REGRA-002 — Phase gates PRE/POST obrigatórios

Cada `## Phase N` numerada **DEVE** invocar:

- **Antes:** `Skill(skill: "x-internal-phase-gate", args: "--mode pre --phase N --expected-artifacts ...")`
- **Depois:** `Skill(skill: "x-internal-phase-gate", args: "--mode post --phase N --parent-task-id ...")`

Gates falham com exit 12 (`PHASE_GATE_FAILED`) se:
- Sub-tasks anteriores não estão todas `completed`
- Artefatos esperados não existem em disco

**Enforcement:** CI audit `scripts/audit-phase-gates.sh` (exit 26).

### REGRA-003 — Wave paralela padronizada

Quando um orquestrador dispara Batch A/B (subagentes em paralelo), **DEVE**:

1. Emitir `TaskCreate` para cada sub-wave (até 10 tasks em Batch A)
2. Recordar IDs em um map `waveTaskIds`
3. Atualizar todos para `completed` em Batch B
4. Invocar `x-internal-phase-gate --mode wave` para validar completeção

Exemplo: `x-story-implement` Phase 1 onda de 5 planejadores (arch, impl, test, tasks, security, compliance).

### REGRA-004 — Loop sequencial com `addBlocks`

Quando um orquestrador itera sobre items (stories, tasks), **DEVE**:

1. Por iteração: emitir `TaskCreate`
2. Chainar via `TaskUpdate(taskId: previous, addBlocks: [current])`
3. Renderiza como "blocked by #N" na CLI
4. Invocar sub-skill em série (pattern 4.4 da spec)

Exemplo: `x-epic-implement` Phase 3 (loop de stories), `x-task-implement` Phase 2 (loop de TDD cycles).

### REGRA-005 — Integração com Rule 24 (Execution Integrity)

O `x-internal-phase-gate --mode post` da **última fase produtora-de-evidência** de cada orquestrador **DEVE** incluir os 4 artefatos obrigatórios da Rule 24 em `--expected-artifacts`:

- `verify-envelope-STORY-ID.json`
- `review-story-STORY-ID.md`
- `techlead-review-story-STORY-ID.md`
- `story-completion-report-STORY-ID.md`

Promove enforcement da Rule 24 de "stop-hook detecta depois" para **gate sync antes** da fase ser considerada completa.
