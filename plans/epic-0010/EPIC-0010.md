# Epico: Otimizacao de Paralelismo e Estrategia de Worktree nas Skills de Orquestracao

**Autor:** Arquiteto de Software / Claude
**Data:** 2026-03-24
**Versao:** 1.0
**Status:** Em Refinamento

## 1. Visao Geral

**Chave Jira:** EPIC-0010

A analise profunda das skills de orquestracao (`x-dev-epic-implement`, `x-dev-lifecycle`, `x-review`, `x-dev-implement`) revelou **6 gargalos de paralelismo** e **1 bug funcional** (race condition) que impactam significativamente o tempo de execucao de epicos. Atualmente, a execucao paralela via git worktrees e opt-in (`--parallel`), stories independentes dentro da mesma fase sao executadas sequencialmente por default, e fases internas do lifecycle que poderiam rodar em paralelo estao serializadas sem necessidade.

Este epico corrige o bug funcional, otimiza o paralelismo em todas as camadas da cadeia de orquestracao, e implementa uma estrategia proativa de deteccao de conflitos (pre-flight conflict analysis) para tornar a execucao paralela via worktrees segura e eficiente por default.

**Escopo incluido:** Correcao de bug na Phase 1B/1C, inversao do default sequential→parallel, pre-flight conflict analysis, rebase-before-merge strategy, paralelizacao de phases internas (1A, 3, consolidation, Phase 2 by layer), documentacao da estrategia de worktree.

**Escopo excluido:** Alteracoes no mecanismo de worktree do Claude Code (Agent tool), mudancas estruturais no formato de checkpoint/execution-state.json (novos campos obrigatorios, remocao de campos existentes); este epico pode introduzir novos valores/estados e ajustar defaults dentro da estrutura atual, alteracoes em skills de review (`x-review`, `x-review-pr`) que ja estao bem paralelizadas.

## 3. Definicoes de Qualidade Globais

### Global Definition of Ready (DoR)

- Skills alvo lidas e compreendidas em sua totalidade
- Dependencias entre stories validadas (grafo aciclico)
- Templates de skill (`SKILL.md`) com frontmatter YAML valido
- Impacto em skills downstream mapeado

### Global Definition of Done (DoD)

- **Cobertura:** N/A (este epico altera documentacao de skills, nao codigo fonte)
- **Consistencia:** Todas as skills modificadas mantem frontmatter YAML valido e estrutura de secoes intacta
- **Backward Compatibility:** Flags existentes (`--parallel`, `--skip-review`, `--dry-run`, `--resume`) continuam funcionando
- **Documentacao:** Cada skill modificada tem secao de integracao atualizada
- **TDD Compliance:** Commits show test-first pattern. Explicit refactoring after green. Tests are incremental (from simple to complex via TPP — Transformation Priority Premise).
- **Double-Loop TDD:** Acceptance tests derived from Gherkin scenarios (outer loop). Unit tests guided by TPP (inner loop).

## 4. Regras de Negocio Transversais (Source of Truth)

| ID | Titulo | Descricao |
| :--- | :--- | :--- |
| **[RULE-001]** | Context Isolation | Subagents recebem APENAS metadata (IDs, paths, flags). Nunca passar codigo fonte, diffs ou knowledge packs diretamente. O subagent nasce com contexto limpo e morre apos conclusao. |
| **[RULE-002]** | Checkpoint Atomicity | O checkpoint (`execution-state.json`) deve ser atualizado apos CADA operacao individual (merge, falha, status change). Nunca batch updates. Se o orquestrador crashar, o checkpoint reflete o ultimo estado consistente. |
| **[RULE-003]** | Single Message Dispatch | Subagents que devem rodar em paralelo DEVEM ser lancados em uma UNICA mensagem (single tool call block). Lancar em mensagens separadas serializa a execucao. |
| **[RULE-004]** | Integrity Gate Mandatory | O integrity gate entre fases e obrigatorio e nao pode ser bypassed. Toda transicao de fase requer compilacao + testes + coverage passing. |
| **[RULE-005]** | Dependency-Safe Dispatch | Apenas stories com TODAS as dependencias em status SUCCESS podem ser despachadas. Stories BLOCKED nunca sao despachadas. |
| **[RULE-006]** | Worktree Branch Isolation | Cada worktree opera em branch isolada (`feat/epic-{epicId}-{storyId}`). O merge no epic branch e sequencial, ordenado por critical path priority. |
| **[RULE-007]** | Critical Path Priority | Merges e dispatches priorizam stories no caminho critico. Isso minimiza o tempo total do epico. |
| **[RULE-008]** | SubagentResult Contract | Todo subagent deve retornar `{ status, commitSha, findingsCount, summary }`. Campos ausentes = story FAILED. |
| **[RULE-009]** | Backward Compatibility | Todas as alteracoes em skills devem manter flags e comportamentos existentes funcionando. Novas flags sao aditivas. |
| **[RULE-010]** | Pre-flight Before Parallel | Antes de despachar stories em paralelo, a file-overlap matrix deve ser computada. Stories com alto overlap sao demotadas para execucao sequencial. |
